## 2. Scope (§7/§11/§12)

Prompt 124 §11 allowed scope: "Status registry/schema/migrations/resolver/UI metadata/tests/docs and bounded legacy map," 5-15 files, 1-3 migrations. §12 forbidden: "Domain transition logic, client authorization, destructive status rewrite." §24 (binding): "Workflow/authorization controls transitions; status engine describes/resolves" — this capability never decides whether a transition is legal; that remains `PLT-122`'s (or a future domain's) own transition authority.

- **Zero real domain status sets are seeded in this migration.** `docs/architecture/07_CONFIGURATION_ENGINE_WORKSTREAM.md` §7.2 catalogues the real, sourced 24 status transitions across 6 domain entities (Lead/Quotation/Shipment/Invoice/Vendor/Ticket), but its own §13 Module Adoption Map and §15 Atomic Backlog place every one at Phase 2 (Commercial), 3 (Operations), 4 (Finance), 6 (Procurement), or 7 (HRIS/Ticketing) — verbatim: "Workflow/Status/Numbering/Form engines | 1 | Bootstrap all 4 as empty templates." This migration is exactly that bootstrap: the registry/presentation/legacy-mapping *mechanism*, no `status_set` row of any kind. The one safe representative example this checkpoint uses is the real, sourced Quotation shape (`Draft → Submitted(Under Approval) → Approved → Sent → Accepted|Expired`) — built and proven entirely inside `scripts/db-tests/status.sql`, not seeded into the migration; activating it against a real `quotations` table is explicitly Phase 2 scope.
- **Canonical status meaning is permanent; presentation is the only tenant-configurable surface** (§24, `07_*.md` §6's "canonical semantics versus tenant labels" rule). `app.canonical_statuses` rows are registered once (Supreme-only, idempotent) and never mutated in place; only a tenant's own presentation is ever versioned/republished.
- **Presentation reuses `PLT-121`'s Configuration Engine once per status set**, not once globally. `config_objects_scope_shape_check` forces `scope_id` to be null at `scope_level='tenant'`, so the one shared, seeded generic `config_type_code='status'` could host at most one tenant-scoped object ever — insufficient for multiple independently-versioned status sets (Quotation, Shipment, Invoice, ...). `app.register_status_set()` resolves this by calling `PLT-121`'s own `app.register_config_type('status:' || code, ...)` to mint a dedicated presentation type per set, composing the existing extensibility mechanism (`PLT-120`/`122`'s "registry, not enum" discipline) rather than forking a parallel one. The originally-seeded generic `'status'` type remains exactly what `PLT-121` disclosed it to be — an empty, unused template placeholder.
- **Accessible non-color cues are structurally required** (§15: "accessible non-color cues"). `app.validate_status_presentation()` rejects a color-only status label entry at publish time (`status_missing_accessible_cue`) — proven directly.
- **Legacy mapping has no silent collision** (§25). `app.status_legacy_mappings` carries a real `unique(status_set_code, legacy_value)` constraint; remapping an already-mapped legacy value to a *different* canonical status raises `status_legacy_mapping_collision` rather than silently overwriting.
- **No configuration item can grant a transition or hide a prohibited state** (§16). `app.resolve_status_presentation()` always resolves a real, existing canonical status or fails with `status_unknown_code` — there is no code path letting a tenant's presentation suppress a canonical status's existence.

## 3. Implementation (§20)

### 3.1 Canonical registry and legacy mapping

`supabase/migrations/20260719100000_create_status_engine.sql`. `app.status_sets` (registry, code + a dedicated `status:<code>` config_type minted per set), `app.canonical_statuses` (permanent machine IDs, `category` check(`initial`/`active`/`terminal`), `is_terminal` as a generated column), `app.status_legacy_mappings` (`unique(status_set_code, legacy_value)`). `app.register_status_set()`/`app.register_canonical_status()`/`app.register_status_legacy_mapping()` — all idempotent, Supreme-Admin-only, audited via `capture_audit_event(null, ...)` (platform-wide, matching `PLT-120`/`121`/`122`'s registry precedent).

### 3.2 Presentation versioning and structural validation

`app.validate_status_presentation(p_version_id)` derives the status set from the bound `status:<code>` config_type, reads the single `labels` config_item (a jsonb object keyed by canonical code), and raises 5 distinct named exceptions: `status_not_a_presentation_version`, `status_missing_labels`, `status_unknown_code`, `status_missing_label`, `status_missing_accessible_cue`, `status_invalid_color`. `app.publish_status_presentation()` runs this gate then composes `PLT-121`'s own `app.publish_config_version()`.

### 3.3 Resolution and read view-models

`app.resolve_status_presentation()` composes `PLT-121`'s own `app.resolve_config()` 6-level precedence walk against the status set's dedicated type, falling back to a structural, code-derived label (`initcap(replace(code,'_',' '))`) and a neutral `circle` icon — never a fabricated tenant-specific color — when nothing has published yet (§22: "falls back safely"); raises `status_unknown_code` only if the canonical code itself was never registered. `app.resolve_legacy_status()` and `app.get_status_set_registry()` round out the read surface. `server/queries/status.ts`'s `StatusPresentationCache` mirrors `PLT-106`/`117`/`118`/`119`/`121`'s explicit-invalidation TTL cache shape.

### 3.4 Access/audit/tests/docs

RLS: `status_sets`/`canonical_statuses`/`status_legacy_mappings` are broadly `SELECT`-able platform reference data for `authenticated`, matching `PLT-122`'s `workflow_hooks` precedent exactly. Every registration self-captures a canonical `app.audit_logs` entry — no bespoke `*_history` table for this capability. Per `ERR-2026-004`, the migration ends with its own `revoke execute on all functions in schema app from public;` before role-specific grants.
