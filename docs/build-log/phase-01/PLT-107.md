## 2. Scope and a disclosed constraint (§7/§8)

**No live Supabase project exists for this repository** — established at `PH0-085`/`094` (`ADR-0010`) and unchanged through every checkpoint since. ## 3. Implementation (§20)

### 3.1 Identity/membership boundary (task 1)

Authentication proves identity only — entitlement (`PLT-106`) and RBAC (`PLT-111`/`112`) determine access (Prompt 107 §24). `app.tenant_user_identities` (`supabase/migrations/20260716095343_link_auth_identities.sql`) links `auth.users.id` to `app.tenants`, deliberately carrying no role/permission field. One identity may be linked to multiple tenants (proven in §4.1, not assumed) — the four-layer model does not imply single-tenant membership.

### 3.2 Server/client auth clients and identity linkage (task 2)

Real, tested SQL: `app.link_auth_identity()` (idempotent — a repeated invite for the same `(auth_user_id, tenant_id)` pair returns the original row, matching `app.provision_tenant`'s established pattern) and `app.revoke_auth_identity()` (guarded, trigger-enforced `invited→active→revoked`, terminal). **Real, disclosed `NOT_RUN`:** the live `@supabase/ssr` `createServerClient`/cookie-reading/middleware wiring that would call these RPCs from an actual Next.js request — no `app/` router exists yet in this repository (the portal capabilities, `PLT-135`/`136`, are the first real consumers) and no live Supabase project exists to authenticate against.

**Local-only test fixture, not a real migration:** `scripts/db-tests/fixtures/auth-schema-stub.sql` creates a minimal `auth.users(id, email, created_at)` stub so the FK/linkage/RLS logic can be proven against a real Postgres instance today. Its own header states, and this build log restates, that it is never applied to a real Supabase project — `scripts/db-tests/run.sh` was extended to load `fixtures/*.sql` before `supabase/migrations/*.sql`, keeping the two directories structurally separate so this distinction cannot be lost by accident.

### 3.3 Invitation/activation/recovery/logout/revocation (task 3)

Invitation and revocation are real, tested, and reasoned (`revoke_auth_identity` requires a non-empty `reason`, Prompt 107 §22/§24). **Real, disclosed `NOT_RUN`:** email delivery for invitation/recovery links (depends on `PLT-127`, Notification Engine, not yet built) and an actual logout/session-invalidation call against a live GoTrue endpoint (no live project). §16's redirect-safety requirement for these flows is implemented and tested today regardless (`lib/auth/redirect-allowlist.ts`, §3.4), since that logic does not depend on a live auth backend.

### 3.4 MFA-ready extension and pure security primitives (task 4)

`app.tenant_user_identities.mfa_enrolled` is a real, minimal MFA-readiness column (RPD-023: MFA mandatory for privileged roles) — full enrollment/verification/challenge flow remains `NOT_RUN` (needs a live GoTrue MFA endpoint). `lib/auth/session-cookie-options.ts` (`httpOnly`/`secure`/`sameSite=lax`/`path`/`maxAge` — the `maxAge` default of 7 days is this checkpoint's own disclosed construction, no ratified value exists) and `lib/auth/redirect-allowlist.ts` (blocks `//`, `/\`, `javascript:`/`data:`/`vbscript:` schemes, and any absolute URL to a non-allowlisted origin) are pure functions, real and fully tested without needing a live request context.

## 4. Tests (§27/§28/§29)

### 4.1 Real database tests (`scripts/db-tests/auth-identity.sql`, via `pnpm run db:test`)

7 scenario groups: idempotent linkage (a repeated invite returns the original row, exactly 1 row exists); **one identity linked to two tenants** (proves the schema does not assume single-tenant membership); **cross-tenant isolation** (a second identity's linkage to one tenant is invisible when querying another tenant's linkages); activation/revocation lifecycle with both transitions recorded in history; terminal-state enforcement (revoked cannot be revived); revoking a non-existent linkage fails cleanly (`no_data_found`, not silent); the same anon/authenticated schema-privilege defense-in-depth `PLT-105`/`106` established.

### 4.2 TypeScript unit tests

`lib/auth/session-cookie-options.test.ts` (5 tests): `httpOnly`/`sameSite`/`path` are always fixed; `secure` is environment-driven (true in production, false otherwise so local HTTP development still works); default and override `maxAge`.

`lib/auth/redirect-allowlist.test.ts` (9 tests): same-origin relative paths accepted; empty/whitespace rejected; **both classic open-redirect bypasses** (`//evil.example` protocol-relative, `/\evil.example`/`\evil.example` backslash) rejected; `javascript:`/`data:`/`vbscript:` schemes rejected (case-insensitively); an absolute URL to a non-allowlisted origin rejected; an absolute URL to an *explicitly* allowlisted origin accepted; a non-URL, non-path string rejected with the correct reason code.

`server/contracts/auth/identity.test.ts` (5 tests), `server/queries/auth-identity.test.ts` (3 tests), `server/mutations/auth-identity.test.ts` (4 tests): input validation (a fresh link can never start `revoked`; revocation requires a non-empty reason), row mapping, exact RPC/query params, error wrapping, and the empty-result-is-not-an-error case for a lookup.
