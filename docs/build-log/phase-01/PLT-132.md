## 2. Scope (§7/§11/§12)

Prompt 132 §11 allowed scope: "Job queue schema/migrations/service/worker/contracts/tests/docs/runbook and isolated example," 5-15 files, 1-3 migrations. §12 forbidden: "External queue replacement, unmeasured worker split, domain batch implementation."

- **This task widens `app.jobs`, it does not create it** — `PLT-131`'s own migration header explicitly disclosed that its `job_type` CHECK (`'import','export'` only) would be widened here. Research (RPD-012, `01_MODULE_DEPENDENCY_MAP.md:58`, `14_REQUIREMENT_PHASE_TRACEABILITY.md:102`) confirmed this task is `JOB`'s own named owning capability, and that "PostgreSQL durable queue is the initial background-job mechanism" (RPD-012, ratified) is the binding technology decision — no external queue is introduced, matching §12's own prohibition.
- **The eight widened `job_type` codes are the real, sourced use-case list** `02_CANONICAL_DATA_FLOW_MAP.md` §5.3 names verbatim (report generation, notification batch, webhook retry, document/PDF generation, dashboard refresh, loyalty expiration, recurring billing, integration sync) — **no business logic for any of them is implemented here**, only the generic queue mechanics (enqueue/claim/heartbeat/complete/fail/DLQ/requeue) a future domain-specific worker for one of these types would sit on top of, matching §12's "no domain batch implementation" prohibition exactly. Proven in this checkpoint's own db-test using synthetic non-import/export job rows.
- **`app.jobs.locked_by`/`locked_until` are wired to a real atomic claim function for the first time** (`app.claim_next_job()`), exactly as `PLT-131`'s migration header disclosed was deferred to this task. `SELECT ... FOR UPDATE SKIP LOCKED` is PostgreSQL's own standard, documented-since-9.5 mechanism for "many workers polling one queue, no two may claim the same row" — genuinely provable without a live multi-process worker fleet, since `SKIP LOCKED`'s row-level locking semantics are a documented PostgreSQL guarantee, not something this migration invents. A stale lease (an `in_progress` row whose `locked_until` has passed) is structurally eligible for reclaim by the same claim query — the real crash-recovery mechanic §23 requires.
- **Exponential backoff with jitter is a real, tested, disclosed construction, ratified in a new ADR.** No numeric formula exists anywhere in the architecture corpus or its own reusable job template (`60_BACKGROUND_JOB_TEMPLATE.md`) for backoff base/multiplier/cap, jitter algorithm, or lease duration — confirmed via this checkpoint's own research, and no existing `ADR-CAND-ARCH-0NN` candidate covers this dimension either. **`ADR-0013` newly ratifies a freshly-minted `ADR-CAND-ARCH-028`** (equal-jitter exponential backoff: base 30s, 2x multiplier, capped at 3600s; 300s default worker lease), the same "reasoned default absent blueprint-mandated numbers" class `ADR-0002`/`ADR-0004`/`ADR-0011`/`ADR-0012` already used.
- **`app.event_logs` is built here, closing a three-document-deep deferral chain** — `05_DATABASE_SCHEMA_WORKSTREAM.md` line 112 names it as the outbox-pattern table for async job/webhook dispatch; `PLT-116`'s own migration header explicitly deferred it to `PLT-129..132`; `PLT-130`'s build log re-confirmed the deferral lands specifically here. `app.append_event_log()` is the write path; `app.dispatch_event_as_job()` is the real outbox-drain mechanic (idempotent — re-dispatching an already-dispatched event returns the same linked job). The live scheduler that would call it periodically is disclosed `NOT_RUN` (no cron/poll infrastructure exists anywhere in this repository yet).
- **`app.jobs`' existing `authenticated`-scoped RLS policy (`PLT-131`) is deliberately left unchanged, not tightened to `service_role_only`.** This checkpoint's own research found `06_RLS_RBAC_WORKSTREAM.md` names `jobs` under its `service_role_only` policy family, conflicting with `PLT-131`'s already-shipped, already-tested direct-table `authenticated` SELECT policy. Retroactively tightening RLS now would be a breaking behavior change to already-verified functionality with no requesting need — disclosed as an intentional continuation of `PLT-131`'s own precedent, not an oversight. `app.event_logs` instead follows the literal `append_only_ledger` family text exactly (zero `authenticated` grant, matching `app.audit_logs`' precedent).
- **`app.cancel_import_export_job`/`app.acknowledge_job_cancellation`/`app.record_job_failure`/`app.requeue_dead_letter_job` (all `PLT-131`) are reused directly for the generic job lifecycle, not forked** — despite their import/export-flavored names, none of their bodies ever inspected `job_type`. Three are extended via `CREATE OR REPLACE FUNCTION` (lease release, `next_attempt_at` scheduling/clearing) with identical signatures and identical exception message prefixes preserved verbatim, so `PLT-131`'s own already-passing db-test assertions remain valid unmodified — proven directly by re-running `scripts/db-tests/import-export.sql` unmodified against this checkpoint's migration before writing any new test.
- **Threshold-triggered worker separation is correctly NOT decided here** — `08_API_INTEGRATION_WORKSTREAM.md:170` explicitly states its own document "does not lower that threshold decision," and §24's own business rule requires "separate workers require measured threshold ADR." No live traffic exists to measure against; the job queue remains in-process, matching MVP physical architecture. Disclosed as genuinely open, not resolved by `ADR-0013`.

## 8. Errors / recovery / risks / issues

One real defect was found and fixed during this checkpoint's own db-test authoring, not in the migration: the `app.acknowledge_job_cancellation` (extended) scenario group enqueued a `notification_batch` job at default priority (0) and called `app.claim_next_job()` expecting to claim it — but an earlier scenario group (`app.enqueue_job`) had deliberately left a *different* `notification_batch` job unclaimed at priority 5 (created only to prove idempotency, never cleaned up). `claim_next_job`'s own priority-ordering (working exactly as designed) claimed that older, higher-priority leftover job instead of the one this later scenario group actually meant to test — a test-isolation gap across scenario groups sharing a `job_type`, not a queue-mechanics bug. Fixed by giving the later group's job priority 100 (deterministically wins regardless of any leftover) and adding an explicit assertion that the claimed job's id matches the one just enqueued, so a similar cross-group leak would fail loudly in the future rather than silently claim the wrong row. The core queue-correctness guarantees this checkpoint gives the closest scrutiny — atomic claim ordering, stale-lease reclaim, live-lease non-reclaim, backoff-window gating — all proved correct on the first `db:test` run once the test fixture was fixed.

No new issue opened. Real, disclosed `NOT_RUN`: any live worker process actually calling `claim_next_job`/`heartbeat_job`/`complete_job` in a real polling loop (no such process exists anywhere in this repository yet — Prompt 132 §15 itself defers a business job UI to a later task, and no `workers/` process boundary exists, correctly deferred per `08_*.md:170`'s own "does not lower that threshold decision"); a live scheduler calling `app.dispatch_event_as_job()` periodically to drain `app.event_logs` (no cron/poll infrastructure exists anywhere in this repository yet); genuine multi-connection concurrent claims racing against each other (proven structurally via `SELECT ...
