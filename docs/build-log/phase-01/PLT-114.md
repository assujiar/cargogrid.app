## 3. Implementation (§20)

### 3.1 Resource/field IDs, sensitivity, record policy condition model (task 1)

`supabase/migrations/20260716110430_create_field_record_access.sql`: `app.can_access_record(auth_user_id, tenant_id, owner_user_id, shared_org_unit_ids, customer_account_ref)` composes three already-`VERIFIED` primitives — `app.has_active_tenant_membership()`/`app.is_supreme_admin()` (`PLT-113`), `app.users.org_unit_id` (`PLT-110`) for team/branch sharing, and `app.principal_memberships`' `customer_user` layer (`PLT-108`) for customer-account scope — rather than inventing new schema. Tenant membership is a hard prerequisite that fails closed regardless of any other match (proven in §4.1's IDOR-style cross-tenant test).

### 3.2 Server policy evaluator/projection/input whitelist and record-scope guard (task 2)

`app.mask_email()` (deterministic redaction, e.g. `j***@example.com`) and `app.users_directory` (the field-masked projection view) are the "projection" half. **A real, empirically-discovered correction during authoring, not a design left untested:** the first draft used `revoke select (email) on app.users from authenticated`, which turned out to be a no-op — Postgres column-level and table-level ACLs are additive, not layered with override semantics, so a column-level `REVOKE` cannot carve an exception out of a broader table-level `GRANT` `PLT-113` already issued. Proven empirically against a disposable debug database (`has_column_privilege()` still returned `true` after the revoke), then fixed to the documented-correct pattern: revoke the table-level grant entirely, then re-grant `SELECT` on an explicit column list that omits `email`. Verified working in §4.1.

"Input whitelist" (mass-assignment protection) is **already structurally guaranteed**, not new code this checkpoint had to add: every mutation function in this repository (`PLT-105`–`111`) takes an explicit, named parameter list — never a generic "apply these fields" pattern — and every Zod contract's default (non-`.passthrough()`) `.parse()` strips unknown keys before the mutation layer ever reads a named field off the result. §4.2 adds one test proving this holds, confirming rather than inventing the guarantee.

### 3.3 Representative REST/GraphQL/search/export/file paths and safe UI metadata (task 3)

`server/queries/field-access.ts` (`canAccessRecord()`, `listUserDirectory()`) is the representative adapter — `email_masked` on every `app.users_directory` row is exactly the "safe UI metadata" Prompt 114 §15 asks for ("Render hidden/masked/read-only/denied states from server metadata, not local guesses"). **Real, disclosed `NOT_RUN`:** wiring into any live REST/GraphQL/search/export/file entry point — none exists yet in this repository, the same disclosed pattern every prior "no live consumer" capability this session used.

**A second real, empirically-discovered correction during authoring:** the view's first draft called `app.evaluate_permission()` (`PLT-112`) directly. Since that function is `SECURITY INVOKER` (the default) and its own internal reads (`app.permissions`, `app.role_version_permissions`, ...) are deliberately `service_role`-only per `PLT-113`'s own scope decision, calling it from `authenticated`'s context failed with `permission denied for table permissions` — proven by running the test, not assumed. Rather than widen `PLT-113`'s deliberate catalogue-table scope decision (which would have been scope creep unrelated to this checkpoint's own purpose), a narrow `SECURITY DEFINER` wrapper, `app.has_view_personal_data()`, was added instead — its elevated privilege context carries through to the `SECURITY INVOKER` `evaluate_permission()` it calls internally (proven empirically to work once added), while `authenticated` itself never needs direct catalogue-table access.

## 8. Errors / recovery / risks / issues

Two real bugs found and fixed during authoring, both already documented in §3.2/§3.3 with the empirical evidence that surfaced each one (a disposable debug database for the column-ACL question; the actual failing `db:test` run for the view's permission-check question) — recorded here as the summary: (1) a column-level `REVOKE` cannot override a prior table-level `GRANT` in Postgres (additive ACL semantics, not layered) — fixed by revoking the table grant and re-granting an explicit column list; (2) a view calling a `SECURITY INVOKER` function needs the *querying* role to hold every privilege that function's own body needs, not just privilege on the view itself — fixed with a narrow `SECURITY DEFINER` wrapper scoped to exactly one question. Two additional test-authoring mistakes, also fixed before the gate suite: a self-escalation trip-up when the test's own actor and assignee were the same identity for a protected-permission grant (fixed by using a different actor), and a service-role row-count assertion that assumed 5 users when only 4 existed at that point in the shared `db:test` run (fixed to the correct count). No new issue opened. Real, disclosed `NOT_RUN`: live wiring of either mechanism into a REST/GraphQL/search/export/file entry point (§3.3 — no such surface exists yet); `app.can_access_record()` proven only by direct call, never by a live RLS policy on a real `owner_user_id` table (§2 — no such table exists in Phase 1).
