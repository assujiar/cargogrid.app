## 2. Scope (§7/§11/§12)

Prompt 134 §11 allowed scope: "PostGIS extension/spatial primitive migrations/helpers/types/tests/docs and isolated example," 5-15 files, 1-3 migrations. §12 forbidden: "TMS routing/geofence features, live map provider integration, broad coordinate backfill."

- **No standalone spatial table is created, structurally, not by omission.** `docs/architecture/05_DATABASE_SCHEMA_WORKSTREAM.md` §6/line 108 already resolved this exact question: "spatial columns (`geography(Point)`/`geography(LineString)`) added directly to `shipments`/`shipment_milestones`/`warehouses` rather than a separate geo table, since location is an attribute of those entities, not a new canonical entity." Those tables do not exist yet (Operations is Phase 3) — this checkpoint's own deliverable is the extension plus reusable validation/serialization/query-limit helper functions a future Phase 3 migration will call, not a table. The one representative example Prompt 134 §20 task 3 requires is proven entirely inside `scripts/db-tests/postgis.sql` using a `pg_temp` table created and dropped within that script alone — never a permanent migration artifact, the same "prove it in the db-test, not in permanent schema" discipline `PLT-122`/`124` already applied, taken one step further here since even a structural example table would contradict `05_*.md`'s own "not a new canonical entity" ruling.
- **`geography`, not `geometry`, is the only spatial column type this convention endorses** — already resolved by `05_*.md` line 108, and this checkpoint's own direct testing against PostGIS 3.4.2 confirmed *why*: casting any non-4326-SRID geometry to `geography` raises `Only lon/lat coordinate systems are supported in geography` — SRID 4326 (WGS84) is therefore structural to the type itself, not a convention that could silently drift.
- **A real, concrete defect class found by this checkpoint's own testing, not hypothesized**: PostGIS's own `geometry -> geography` cast does **not** reject an out-of-range latitude/longitude — it silently *coerces* (clamps) the value into `[-180 -90, 180 90]` (`ST_MakePoint(0, 95)::geography` succeeds with a `NOTICE`, proven directly). Left unguarded, this is exactly the "silent axis swap" Prompt 134 §19/§23 forbid. `app.geojson_point_to_geography()` explicitly range-checks and rejects before ever constructing a geography value — the real, tested mitigation.
- **`ADR-0014` newly ratifies a freshly-minted `ADR-CAND-ARCH-029`** (this checkpoint's own research confirmed no existing candidate covers PostGIS version/SRID/query-radius numeric thresholds) — PostGIS `3.4.x`, SRID `4326` (restating what `05_*.md` line 108 already implies as a concrete number), and a 500,000-meter (500km) bounded-radius query cap, the same "reasoned default absent blueprint-mandated numbers" class `ADR-0002`/`ADR-0004`/`ADR-0011`/`ADR-0012`/`ADR-0013` already used.
- **The local sandbox and CI environment both needed a real, disclosed environment change**: this Postgres 16 sandbox had no PostGIS package installed at all (`pg_available_extensions` returned zero `postgis*` rows before this checkpoint) — installed via `apt-get install postgresql-16-postgis-3` (3.4.2), confirmed working by direct `CREATE EXTENSION`/`PostGIS_Version()` smoke test before writing any migration. CI's `db` job previously used a plain `postgres:17` service image (no PostGIS) — updated to `postgis/postgis:17-3.4` (`.github/workflows/ci.yml`), the official PostGIS image pairing with this repository's own already-pinned Postgres major version (`ADR-0004`), so `create extension postgis` has the extension's control/library files available in CI the same way it now does locally.
- **No privileged bulk-mutation function exists in this migration** (§18's own "audit impact... where applicable" qualifier) — every function is a pure, `stable`/`immutable` validator/serializer with no table to write to yet, so there is nothing to self-audit here; a future migration that adds a real `geography` column to a real tenant-scoped table inherits that table's own RLS/audit posture automatically (a new column, not a new table).

## 3. Implementation (§20)

### 3.1 Database

`supabase/migrations/20260722090000_enable_postgis_spatial_foundation.sql` (31st migration). `create extension if not exists postgis;`. New functions: `app.postgis_max_query_radius_meters()` (the governed 500,000m cap, single source of truth), `app.validate_geography_point()` (structural point validity — non-null, non-empty, `GeometryType = 'POINT'`, `ST_IsValid`, intended as a future domain table's own CHECK constraint), `app.geojson_point_to_geography()` (GeoJSON → `geography(Point,4326)`, explicit range-rejecting, RFC 7946 `[longitude, latitude]` axis order), `app.geography_to_geojson_point()` (the inverse serializer), `app.bounded_st_dwithin()` (the one governed bounded-radius proximity primitive, rejects a non-positive or over-cap radius before ever calling `ST_DWithin`).
