## 2. Scope (§7/§11/§12)

Prompt 130 §11 allowed scope: "Shared API middleware/schema/context/errors/pagination/server/tests/docs and bounded representative operations," 5-15 files, 1-3 migrations. §12 forbidden: "Broad domain APIs, duplicated interface business logic, unbounded queries."

- **No live HTTP route or GraphQL server exists anywhere in this repository, and building one here would be out of this prompt's own scope.** `08_API_INTEGRATION_WORKSTREAM.md` line 11 discloses "zero API route, zero GraphQL schema" outright. That same document's own atomic backlog (§15, lines 242-245) splits Prompt 130's territory into three separate slices: "API/webhook/job foundation" (shared error/pagination/idempotency/correlation contracts, 8-stage flow wiring — **this checkpoint**), "GraphQL foundation" (an actual schema-first GraphQL server — a later task), and "REST v1 root" (the actual `app/api/v1/**` route scaffold — also a later task), both of the latter explicitly depending on this one. Building live routes here would be scope creep beyond what this prompt owns, not merely an environment limitation.
- **`app.api_logs` is the canonical per-request observability record** — the last of the five distinct append-only tables `05_DATABASE_SCHEMA_WORKSTREAM.md` §6 names (`audit_logs` at `PLT-116`, `file_access_logs` at `PLT-128`, `support_access_events` at `PLT-115`, and now `api_logs`; `event_logs` remains deliberately deferred to the Background Job Framework, Prompt 132, since it serves the outbox-pattern role for async job/webhook dispatch, not per-request API observability). It does not also write to `app.audit_logs` for the same event — the five-table split exists precisely so each event class has one canonical home.
- **`ADR-0012` resolves `ADR-CAND-ARCH-017`'s depth/complexity numeric-limit sub-question** (previously `BLOCKED` — "needs enforcement surface"; this checkpoint is that surface, per `08_*.md` line 244's own atomic-backlog citation): depth limit 8, complexity budget 1000 units via a per-field-type cost table — the candidate's own recommended defaults. `server/policies/graphql-complexity.ts`'s `computeQueryDepth()`/`computeQueryComplexity()`/`checkQueryDepth()`/`checkQueryComplexity()` are genuinely provable without any live GraphQL server, since scoring an already-structured field-selection tree is pure computation — this module deliberately does not parse raw GraphQL query text (no `graphql` package is installed anywhere in this repository; building a bespoke parser here would be exactly the "generic integration hub" §12 forbids).
- **The shared 8-stage access-evaluation pipeline is composed, not reinvented.** `server/policies/api-request-context.ts`'s `resolveApiRequestContext()` walks `06_RLS_RBAC_WORKSTREAM.md` §3's fixed stage order (1 entitlement → 2 tenant membership → 3 RBAC action → 4 scope [no-op, disclosed] → 5 field/record-level [conditional] → 6 status/value rule [no-op, disclosed] → 7 RLS [automatic] → 8 audit [caller's own responsibility]) using exclusively already-built, already-tested primitives (`PLT-106`'s `evaluateEntitlement()`, `PLT-108`'s `resolveAccessContext()`, `PLT-112`'s `assertPermission()` — the exact shared kernel `06_*.md` §3 already names as this pipeline's stage-3-8 authority, reused directly rather than duplicated — and `PLT-114`'s `canAccessRecord()`). This is the literal, physical form of "Neither interface is secondary or may bypass common service/access rules" (§24): the one function both a future REST handler and a future GraphQL resolver would call.
- **Pagination is cursor-based and bounded** (`server/policies/pagination.ts`) — a cursor opaquely encodes the last-seen row's own keyset ordering value, never a raw offset, matching `05_*.md` §7's keyset-mandatory convention for high-volume append-ordered tables; page size is capped at 100 (`MAX_PAGE_SIZE`), the concrete unbounded-payload guard §17 requires.
- **Idempotency/correlation are real, typed contracts** (`server/contracts/api/api.ts`) reusing the exact vocabulary already established: `CorrelationIdSchema` is the same value both an `X-CargoGrid-Request-Id` header and `app.capture_audit_event()`'s `p_correlation_id` thread through; `IdempotencyKeySchema` matches every domain table's own already-shipped `idempotency_key` column convention.
- **The error contract is real and shared** (`ApiErrorSchema`/`buildApiError()`) — the exact Tech Arch §25.6 shape (`code`/`message`/`details[]`/`request_id`/`timestamp`) applied identically to a REST body and a GraphQL error's `extensions`, the one function both interfaces would call to construct it.

## 3. Implementation (§20)

### 3.1 Database

`supabase/migrations/20260719160000_create_api_foundation.sql`. `app.api_logs` — `actor_type`/`api_key_id` map all four principal layers plus API-key/service identities onto one row shape (§26), composing `PLT-129`'s `app.api_keys` via a plain FK; `api_logs_actor_shape_check`/`api_logs_interface_fields_check` are real structural `CHECK` constraints proving an `api_key` actor always carries `api_key_id`, a `user` actor always carries `actor_auth_user_id`, and REST/GraphQL-specific columns never leak into the other interface's row. `app.record_api_request()` — the bounded adapter interface a future request-handling middleware would call exactly once per request, disclosed `NOT_RUN` for any live caller.

### 3.2 Contracts, pagination, and GraphQL complexity

`server/contracts/api/api.ts` — `ApiErrorSchema`/`buildApiError()`, `CursorPaginationParamsSchema`/`OffsetPaginationParamsSchema`/`PageInfoSchema`/`paginatedResultSchema()`, `IdempotencyKeySchema`/`CorrelationIdSchema`, `API_VERSION`, `ApiLogSchema`/`parseApiLog()`. `server/policies/pagination.ts` — `encodeCursor()`/`decodeCursor()`/`buildPageInfo()`, real base64url opaque-cursor round-tripping. `server/policies/graphql-complexity.ts` — the `ADR-0012` scorer, operating on a minimal `QueryField` selection-tree shape rather than a real GraphQL AST.

### 3.3 Shared access-evaluation pipeline and mutation

`server/policies/api-request-context.ts` — `resolveApiRequestContext()`, composing stages 1/2/3/5 exclusively from already-built primitives; stage-specific denials propagate as each stage's own already-established typed error (`AccessContextResolutionError`, `RbacDenialError`) rather than being re-wrapped, preserving their own reason codes; a new `ApiRequestContextDenialError` covers only the stages (entitlement, record-access) that had no error class of their own yet. `server/mutations/api-log.ts` — `recordApiRequest()`, the thin wrapper around `app.record_api_request()`.
