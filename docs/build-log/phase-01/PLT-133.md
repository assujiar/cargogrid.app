## 2. Scope (§7/§11/§12)

Prompt 133 §11 allowed scope: "Feature-flag schema/migrations/service/evaluator/admin slice/tests/docs," 5-15 files, 1-3 migrations. §12 forbidden: "Domain feature behavior, authorization weakening, tenant code forks."

- **A feature flag's target/version storage is not a new table family — it is `PLT-121`'s own `config_type`/`config_object`/`config_version`/`config_item` mechanism, reused directly**, exactly the "shared mechanism, not a fork" pattern `PLT-122` (workflow) and `PLT-124` (status) already established for the same Configuration Engine. `app.register_feature_flag()` mints its own dedicated `config_type` (`'feature:' || flag_key`) via `PLT-121`'s own `app.register_config_type()` — the identical reason `PLT-124`'s `app.register_status_set()` does the same: one shared generic type (`'feature'`, already seeded empty by `PLT-121`) cannot host more than one independently-versioned object per scope level.
- **Feature flags support only `global`/`tenant` scope, a deliberate, disclosed narrowing of the Configuration Engine's full 6-level precedence.** Tech Arch's own flag dimensions (environment/tenant/rollout/cohort/effective-date/kill — `scripts/feature-flags/flags.ts`, `docs/standards/FEATURE_FLAG_STANDARDS.md`) never included an org-hierarchy company/branch/role/user precedence. `app.create_feature_flag_draft()`/`app.set_feature_flag_items()` reject any other `scope_level` structurally.
- **The kill switch and environment allowlist are global-only, database-enforced non-overridable dimensions** — `app.set_feature_flag_items()` raises `check_violation` (`feature_flag_tenant_scope_no_kill_switch`) on any attempt to set `kill_switch`/`environments` at `scope_level='tenant'`. This is this checkpoint's own considered resolution of a real tension the Phase 0 evaluator's single-object precedence (kill switch checked unconditionally first) did not have to face: Platform Core's own config-engine precedence model is most-specific-wins, which would otherwise let a tenant `tenant_state='allow'` override resurrect a platform-wide kill. Proven directly in `scripts/db-tests/feature-flags.sql`'s own "kill_switch is non-overridable at tenant scope" scenario group (a tenant `allow` override published *after* a global kill still evaluates to `kill_switch`, not `tenant_override_allow`).
- **`app.evaluate_feature_flag()` is entitlement-aware but never entitlement-granting** (§16: "Flags cannot grant permission/module entitlement... "). A flag registered with a `module_code` calls `PLT-106`'s own read-only `app.evaluate_entitlement()`; an unentitled tenant is denied (`module_not_entitled`) regardless of rollout percentage. The evaluator has no code path that can widen access beyond what `PLT-106` already allows, and no code path touches `app.roles`/`app.role_assignments`/any RLS policy at all — structurally incapable of granting authorization, the same discipline `scripts/feature-flags/flags.ts`'s own Phase 0 header established, carried forward rather than weakened.
- **Bucketing is SQL-native (`md5`-based), not a port of `scripts/feature-flags/flags.ts`'s `sha256`-based `bucketFor()`.** The two are not required to produce bit-identical bucket assignments for the same tenant+flag — Prompt 133 §14 names *this* capability's own evaluator ("server evaluation authoritative") as the authority for Platform Core flags; the Phase 0 TS module is untouched and remains available for its own existing callers. Both are deterministic and evenly distributed, the actual correctness property either needs — proven directly (repeated evaluation for the same tenant+flag is identical; `app.feature_flag_bucket()` is always in `[0, 99]`).
- **No real domain flag is seeded anywhere in the migration itself** — the one safe, representative example (`platform.example_rollout`, plus `platform.entitled_rollout` for the entitlement-integration scenario) is registered and exercised entirely inside `scripts/db-tests/feature-flags.sql`, the identical "prove it in the db-test, not in permanent migration data" discipline `PLT-122`/`124`'s own headers already applied.
- **No new ADR was required.** Every numeric/structural choice this checkpoint made (the 7-value environment enum, reused verbatim from `scripts/feature-flags/flags.ts`'s already-established `Environment` type; the `md5`-based bucket function, an implementation detail disclosed in the migration's own header, not an unsourced business threshold like `PLT-132`'s backoff formula) either reuses an already-ratified source or is a mechanical implementation detail with no competing candidate in `docs/adr/README.md` §5.2's open list.

## 3. Implementation (§20)

### 3.1 Database

`supabase/migrations/20260721090000_create_feature_flags_platform.sql` (30th migration). New table: `app.feature_flags` (the flag catalogue: `flag_key` primary key, `module_code` optionally FK to `PLT-106`'s `app.entitlement_modules`). New composite type: `app.feature_flag_decision` (`enabled`, `reason`, `resolved_scope_level`, `resolved_version_id`, `evaluated_at`). New functions: `app.register_feature_flag` (Supreme-only, idempotent, mints a dedicated `config_type`), `app.create_feature_flag_draft` (validates flag/scope before delegating to `PLT-121`'s `app.create_config_draft`), `app.set_feature_flag_items` (structural validation — rollout `[0,100]`, environment subset, global-only kill_switch/environments, valid tenant_state — then delegates to `app.set_config_items`), `app.feature_flag_bucket` (deterministic `md5`-based `[0,99]` bucketing), `app.evaluate_feature_flag` (`SECURITY DEFINER`, the fixed precedence order: unknown flag → entitlement → unconfigured → kill_switch → environment_gate → tenant deny/allow → cohort_mismatch → rollout_bucket → default), `app.kill_feature_flag` (the panic-button convenience path — Supreme-only, forces `kill_switch=true` on a fresh global draft while preserving the other currently-published dimensions, then publishes immediately), `app.debug_feature_flag` (privileged explain path, authority split identical to `app.check_config_object_authority`, self-audited distinctly from the read-only evaluation it wraps). Publish/discard/rollback are **not** re-declared — `PLT-121`'s own `app.publish_config_version`/`app.discard_config_draft`/`app.rollback_config_version` operate on a flag's draft `version_id` unchanged, with no flag-specific parameter at all. Per `ERR-2026-004`: this migration carries its own explicit `revoke execute on all functions in schema app from public;` before its final grants.

### 3.2 Contracts, mutations, queries

`server/contracts/feature-flag/feature-flag.ts` (new) — `FeatureFlagSchema`/`FeatureFlagDecisionSchema`, the `FEATURE_FLAG_ENVIRONMENTS`/`FEATURE_FLAG_TENANT_STATES`/`FEATURE_FLAG_DECISION_REASONS` enums (mirroring the migration's own allowlists), and every RPC's input schema. Publish/discard/rollback intentionally reuse `server/contracts/config/config.ts`'s existing `ConfigVersion`-typed contracts directly — not re-declared. `server/queries/feature-flags.ts` (new) — `evaluateFeatureFlag` plus `FeatureFlagCache` (keyed by flag/tenant/environment/cohort-set per Prompt 133 §17, TTL-bounded, `invalidate()`-driven "prompt invalidation"). `server/mutations/feature-flags.ts` (new) — `registerFeatureFlag`/`createFeatureFlagDraft`/`setFeatureFlagItems`/`killFeatureFlag`/`debugFeatureFlag`, each invalidating a given `FeatureFlagCache` on success.

## 4. Tests (§27/§28/§29)

### 4.1 Real database tests (`scripts/db-tests/feature-flags.sql`, via `pnpm run db:test`)

13 scenario groups, using two tenants (`acmeflag`, `gizmoflag`) each with a `tenant_admin`/regular `org_user`, a global Supreme Admin, and a published entitlement package granting only `COM` to `acmeflag` (`gizmoflag` deliberately receives no entitlement assignment at all):

`app.register_feature_flag` — Supreme-only, idempotent, rejects an unknown `module_code`, mints its own `config_type`; `app.create_feature_flag_draft` — rejects an unknown `flag_key` and any scope beyond global/tenant, authority-gated (Supreme for global, `tenant_admin`/Supreme for tenant); `app.set_feature_flag_items` — global scope requires the full validated dimension set (rollout `150` rejected, unknown environment `moon_base` rejected, `tenant_state` rejected at global scope), `kill_switch`/`environments` structurally rejected at tenant scope, invalid `tenant_state` rejected; unknown-flag and never-published-flag both evaluate to safe-default-closed (`unknown_flag`/`unconfigured`); environment gate, then deterministic `0%`/`100%` rollout bucketing, with a determinism re-check and an explicit `[0,99]` bound proof; cohort targeting (`cohort_mismatch` when the caller's cohorts do not intersect the flag's, passthrough to bucketing when they do); **tenant override deny/allow winning over a `100%` global rollout**, **cross-tenant isolation** (`gizmoflag` unaffected by `acmeflag`'s own tenant-scoped override in both directions), and **the global kill switch winning over a tenant `allow` override** (the core non-overridable-kill-switch guarantee, proven end-to-end through the evaluator, not just the structural `set_feature_flag_items` rejection); **module-gated entitlement enforcement** — `acmeflag` (COM-entitled) passes through to bucketing, `gizmoflag` (no entitlement at all) is denied `module_not_entitled` even at a `100%` rollout; `app.kill_feature_flag` — Supreme-only, preserves the prior published `environments`/`rollout_percentage`/`cohorts` while forcing `kill_switch=true`, disables the flag for every tenant immediately, and `PLT-121`'s own `app.rollback_config_version` (reused unmodified) un-kills by restoring the pre-kill snapshot; `app.debug_feature_flag` — authority-gated identically to `app.check_config_object_authority`'s own split (regular `org_user` denied, `tenant_admin` denied platform-wide debug, both Supreme and the owning `tenant_admin` see the identical decision `app.evaluate_feature_flag` itself would return); every privileged/lifecycle action (`register_feature_flag`/`set_feature_flag_items`/`kill_feature_flag`/`debug_feature_flag`) self-captures a canonical `app.audit_logs` entry; schema-privilege defense in depth (`authenticated` has `SELECT` on the `app.feature_flags` catalogue only, no direct access to draft-bearing config tables; `anon` denied entirely, including `EXECUTE` on `evaluate_feature_flag`/`kill_feature_flag`/`set_feature_flag_items`).

## 8. Errors / recovery / risks / issues

One real defect was found and fixed during this checkpoint's own db-test/migration authoring: the first `app.set_feature_flag_items` call inside `app.kill_feature_flag`/`app.set_feature_flag_items`'s own generic-audit call passed the raw `jsonb_build_array(...)` item set directly as `capture_audit_event`'s `p_after_value` — but `PLT-116`'s own `app.redact_audit_payload()` calls `jsonb_each()` internally, which requires a JSON *object*, not an *array*, and raised `cannot call jsonb_each on a non-object` the first time `db:test` actually exercised the path (a real, caught-by-the-gate defect, not a hypothetical). Fixed by wrapping the item array in `jsonb_build_object('items', v_items)` before the audit call — the same shape every other Configuration-Engine-derived capability's own generic `set_config_items` audit call already uses (`jsonb_build_object('item_count', ...)`, never a bare array). Separately, this checkpoint's synthetic auth-identity UUIDs (`...-0000-000000001401`..`1404`) initially collided with `PLT-122`'s own `workflow.sql` fixture range (`db:test` applies every `scripts/db-tests/*.sql` file against the same disposable database, so `auth.users` primary-key collisions across files are a real, previously-undocumented hazard for this checkpoint's own synthetic ID choice) — fixed by moving to an unused range (`...-0000-000000005001`..`5004`), confirmed against every existing db-test file's own ID usage before re-running.

No new issue opened. Real, disclosed `NOT_RUN`: any live application/route layer actually calling `evaluateFeatureFlag`/`FeatureFlagCache` in a real request path (no such route exists anywhere in this repository yet, the same "mechanism proven, live wiring deferred" posture `PLT-121`'s own resolver and `PLT-106`'s own entitlement evaluator already carry); a live admin/portal UI consuming `app.list_config_versions` for a flag's history (Prompt 133 §15 itself defers this to a later task); `pnpm run test:e2e` (sandbox Playwright browser-binary revision skew, unrelated to this diff, same condition disclosed at `PLT-117..132`).
