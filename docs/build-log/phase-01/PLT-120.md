## 3. Implementation (§20)

### 3.1 Master registry/types, ownership policy, canonical IDs/codes (task 1)

`supabase/migrations/20260717120000_create_master_data.sql`. `app.master_types` registers what kinds of master data exist (`code`/`name`/`scope`/`owner_module_code`), seeded with the one sourced `vendor_rate` row; `app.register_master_type()` is idempotent and Supreme-Admin-only (extending the catalogue is a platform-level decision). `app.master_records` is the generic canonical-instance table — `code` is the stable natural key, `aliases` are bounded (≤20), plain-text alternate lookup labels for the *same* canonical record (Prompt 120 §24: "labels/aliases do not duplicate truth"), never a second source of truth.

### 3.2 Lifecycle/version/effective/alias/merge-safe model and service (task 2)

`app.create_master_record()` (idempotent per `(master_type_code, tenant_id, code)`, authority-gated: Supreme Admin for a `scope='global'` type's records, `PLT-115`'s `app.is_support_grant_authority()` for a `scope='tenant'` type's records), `app.update_master_record()` (real optimistic concurrency — a stale `expected_version` is rejected, never silently overwritten, the same discipline `PLT-109`'s `move_org_unit()` established), `app.deactivate_master_record()`. **`app.merge_master_records()` is the merge-safe dedupe model** (§20 task 2, §24: "referenced master deactivation/merge preserves lineage") — the source row is never deleted; it is marked `canonical_status='merged'` and permanently pointed at its survivor via `merged_into_id`, with its own former code and aliases folded into the target's alias list. Rejects merging a record into itself, across master types, across tenant scope, or where either side is not currently `active`.

### 3.3 Search/resolve/pagination/cache and dependency checks (task 3)

`app.resolve_master_record()` resolves a code *or* alias to its live canonical record, **transparently following a bounded (≤10 hop) merge chain** — resolving a since-merged code lands on the live survivor, not a dead row, proven directly. Structurally never ambiguous (code/alias uniqueness plus a code-match-first tiebreak guarantee at most one starting row) and never cross-tenant (scoped by an explicit `tenant_id` parameter, backed by the RLS policy below as a second layer). `app.search_master_records()` is real keyset-paginated substring search (§17: "no full datasets"), capped at 200 rows per call. Both are plain (non-`SECURITY DEFINER`) functions relying on RLS for isolation — the same "plain grant + RLS" posture `PLT-113` established for primary tables, since master-data lookup (unlike `PLT-117`/`118`/`119`'s evaluators) has no pre-authentication use case.

### 3.4 Initial platform masters, RLS/RBAC/import hooks, tests/audit/docs (task 4)

RLS: `app.master_types` is broadly `authenticated`-readable (platform-owned reference data, no PII); `app.master_records` composes `PLT-113`'s `app.has_active_tenant_membership()`/`app.is_supreme_admin()` with a new `OR tenant_id is null` global-visibility branch — the first table in this repository to model "some rows are visible to every tenant" rather than strict per-tenant isolation.
