## 3. Implementation (§20)

### 3.1 Supported locale/timezone/currency/terminology scope and fallback precedence (task 1)

`supabase/migrations/20260717112000_create_localization.sql`. `app.validate_locale_code()`/`app.validate_timezone_name()`/`app.validate_currency_code()` are real, bounded, sourced validators (not open IANA/ISO-4217 acceptors) wired as table `CHECK` constraints. `app.canonical_terms` is a real seeded catalogue (21 rows, sourced per §2 above); `app.validate_terminology_overrides()` rejects any key absent from it, and any value that is empty, oversized, or contains an angle bracket — enforced via a `BEFORE INSERT OR UPDATE` trigger (not a `CHECK` constraint, since validation requires a cross-table lookup, `STABLE` not `IMMUTABLE`).

### 3.2 Typed catalogue/resolver/format utilities and versioned tenant overrides (task 2)

`app.tenant_locale_versions` mirrors `PLT-117`'s `app.tenant_brand_versions` draft/publish/rollback discipline exactly — `app.create_tenant_locale_draft()` (idempotent, authority-gated via `PLT-115`'s `app.is_support_grant_authority()`), `app.set_tenant_locale_config()`, `app.publish_tenant_locale_version()` (supersedes the prior published version), `app.discard_tenant_locale_draft()`, `app.rollback_tenant_locale_version()` (clones a snapshot into a new version, never mutates history). `app.resolve_tenant_locale()` resolves a tenant's effective config — its own published version, or the Indonesia-first platform default (`id`/`Asia/Jakarta`/`IDR`, a reasoned inference from ratified RPD-016 sequencing, disclosed as distinct from `DESIGN_SYSTEM.md` §3's still-genuinely-open CargoGrid brand-color decision).

### 3.3 Shared shell/design-system/auth/platform integration, notification template hooks (task 3)

**The real three-tier fallback chain** (§22: "user→tenant→platform default") is `app.resolve_locale_context()`, composing `app.resolve_tenant_locale()` with an optional per-user preference lookup — three new nullable columns added to `PLT-110`'s already-shipped `app.users` via additive `ALTER TABLE` (not an edit to an applied migration; AGENTS.md's "add a new migration" rule applied literally to a table alteration, the same category of change as `PLT-119`'s own new table, just against an existing one). Each of locale/timezone/currency resolves independently (a user can override just their own timezone while still inheriting the tenant's locale/currency) — proven directly in `db:test`. No shared UI shell/notification-template surface exists yet in this repository (the same disclosed scope boundary `PLT-117`/`118` already established) — `server/queries/localization.ts`'s `resolveLocaleContext()` is the typed entry point a future shell/template-rendering surface calls.

### 3.4 Missing-key/fallback/injection/SSR/cache/accessibility tests, docs (task 4)

"Missing key" (§25) is structurally inverted from a traditional i18n bundle: since no UI copy-translation bundle exists yet (no shared UI shell, per §2's disclosed boundary), this checkpoint's real, testable form of "missing/unused keys detectable" is that an override for an *undefined* `canonical_terms` code is rejected outright at write time — a typo or invented key can never silently persist, proven directly. SSR/hydration-specific tests are deferred to the future UI slice for the same reason (nothing renders yet); cache determinism is proven via `LocaleContextCache`'s unit tests (per-`(tenant,user)` keying, TTL, explicit invalidation).
