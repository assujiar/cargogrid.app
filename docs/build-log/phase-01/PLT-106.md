## 3. Implementation (¬ß20)

### 3.1 Taxonomy and versioned model (task 1)

`supabase/migrations/20260716094432_create_entitlements.sql`: `app.entitlement_modules`/`app.entitlement_features` ‚Äî seeded with the real, already-`VERIFIED` catalogue from `01_*.md` ¬ß3.2 (9 modules, 41 features) rather than fabricated placeholder data. `app.entitlement_packages` ‚Äî versioned (`code`+`version` unique), `granted_modules text[]` + `feature_limits jsonb` (numeric quota per feature; JSON `null` = unlimited), `status` (`draft ‚Üí published ‚Üí retired`); a published version is treated as immutable by convention today (no trigger yet ‚Äî disclosed, not fabricated; revisit at `PLT-138` hardening if needed).

### 3.2 Tenant assignment and transitions (task 2)

`app.tenant_entitlements`: `trial ‚Üí active ‚Üí suspended ‚Üí active ‚Üí expired|cancelled` (terminal), trigger-enforced identically to `PLT-105`'s `app.tenants` pattern. A partial unique index (`status in ('trial','active','suspended')`) makes more than one non-terminal assignment per tenant structurally impossible, not merely conventional. `app.assign_entitlement()` supersedes any existing non-terminal assignment (cancels it, records both the supersession and the new assignment in `app.entitlement_assignment_history`) rather than erroring ‚Äî Prompt 106 ¬ß20 task 2's "renewal" case. `app.tenant_entitlement_overrides` is a real, reasoned, mandatorily-expiring data row (never an application-code branch keyed on tenant id ‚Äî Prompt 106 ¬ß24).

### 3.3 The evaluator (task 3)

`app.evaluate_entitlement(tenant_id, module_code, feature_code?, as_of?)` returns a typed `app.entitlement_decision` composite (`allowed`, `reason`, `limit_value`, `package_code`, `evaluated_at`). Deterministic precedence, highest first: (1) a non-expired, non-revoked override always wins in either direction; (2) otherwise the tenant's current published package grant; (3) otherwise deny. Fail-closed on every unknown/missing/stale case (¬ß23/¬ß25): no assignment, a draft/retired package, and ‚Äî the one case worth naming explicitly ‚Äî **a trial past its own `trial_ends_at` is treated as expired at evaluation time even though its row status is still literally `trial`**, since no lifecycle job exists yet (`PLT-132`, Background Job Framework, is a later capability) to have transitioned it. This is stage 1 only (`06_*.md` ¬ß3) ‚Äî the function grants no role/record/field access on its own (¬ß16/¬ß24), structurally: it reads only `app.tenant_entitlements`/`app.entitlement_packages`/`app.tenant_entitlement_overrides`, nothing auth-adjacent.

### 3.4 Server guard primitive and cache (task 4)

`server/queries/entitlement.ts` (read path, per `04_REPOSITORY_TARGET_STRUCTURE.md`'s queries-vs-mutations split): `evaluateEntitlement()` wraps the RPC; `EntitlementCache` is a per-tenant cache with **explicit** `invalidate(tenantId)` (Prompt 106 ¬ß17/¬ß25: "Cache invalidates on assignment/version/lifecycle change" ‚Äî not a TTL-only hope) plus a bounded TTL as defense in depth, the identical discipline `scripts/feature-flags/flags.ts`'s `FlagCache` established at Phase 0 for a genuinely different concept (release/rollout control vs. subscription entitlement ‚Äî kept as two separate modules, not merged, per `07_*.md` line 73's own distinction). `server/mutations/entitlement.ts` (`assignEntitlement()`/`transitionEntitlementStatus()`) calls `cache.invalidate(tenantId)` after every successful write ‚Äî proven, not assumed (¬ß4.2 below).
