## 2. Scope (§7/§11/§12)

`00_PLATFORM_CORE_WBS.md` row 9: allowed scope is "RBAC evaluator/guards/cache/schema integration/tests/docs"; DB impact is "Indexed assignment/scope evaluation; no RLS replacement." Forbidden: broad domain integration, RLS weakening, role-name authorization, full portals (§12) — no RLS policy is written here (`PLT-113`'s scope), no business-domain table is touched, and the evaluator structurally never references `app.roles.name` anywhere in its logic (proven in §4.1).

This is stage 3 of the 8-stage access flow (`docs/architecture/06_RLS_RBAC_WORKSTREAM.md` §3: "3 RBAC action: role allows action") — the first Platform Core capability that actually *reads* `PLT-111`'s role/permission bindings to permit or deny anything; `PLT-111` only modeled them.

**Disclosed, deliberate combination policy (task 1's "define evaluator inputs, permission/scope precedence and deny behavior"):** multiple granted roles combine additively (union) — there is no "explicit deny" role assignment concept anywhere in `PLT-111`'s schema, unlike `PLT-106`'s entitlement overrides which did carry an explicit deny direction. Since RBAC roles only ever grant, "any granting role permits" is the only coherent combination policy, and it is what this checkpoint implements and tests (§4.1).

**Supreme Admin's real, sourced exception:** RPD-022's disclosed absolute-CRUD exception (`docs/architecture/06_RLS_RBAC_WORKSTREAM.md` §8) bypasses the role/permission lookup entirely for a live global `supreme_admin` `principal_membership` (`PLT-108`) — real, grounded, and proven in §4.1, not an invented shortcut.

**Disclosed, bounded limitation (a direct consequence of `PLT-111`'s own design, surfaced here rather than silently accepted):** when a role's newest version is published, `PLT-111`'s `publish_role_version()` archives the *role version* it supersedes but never touches `app.role_assignments` rows still pointing at that now-archived version. This checkpoint's evaluator therefore denies those "stale" assignments (proven in §4.1, matching Prompt 112 §23's own "stale ... permission fails closed" requirement) — but this also means every user assigned to a role's prior version silently loses access on republish until someone explicitly re-assigns them to the new version. No auto-carry-forward exists anywhere in this repository. A future capability could add one; this checkpoint deliberately does not, to avoid scope creep into `PLT-111`'s already-`VERIFIED` files.

## 3. Implementation (§20)

### 3.1 Evaluator inputs, precedence, deny behavior (task 1)

`supabase/migrations/20260716104519_create_rbac_evaluator.sql`: `app.evaluate_permission(auth_user_id, tenant_id, resource_module_code, action)` returns `app.rbac_decision` (`allowed`, `reason`, `permission_id`, `role_id`, `role_version_id`, `evaluated_at`) — always a decision record, never a raised exception, matching `PLT-106`'s `evaluate_entitlement()` shape (a caller needs the `reason` even on denial for UI messaging, unlike `PLT-108`'s resolver which always either resolves or raises). Deny paths are distinguished by reason (`unknown_permission`/`no_active_assignment`/`no_granting_role`), each proven separately in §4.1 — never a single undifferentiated `false`.

### 3.2 Shared authorization service/guards and cache invalidation (task 2)

`server/queries/rbac.ts` (`evaluatePermission()`, `RbacDecisionCache`) mirrors `PLT-106`'s `EntitlementCache` explicit-invalidation pattern exactly, deliberately a separate cache instance — RBAC decisions are a distinct concept from `TEN-IAM` entitlement. `server/policies/permission-check.ts` (`assertPermission()`, throwing `RbacDenialError`) is the shared guard — this is the **exact** module path `docs/architecture/06_RLS_RBAC_WORKSTREAM.md` §3 already names ("the shared `server/policies/permission-check.ts` module"), reused verbatim rather than invented under a new name.

### 3.3 Representative server/API/job/file path integration (task 3)

`assertPermission()` is the one representative adapter — resolves a decision (cached if supplied) and fails closed by throwing, never returning a value a caller could mistakenly treat as "proceed." **Real, disclosed `NOT_RUN`:** wiring this guard into any live REST/GraphQL/server-action/job/file entry point — no such surface exists yet in this repository (Phase 1 has not reached `PLT-129`–`132`, API/Jobs, or `PLT-135`/`136`, the Admin portals), the same disclosed pattern `PLT-108`'s `resolveAccessContext()` already established for this exact reason. **Cache-invalidation caller contract, explicitly disclosed rather than silently assumed:** this checkpoint does not modify any `PLT-111` file to call `cache.invalidate()` automatically after a role/assignment mutation — a future caller composing both capabilities (e.g. an admin route that calls `assignRole()` then `assertPermission()`) is responsible for invalidating first, or omitting the cache entirely for a guaranteed-fresh check. Documented here so `PLT-113`+ does not have to rediscover it.

See §4. No new observability instrumentation was added — `scripts/observability/logger.ts` (`PH0-93`) is already the shared logging module any future route wiring `assertPermission()` would call around a thrown `RbacDenialError`; no new logging surface is needed at the evaluator level itself.

## 4. Tests (§27/§28/§29)

### 4.1 Real database tests (`scripts/db-tests/rbac-enforcement.sql`, via `pnpm run db:test`)

12 scenario groups: a granted permission evaluates `allowed=true` with the granting role/version traced in the decision; an ungranted permission for the same identity denies with a reason distinct from "no assignment at all"; an identity with zero assignments denies with its own distinct reason; an unknown module/action pair fails closed rather than raising; a revoked assignment denies even though its role version is still published; **a stale assignment — still `active`, but pointing at a role version a later publish has since archived — denies, then access is restored once re-assigned to the newly published version** (proving the denial was caused by staleness specifically, not an unrelated break); **a role literally named `SuperAdminBypassAllChecks`, bound to an unrelated permission, still denies the permission it does not actually grant** — the concrete structural proof that role names never authorize; multiple granted roles combine additively (two distinct roles' distinct grants both independently allow); cross-tenant isolation (a grant in one tenant never applies when evaluated against another); the Supreme Admin exception bypasses the lookup entirely with zero role assignments; the same anon/authenticated schema-privilege defense-in-depth `PLT-105`–`111` established (`anon` cannot even call `evaluate_permission`).

### 4.2 TypeScript unit tests

`server/contracts/rbac/rbac.test.ts` (4 tests): allowed/denied decision row mapping (including null role/version references on denial); `asOf` stays optional; empty `resourceModuleCode`/`action` rejected at the contract boundary.

`server/queries/rbac.test.ts` (6 tests): exact snake_case RPC params; database errors and a null/non-object success response both wrap into a typed `RbacEvaluationError` rather than fabricating a decision; **the cache serves a repeated identical query without a second RPC call, until `invalidate()` is called**; a distinct permission for the same identity is cached separately; an expired entry (past its TTL) is treated as a cache miss.

`server/policies/permission-check.test.ts` (3 tests): `assertPermission()` returns the decision when allowed; throws a typed `RbacDenialError` carrying the reason and input when denied — never a silent `false` a caller could ignore; an optional cache is passed through to the underlying evaluator (a second identical call makes no second RPC call).

## 8. Errors / recovery / risks / issues

Two in-authoring fixes, neither surviving to the gate suite: (1) the "stale assignment" psql test's re-assignment-restores-access follow-up initially published a brand-new role version with zero bound permissions (the `set_role_version_permissions()` call was missing before `publish_role_version()`), so the "restored" assertion failed — fixed by binding the permission before publishing; (2) this checkpoint's own `rbac-enforcement.sql` originally reused the exact role name `'Finance Approver'` that `PLT-111`'s `role-permission.sql` also creates (in a different tenant) — since all `db:test` files share one disposable database in sequence, and `role-permission.sql`'s own queries do not filter by `tenant_id` when looking up that name, the shared name caused a real "more than one row returned by a subquery" error once both files ran in the same run — fixed by renaming this checkpoint's role to `'RBAC Finance Approver'`, a naming discipline this build log now records so a future capability's test file does not repeat the collision. No new issue opened. Real, disclosed `NOT_RUN`: live guard wiring into any REST/GraphQL/job/file entry point (§3.3 — no such surface exists yet).
