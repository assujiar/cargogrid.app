## 2. Scope (§7/§11/§12)

Prompt 128 §11 allowed scope: "Document/file metadata schema/migrations/service/storage/scan adapter/upload UI/tests/docs/runbook," 5-15 files, 1-3 migrations. §12 forbidden: "Public bucket/default URLs, scan bypass, real sensitive files, domain document workflows."

- **No real storage or malware-scan provider is ever called** (§12/§16/§17). No live Supabase project exists in this sandbox (the same disclosed constraint `PLT-107`/`118` already recorded). `app.files.storage_path` is a real, unguessable, server-generated object key (`tenant_id` + `document_type_code` + a fresh random uuid) — deliberately never derived from the client-supplied filename, the structural reason path traversal via that vector is impossible. `app.record_file_scan_result()` is the real, tested, bounded adapter interface a future scan-provider webhook/job would call; this migration never calls it against a fabricated scanner itself.
- **A document *type* is not a new registry mechanism** — `app.register_document_type()` mints a dedicated `document:<code>` config_type via `PLT-121`'s own `app.register_config_type()`, the same per-instance registry composition `PLT-124/125/126/127` all already established. Unlike those four, `PLT-121`'s original 10-type Module Adoption Map never seeded a generic `'document'` placeholder in the first place — there is no unused template to disclose here, only fresh per-type registrations. Each tenant then drafts/publishes its own rules (allowed MIME types, max size, retention class, default classification, legal-hold eligibility) for a given type code.
- **This entire capability is server-mediated only** (a deliberate, disclosed design choice, not an oversight). Every mutating function is granted to `service_role` only, never `authenticated` — real signed-URL issuance inherently requires storage credentials only server code holds, so there is no legitimate direct-client RPC surface the way `PLT-127` had genuine self-service actions. Consequently **none of this migration's functions needs `SECURITY DEFINER`**: `service_role` already holds direct table grants on every table these functions touch, so plain `SECURITY INVOKER` (the default) resolves correctly. Authenticated read access is direct-table RLS, not a resolver function — the "shared business data" precedent (`PLT-113/114/120`), not the "personal inbox" precedent (`PLT-127`), since a file is inherently shared, record-scoped data.
- **`classification` is wired to `scripts/data-classification/registry.ts`'s `SENSITIVITY_LEVELS` scale** (`docs/standards/DATA_CLASSIFICATION_STANDARDS.md` §8's named gap: "files.classification column values wired to this registry — No files table exists yet"). The wiring is a literal CHECK-constraint match against the five levels, not a static `ClassificationEntry` — a file's classification is a dynamic per-row value chosen at upload time, not a fixed schema-field sensitivity the `ClassificationEntry` model classifies once at author time. `app.classification_level_rank()` mirrors `registry.ts`'s `LEVEL_ORDER` exactly (the SQL/TypeScript boundary cannot literally share code) so a file's classification can never be set weaker than its document type's configured default.
- **Scope boundary**: only `app.files` and `app.file_access_logs` are created here. `05_DATABASE_SCHEMA_WORKSTREAM.md` §6 groups `notifications, files, audit_logs, event_logs, api_logs, file_access_logs, support_access_logs` into one wave, but `notifications`/`audit_logs`/`support_access_events` already shipped at `PLT-127`/`116`/`115`; `event_logs`/`api_logs` are generic platform event/webhook infrastructure with no file-specific reason to build here and are left to their own future owning prompt.
- **The malware-scan gate and the record/sensitivity-access gate are independent**, composed together only inside `app.authorize_file_access()`. RLS on `app.files` deliberately does **not** reference `malware_scan_status` at all — metadata/status visibility ("pending scan is visible with safe status," §22) is a separate concern from content-delivery authorization. A signed URL is withheld from anyone but the uploader while scan status is not `clean`; an `infected` result blocks even the uploader (quarantine).
- **Legal hold overrides deletion, never the reverse**; `app.set_file_legal_hold()` is restricted to support/Supreme authority, not any active tenant member.

## 3. Implementation (§20)

### 3.1 Registry, structural validation, and resolution

`supabase/migrations/20260719140000_create_document_file_engine.sql`. `app.document_types` (registry). `app.validate_document_type_definition()` — 6 distinct named exceptions covering a bounded MIME allowlist (deliberately excludes anything executable/script-like), a 100 MB structural size ceiling, a real retention-class enum, a real classification enum, and a required boolean legal-hold-eligibility flag. `app.publish_document_type_definition()` composes `PLT-121`'s own `app.publish_config_version()`. `app.classification_level_rank()` mirrors `registry.ts`'s ordering. `app.resolve_document_type_definition()` resolves a tenant's currently published rules for a type code, raising `document_type_not_configured` rather than fabricating a default when none has been published.

### 3.2 File metadata, versioning, and lifecycle

`app.files` — one row per uploaded object; `record_type`/`record_id` are a generic polymorphic reference (no live FK yet, the same disclosed pattern `PLT-126`'s `custom_field_values` established); `classification`/`malware_scan_status`/`lifecycle_status` are all real bounded `CHECK` enums; a partial unique index (`version_group_id) where is_latest_version`) keeps exactly one current version per lineage. `app.initiate_file_upload()` — authority, idempotency short-circuit, filename path-traversal rejection, MIME/size/classification validation against the tenant's published definition, legal-hold eligibility/reason checks, then a real server-generated `storage_path`. `app.record_file_scan_result()` — the bounded scan adapter, idempotent-safe on re-resolution to the same status, `document_scan_already_resolved` on a genuine conflict. `app.create_file_version()` — always inserts a new row, flips the prior row to `superseded` first (so the partial unique index is never transiently violated), re-validates MIME/size/classification against the current definition. `app.request_file_deletion()` — soft delete only, idempotent, refuses outright under legal hold. `app.set_file_legal_hold()` — support/Supreme-only, reason required when setting a hold.

### 3.3 Access gate, audit, and RLS

`app.authorize_file_access()` — the single gate a real signed-URL-issuing server action calls: malware-scan gate (infected blocks everyone; not-yet-clean blocks everyone but the uploader) composed with the record/sensitivity access gate (`app.can_access_record()` plus a stricter check for `restricted`/`credential` classification). An actor with zero tenant standing is refused outright with no log row (the same "fails safely, never even logs" treatment `PLT-127` gave a wrong-tenant notification recipient); a tenant member merely denied by scope/sensitivity gets a real, logged, returned denial in `app.file_access_logs` (genuinely new, append-only, file-access-specific evidence, distinct from `app.audit_logs`). Every lifecycle mutation self-captures a canonical `app.capture_audit_event()` entry — no bespoke `*_history` table. RLS on `app.files` composes `app.can_access_record()`, a classification gate, and a deleted-row visibility restriction; `app.document_types` is broadly readable. **An additive grant was needed**: `app.is_support_grant_authority()` (`PLT-115`) was `service_role`-only since it had no RLS-policy consumer yet — this migration adds `grant execute ... to authenticated` so the RLS policy on `app.files` can evaluate it as the calling role (verified safe: the function's own internal `principal_memberships` lookup always targets the caller's own row, which passes that table's existing RLS policy regardless of branch). Per `ERR-2026-004`, the migration ends with its own `revoke execute on all functions in schema app from public;` before role-specific grants.
