## 2. Scope (§7/§11/§12)

`00_PLATFORM_CORE_WBS.md` row 15 / Prompt 118 §11: allowed scope is "Custom-domain schema/migrations/service/resolver/cache/tests/docs/runbook and approved provider adapter," 5-15 files, at most 1-3 migrations. Forbidden (§12): live DNS/cert mutation without external authority, a generic integration hub, auth bypass.

- **No live DNS/TLS operation is performed anywhere in this checkpoint** — a database migration cannot reach external DNS/cert infrastructure. `app.verify_tenant_domain()`'s `p_observed_txt_value` parameter is the provider-independent verification *interface* Prompt 118 §20 task 2 asks for: a future external job performs the real DNS TXT lookup and passes its observed value in; this function only compares it against the stored challenge token. The lookup job itself and any TLS/certificate provisioning are disclosed `NOT_RUN` — no such job/adapter exists yet (`JOB`/`API-WH` are `PLT-129..132`, downstream), the same posture `PLT-107` established for GoTrue.
- **Full Unicode IDN is deliberately out of scope.** Only ASCII hostnames (including already-punycode-encoded labels) validate; a raw Unicode domain is rejected outright — a disclosed, security-conscious boundary (Unicode homograph/lookalike-domain phishing is a real, well-known attack vector), proven directly in `db:test`.
- **"Custom domain changes presentation/routing, not access authorization"** (§24) is structural: `app.resolve_tenant_by_domain()` returns only a bare `tenant_id`/`canonical_status` pair, nothing that could substitute for `PLT-108`'s `app.resolve_access_context()` or any RBAC/RLS check.

## 3. Implementation (§20)

### 3.1 Normalized domain lifecycle, reserved rules, verification, tenant resolution (task 1)

`supabase/migrations/20260717103015_create_custom_domain.sql`. `app.normalize_domain_hostname()` (lowercase, trim whitespace/trailing dot), `app.validate_domain_hostname()` (RFC-1123-shaped, ASCII-only, alphabetic-only final label — structurally excludes IPv4 literals), `app.is_reserved_domain_hostname()` (the platform's own `cargogrid.app`/`*.cargogrid.app` namespace, `localhost`, IPv4 literals) are all real, independently testable functions, the first two wired as table `CHECK` constraints (a database guarantee, not a convention). `app.tenant_custom_domains` models `pending_verification -> verified -> active`, with `disabled`/`rejected`/`expired` terminal branches.

### 3.2 Provider-independent verification interface, data model/service (task 2)

`app.request_tenant_domain()` (idempotent, authority-gated via `PLT-115`'s `app.is_support_grant_authority()` — Tenant Admin of own tenant or Supreme Admin, per Prompt 118 §26), `app.verify_tenant_domain()` (compares `p_observed_txt_value` against the stored `verification_token`, never re-derives or trusts a caller's own success claim), `app.activate_tenant_domain()`, `app.disable_tenant_domain()` (the kill switch), `app.reject_tenant_domain()` (manual abuse-mitigation path), `app.expire_stale_domain_verifications()` (batch maintenance, disclosed `NOT_RUN` live-job wiring — no scheduler exists yet, `PLT-132` downstream). **Takeover/rebinding prevention is a real database guarantee, not application logic**: a partial unique index on `hostname` spanning the three "claim is live" states (`pending_verification`/`verified`/`active`) makes a second tenant's concurrent claim on the same hostname fail atomically at the database level regardless of application-level check-then-insert ordering — proven directly with a real concurrent-claim rejection test. Once a domain is `disabled`/`rejected`/`expired`, the identical hostname becomes claimable by a different tenant (real-world domain ownership changes hands) — proven as the explicit "rebinding" test.

### 3.3 Request host→tenant context, cache invalidation, security (task 3)

`app.resolve_tenant_by_domain()`: `SECURITY DEFINER`, deliberately granted to `anon` as well as `authenticated`/`service_role` (the same disclosed departure `PLT-117`'s `evaluate_tenant_brand()` established — inbound HTTP requests must resolve host→tenant before any session exists). Returns zero rows unless the domain is `active` **and** its owning tenant is `active` — never a stale/wrong tenant. `server/queries/custom-domain.ts`'s `DomainResolutionCache` mirrors `PLT-106`/`PLT-117`'s explicit-invalidation cache shape, caching a negative result too (short-lived, same TTL) so a freshly-activated domain becomes resolvable without waiting out a long negative-cache window.

### 3.4 Takeover/race/rebinding/cert-failure tests, observability, docs (task 4)

`scripts/db-tests/custom-domain.sql` (16 scenario groups, detailed in §4.1). "Cert failure" is disclosed scope: no certificate provisioning exists anywhere in this repository yet (§2) — the closest applicable analog this checkpoint actually tests is verification-challenge failure/expiry (wrong observed TXT value, expired challenge), which is real and tested. No dedicated runbook was authored this checkpoint (disclosed `NOT_RUN`, same posture `PLT-115` took for its own deferred support-access runbook) — no live DNS/cert operational surface exists yet for a runbook to describe.

### 3.5 A real, severe, repository-wide privilege defect found and fixed during this checkpoint's authoring

**Not part of this checkpoint's own planned scope**, but disclosed in full per this build's established discipline (`PLT-116`'s `users_directory` fix, `PLT-117`'s ambiguous-column fix). While writing `custom-domain.sql`'s own "defense in depth" schema-privilege test — the first test in this repository's history to call `has_function_privilege(...)` against a function meant to be `service_role`-only, rather than only `has_table_privilege` for tables — the assertion failed. Investigation confirmed: **PostgreSQL grants `EXECUTE` to `PUBLIC` on every function by default, and no migration from `PLT-105` through `PLT-117` ever revoked it**, meaning every function documented as "`service_role`-only" in ~12 prior migrations (90+ functions: `provision_tenant`, `invite_user`, `assign_role`, every support-access/audit-trail/white-label mutation, and more) was in fact directly callable by `anon` this entire time. Full detail, evidence, severity analysis, and recovery record: `docs/runtime/ERROR_LEDGER.md` `ERR-2026-004`.

**Fix**: `supabase/migrations/20260717095000_revoke_default_public_function_execute.sql` (applied immediately before this checkpoint's own `custom-domain` migration) revokes `EXECUTE` from `PUBLIC` on every existing `app`-schema function and sets `ALTER DEFAULT PRIVILEGES` so no future migration's functions receive it either. One real casualty found by re-running the full existing db-test suite afterward: `app.mask_email()` (`PLT-114`) had relied on the `PUBLIC` default for `authenticated` callers via `app.users_directory`'s internal call to it (a function-call `EXECUTE` check is evaluated against the querying role, not a view's owner) — re-granted explicitly. This checkpoint's own `custom-domain.sql` migration also carries a redundant, explicit `REVOKE EXECUTE ON ALL FUNCTIONS IN SCHEMA app FROM PUBLIC` statement, adopted as the new standing per-migration convention.

## 4. Tests (§27/§28/§29)

### 4.1 Real database tests (`scripts/db-tests/custom-domain.sql`, via `pnpm run db:test`)

16 scenario groups, using two tenants (a `tenant_admin` for each, a regular `org_user`, a global Supreme Admin):

Normalization/validation/reserved-hostname coverage (uppercase, trailing dot, surrounding whitespace, punycode IDN accepted, raw Unicode IDN rejected, bare single-label hostname rejected, IPv4-literal rejected, platform apex/subdomain/`localhost` reserved); table `CHECK` constraints reject an un-normalized or malformed hostname at insert time directly; `app.request_tenant_domain()` is authority-gated, rejects reserved/malformed hostnames, and is idempotent for a repeated pending request (same token, not regenerated); **takeover prevention** — a second tenant's claim on a hostname another tenant already holds live is rejected atomically (`unique_violation`); `app.verify_tenant_domain()` rejects a wrong observed TXT value and a non-tenant-admin actor, succeeds only with the exact issued token, and rejects re-verifying an already-verified domain; `app.activate_tenant_domain()` only accepts a `verified` domain, and `app.resolve_tenant_by_domain()` only starts resolving the hostname after activation, not before; cross-tenant isolation and case-insensitive Host-header lookup both proven directly against the resolver; a now-`suspended` tenant's previously-active domain stops resolving, then resumes once the tenant is active again; the kill switch (`app.disable_tenant_domain()`) stops resolution and **frees the hostname for a different tenant to claim** (the explicit rebinding proof); `app.reject_tenant_domain()` is Supreme-Admin-usable and only valid on a still-`pending_verification` row; `app.expire_stale_domain_verifications()` correctly expires only a past-due pending request, leaves an unexpired one untouched, and frees its hostname too; every one of the five lifecycle mutations self-captures a canonical `app.audit_logs` entry, plus a platform-wide (`tenant_id is null`) system entry for the batch expiry; `app.list_tenant_domains()` is authority-gated and tenant-scoped; schema-privilege defense in depth — `authenticated`/`anon` hold zero direct `SELECT` on `app.tenant_custom_domains` and zero `EXECUTE` on `app.list_tenant_domains()`, `anon` holds `EXECUTE` only on the public resolver.

### 4.2 TypeScript unit tests

`server/contracts/custom-domain/custom-domain.test.ts` (9 tests): `HostnameSchema` accepts a well-formed hostname and a punycode IDN label, rejects a raw Unicode IDN label, a bare single-label hostname, an uppercase hostname, and an IPv4 literal; row-mapping tests for both a fresh pending request and a fully-populated active domain.

`server/mutations/custom-domain.test.ts` (10 tests): exact snake_case RPC params for all five mutation functions; `insufficient_authority`/`domain_already_claimed`/`reserved_hostname`/`verification_token_mismatch`/`verification_expired`/`domain_not_verified` all classified distinctly.

`server/queries/custom-domain.test.ts` (8 tests): `listTenantDomains`/`resolveTenantByDomain` row parsing (including the array-response shape matching Supabase's table-returning RPC convention); a mixed-case hostname lowercased before the RPC call; `DomainResolutionCache` caches a positive and a negative resolution within TTL, re-fetches once expired, and `invalidate()` forces immediate freshness.
