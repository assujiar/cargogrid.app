## 2. Scope (§7/§11/§12)

Prompt 123 §11 allowed scope: "Approval engine schema/migrations/service/contracts/tests/docs and isolated example," 5-15 files, 1-3 migrations. §12 forbidden: "Domain-specific approval policies beyond example, full inbox UI, finance overrides."

- **An approval *definition* is not a new table family**, exactly like `PLT-122`'s workflow definitions — it is `PLT-121`'s own `config_type_code='approval'` config_object/config_version/config_items, reused directly. This migration adds only what the Configuration Engine does not already provide: approval-specific structural validation at publish time, and the runtime request/step/decision/delegation tables.
- **No domain approval policy beyond one safe, isolated, synthetic example exists anywhere in the migration** (§12). The representative example — a 2-step sequential manager-then-finance approval — is built, published, and exercised end-to-end entirely inside `scripts/db-tests/approval.sql`, alongside a parallel and a threshold variant proving all three routing patterns are genuinely distinct algorithms. No example definition row is seeded into the migration itself.
- **Three deterministic routing patterns** (§21/§22/§25): `sequential` (steps open one at a time; a rejection anywhere fails the whole request immediately), `parallel` (every step opens at once; the request approves only once every step independently reaches its own `required_approvals`), `threshold` (every step opens at once like `parallel`, but the request approves the instant `threshold_required_steps` of the total step count individually reach `approved`, and every still-open step is then marked `skipped`). All three proven directly in `db:test`.
- **Approver resolution is real**, reusing `PLT-112`'s own disclosed join shape (`app.role_assignments` joined to `app.role_versions` filtered to `status='published'`) — the exact convention that makes a stale assignment pointing at a since-superseded role version fail closed, not silently grant approval authority. `app.count_eligible_approvers_for_step()` is a real, tested guard against Prompt 123 §23's "no approver" exception case: `app.request_approval()` refuses to create a request against any step with zero currently-eligible approvers, proven directly.
- **Separation of duties**: `allow_self_approval` (definition-level, default `false`) structurally blocks the requester from deciding their own request's steps unless explicitly allowed — proven with a requester who is also a genuinely eligible approver (so the denial demonstrably comes from the self-approval check, not incidental ineligibility).
- **Delegation is real and bounded** (90 days, a constructed, disclosed default matching `PLT-115`'s 24h/2h support-grant expiry-cap posture, not a fabricated business policy). Delegation chains are deliberately not followed transitively — a bounded, disclosed scope choice that also structurally rules out a delegation-cycle infinite loop.
- **Escalation** (`app.escalate_approval_step()`) is the real, audited manual reassignment action §18/§22 call for. The automatic SLA-expiry trigger job that would invoke it on a timer is disclosed `NOT_RUN` — no scheduler/cron infrastructure exists anywhere in this repository yet.

## 3. Implementation (§20)

### 3.1 Structural validation and definition publish

`supabase/migrations/20260719090000_create_approval_engine.sql`. `app.validate_approval_definition(p_version_id)` reads `pattern`/`steps`/`threshold_required_steps`/`allow_self_approval` off a workflow-typed... approval-typed `config_version`'s own `config_items`, raising 9 distinct named exceptions (`approval_invalid_pattern`, `_missing_steps`, `_invalid_step_order`, `_invalid_approver_type`, `_missing_approver_ref`, `_unknown_role`, `_unknown_user`, `_invalid_required_approvals`, `_invalid_threshold`) — all proven individually in `db:test`. `app.publish_approval_definition()` runs this gate then composes `PLT-121`'s own `app.publish_config_version()`.

### 3.2 Runtime request/step/decision/delegation tables and lifecycle functions

`app.approval_requests` (`config_version_id` FK binds the request to the exact published snapshot at request time, mirroring `PLT-122`'s own instance/definition-snapshot discipline). `app.approval_request_steps` (one row per materialized step, live decision-progress state). `app.approval_decisions` (append-only; `unique(request_step_id, actor_auth_user_id)` is the real, structural "no duplicate decision" guard). `app.approval_delegations` (bounded, `CHECK`-enforced ≤90-day expiry, `not self`, scope-shape enforced). `app.request_approval()` (idempotent on `(tenant_id, idempotency_key)`; proactively rejects zero-eligible-approver steps). `app.decide_approval_step()` (the core engine — self-approval gate, live eligibility check including delegation, atomic `UPDATE ... WHERE status = 'active'` optimistic concurrency, pattern-specific completion logic for all 3 patterns). `app.cancel_approval_request()`. `app.escalate_approval_step()` (gated by `PLT-121`'s own `app.check_config_object_authority()` — definition-admin-grade authority, not the looser instance-level requester/approver authority). `app.create_approval_delegation()`/`app.revoke_approval_delegation()` (delegator-or-tenant_admin/Supreme authority).

### 3.3 Access/audit/tests/docs

RLS: `approval_requests`/`approval_request_steps`/`approval_delegations` get direct RLS-governed `SELECT` for `authenticated` (normal tenant business data, matching `PLT-113`/`PLT-122`'s posture); `approval_decisions` has no direct `authenticated` grant at all — `app.get_approval_request_history()` is its sole read path. `app.list_pending_approval_steps_for_actor()` is the reusable pending-approver-inbox view model Prompt 123 §15 asks for. Every lifecycle mutation self-captures a canonical `app.capture_audit_event()` entry — no bespoke `*_history` table for definitions (only the genuinely new request/step/decision/delegation tables exist). ## 8. Errors / recovery / risks / issues

Three real mistakes found and fixed during authoring, all **before** the first `db:test` run:

1. `app.escalate_approval_step()` initially called `app.check_config_object_authority()` with 5 arguments; the real function (`PLT-121`) takes exactly 3 (`p_scope_level`, `p_tenant_id`, `p_actor_auth_user_id`). Caught via direct signature lookup before running any test.
2. `app.decide_approval_step()`'s `allow_self_approval` extraction initially attempted `value::boolean` directly on a `jsonb` column — an invalid cast in PostgreSQL. Fixed to `(value #>> '{}')::boolean`, the same text-extraction-then-cast pattern already used for `initial_state`/`threshold_required_steps` elsewhere in this migration and `PLT-122`'s own workflow engine.
3. The delegation-eligibility check inside `app.is_eligible_approval_approver()` initially called itself recursively (to support transitive delegation chains) with no depth bound — a genuine infinite-loop risk if two identities ever delegated to each other. Fixed by deliberately scoping delegation to one hop only (the delegator must be *directly* eligible, not eligible via a further delegation), disclosed in the function's own comment as a bounded scope choice that also structurally rules out the cycle risk, rather than adding a depth cap the way `PLT-122`'s reachability walk does.

A test-authoring mistake was also caught and fixed while writing `scripts/db-tests/approval.sql` itself: an early draft used a role's own `id` as a `scope_level='branch'` config object's `scope_id`, which is structurally valid (scope_id is application-validated, not a physical per-scope-level FK) but semantically wrong test data; fixed to use `scope_level='role'` with the same id, which is what it actually represents.

No new issue opened. Real, disclosed `NOT_RUN`: live wiring of any of the 9 TS entry points into a REST/GraphQL/admin surface (no such surface exists yet); a real SLA-expiry auto-escalation scheduler (no cron/job infrastructure exists anywhere in this repository yet — `app.escalate_approval_step()` itself is real and tested, only the automatic timer trigger is deferred); `pnpm run test:e2e` (sandbox Playwright browser-binary revision skew, unrelated to this diff, same condition disclosed at `PLT-117..122`).
