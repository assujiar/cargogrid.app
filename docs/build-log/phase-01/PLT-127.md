## 2. Scope (§7/§11/§12)

Prompt 127 §11 allowed scope: "Notification/template/delivery schema/migrations/service/queue/in-app UI/tests/docs and one bounded adapter interface," 5-15 files, 1-3 migrations. §12 forbidden: "Bulk marketing campaigns, live provider sends without authority, generic integration hub."

- **No real provider send exists anywhere in this migration** (§12/§19). `in_app` delivery is real and complete — the notification row's own existence *is* the delivery. Any non-`in_app` channel (`email`) is real up through `queued`; no code anywhere calls a real email provider. `app.record_notification_delivery_attempt()` is the bounded adapter interface §11 asks for — a future capability that owns a real provider calls it; this migration never calls it against a fabricated provider itself.
- **No domain notification type exists anywhere in this migration.** The one representative example (a `role_assignment_granted`-shaped, en+id-localized, in_app+email template) is built and proven entirely inside `scripts/db-tests/notification.sql`.
- **A notification *template* is not a new table family**, exactly like `PLT-122..126`'s definitions — `app.register_notification_type()` mints a dedicated `notification:<code>` config_type per type via `PLT-121`'s own `app.register_config_type()`, the same registry composition `PLT-124`'s Status Engine established.
- **Template escaping is real, at both write time and render time** (§16). Template text passes through `PLT-121`'s own `<>`-rejecting `app.validate_config_value()` CHECK at definition time; `app.render_notification_template()` applies the identical `<>` rejection to every *context* value substituted in at render time too — the real injection vector is untrusted event/recipient data, not tenant-authored template text.
- **A real link allowlist.** Any context value shaped like `scheme://...` must use `https://`; a bounded known-dangerous-scheme blocklist (`javascript:`/`data:`/`vbscript:`) additionally catches schemes with no `://` (a real bug found and fixed during this checkpoint's own db-test run — see §8).
- **Recipient authorization is real and fails safely** (§16/§23/§25). `app.queue_notification()` requires the recipient to be an *active* member of the target tenant right now — a revoked/deactivated/wrong-tenant identity is refused outright, never queued.
- **Preference-aware fallback is real** (§22). A disabled requested channel falls back to `in_app` when the definition allows it and the recipient hasn't also disabled that; otherwise the attempt is recorded as `skipped`, never silently dropped.
- **The in-app inbox RLS posture is deliberately tighter** than every prior engine's shared-business-data table — `app.notifications`/`app.notification_preferences` are a personal inbox/personal settings, scoped to `recipient_auth_user_id = auth.uid()` / `auth_user_id = auth.uid()` (plus Supreme), not "any active tenant member."

## 8. Errors / recovery / risks / issues

Two real defects found and fixed via actual failing `pnpm run db:test` runs during this checkpoint (neither caught by self-review beforehand):

1. `app.render_notification_template()`'s link-safety check originally only matched `scheme://` URLs and required a non-`https://` scheme to reject. A test asserting `javascript:alert(1)` (a real XSS payload shape) should be rejected failed — `javascript:` URIs have no `//` after the colon, so the check never matched it at all and let it through unrendered-unsafely. Fixed by adding a second, explicit check for a bounded known-dangerous-scheme blocklist (`javascript:`/`data:`/`vbscript:`), deliberately narrow enough not to false-positive on ordinary colon-containing text like "Note: important".
2. `app.record_notification_delivery_attempt()` passed its own attempt-status literal (`'success'`/`'failed'`) directly as `app.capture_audit_event()`'s `p_result` argument, but `app.audit_logs.result` only accepts `'success'`/`'failure'` (PLT-116's own fixed vocabulary) — a real `CHECK` constraint violation surfaced by the test that exercised a failed delivery attempt. Fixed by mapping `'failed'` → `'failure'` explicitly at the call site.

No new issue opened. Real, disclosed `NOT_RUN`: live wiring of any of the 9 TS entry points into a REST/GraphQL/notification-bell UI surface (no such surface exists yet); a real external email/SMS provider integration (no such integration exists anywhere in this repository — `app.record_notification_delivery_attempt()` is the real, tested adapter interface a future capability would call); automatic backoff/retry scheduling (no cron/job infrastructure exists anywhere in this repository yet, the same class of deferral `PLT-123`'s SLA-escalation trigger already disclosed); `pnpm run test:e2e` (sandbox Playwright browser-binary revision skew, unrelated to this diff, same condition disclosed at `PLT-117..126`).
