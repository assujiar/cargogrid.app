### 3.3 Banner/reason/start/end/expiry/revoke and emergency kill switch primitives (task 3)

`app.start_support_session()` requires a fresh (≤5 minute) `reauth_confirmed_at` (Prompt 115 §4 "re-authentication") and is idempotent against a retry while a session is already open. `app.end_support_session()` is idempotent-safe on an already-ended session. `app.revoke_support_access()` is the kill switch (§16): either Supreme Admin or the target tenant's own `tenant_admin` can revoke instantly, cascading to force-end any open session with `ended_reason='revoked'`. Expiry itself is never a manually-triggered transition — matching `06_*.md` §2.3 exactly, an approved-but-past-`expires_at` grant is denied live by `has_active_support_grant()`'s own `expires_at > now()` condition, with no cron/job needed (the same "computed, not stored" posture this session has used since `PLT-105`'s tenant lifecycle). `deriveSupportSessionBanner()` (`server/contracts/support-access/support-access.ts`) is the reusable banner/context/reason/end-action/denied/expired-state primitive Prompt 115 §15 asks for — a pure derivation, since no live portal-shell component exists yet in this repository (Phase 1 has not reached `PLT-135`/`136`, the Admin portals).

### 4.1 Real database tests (`scripts/db-tests/support-access.sql`, via `pnpm run db:test`)

`reason`/`case_id` mandatory (`not_null_violation`); idempotent request-retry for a repeated (tenant, grantee, case) while the prior request is still live; a request exceeding the standard 24h expiry cap is rejected; **RLS visibility is zero while a grant is `pending_approval`** — a request alone grants nothing; approval requires real authority — a regular `org_user` is denied, the grantee cannot self-approve, a `tenant_admin` can; **once approved, `support_access_gated` admits the support agent into the tenant's RLS-governed data** (`app.tenants` visible) while `app.users_directory`'s email stays masked (no `View personal data` permission) and `app.can_access_record()` still denies on ownership — the concrete "layers over, does not bypass" proof; a grant into tenant A leaks zero visibility into tenant B; a stale (10-minute) re-authentication is rejected, a fresh one starts a session, re-starting an already-open session is idempotent; `app.current_support_session()` surfaces the open session; **the kill switch** — a `tenant_admin` revokes an approved grant mid-session, force-ending the open session (`ended_reason='revoked'`) and immediately closing RLS visibility, with revoke itself idempotent-safe on retry; **expiry** — an approved grant past `expires_at` denies access even though never explicitly revoked, proven via a real 3-second wall-clock sleep past a 2-second-out expiry; **emergency access** — rejected with no recorded authority, rejected past the 2h cap, auto-approved with a recorded authority, and gated by Supreme-Admin-only post-review (a non-Supreme-Admin reviewer is rejected, and post-review of a non-emergency grant is rejected); denial — a `tenant_admin` can deny a pending request, and approving an already-denied (terminal) grant is rejected; defense in depth — `anon` denied entirely on every new table, `service_role` sees every grant and the full append-only event trail.

## 8. Errors / recovery / risks / issues

Three real test-authoring mistakes, found and fixed by actually running `pnpm run db:test` and reading the failures, not assumed correct from the SQL alone:

1. An initial "support agent has zero visibility before approval" assertion ran *after* the same grant had already been approved earlier in the same test run (the assertion's own premise no longer held) — fixed by moving that check to right after the request, while the grant was still genuinely `pending_approval`.
2. The expiry test first tried `update ... set expires_at = now() - interval '1 minute'`, which failed the `support_access_grants_expiry_shape` `CHECK` constraint (`expires_at > requested_at`) — because `now()` is frozen at transaction-start for the lifetime of a single `DO` block, `now() - 1 minute` was *always* earlier than `requested_at` (itself `≈ now()` at insert time within that same frozen transaction), never a usable "past" value. Fixed by setting `expires_at` to `requested_at + interval '2 seconds'` (satisfying the constraint) and using a genuine wall-clock `select pg_sleep(3);` as its own separate top-level statement/transaction before checking the now-expired grant — the correct way to simulate real elapsed time inside a single-transaction-per-statement `psql -f` test runner.
3. A "denied grant must never satisfy `has_active_support_grant`" assertion was checked at a point in the test run where a *different*, still-valid emergency grant already existed for the same agent/tenant (from an earlier scenario) — the aggregate `has_active_support_grant()` predicate is deliberately per-(tenant, identity), not per-grant, so it correctly returned `true` for an unrelated reason. Fixed by asserting the denied grant's own `granted_at is null` instead of the aggregate predicate.

No new issue opened. Real, disclosed `NOT_RUN`: live wiring of any of the 9 TS entry points into a REST/GraphQL/server-action/job (§3.2/§6 — no such surface exists yet); a dedicated incident runbook (§3.4 — deferred as speculative documentation ahead of a live on-call consumer, same discipline as `PLT-114`'s deferred caching layer). Real, disclosed open decision for a future capability: whether a support grant should also satisfy `app.can_access_record()`'s ownership/share/customer-scope OR-branches directly for a future business-domain table with `owner_user_id` — not decided here, since no such table exists yet in Phase 1 (§3.2).
