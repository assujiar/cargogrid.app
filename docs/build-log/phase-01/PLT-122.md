## 2. Scope (§7/§11/§12)

Prompt 122 §11 allowed scope: "Workflow engine schema/migrations/service/contracts/tests/docs and isolated example," 5-15 files, 1-3 migrations. §12 forbidden: "Domain workflows beyond example, arbitrary script execution, broad module adoption."

- **A workflow *definition* is not a new table family.** It is `PLT-121`'s own `config_type_code='workflow'` `config_object`/`config_version`/`config_items`, reused directly — the same "shared mechanism, not a fork" discipline `PLT-120`/`121` already established. This migration adds only what the Configuration Engine does not already provide: workflow-specific structural/reachability/dead-end validation at publish time, a hook allowlist, and the runtime instance/transition-history tables.
- **No domain workflow beyond one safe, isolated, synthetic example exists anywhere in the migration.** The representative example — a `draft -> submitted -> approved|rejected` request workflow — is built, published, and exercised end-to-end entirely inside `scripts/db-tests/workflow.sql` against synthetic tenant data. No example workflow row is seeded into the migration itself.
- **Guards/effects are a real, bounded allowlist (`app.workflow_hooks`), not arbitrary code** (§24: "Hooks are allowlisted domain contracts, not arbitrary code"). Exactly two safe, real, template hooks are seeded: `always_true` (guard) and `noop` (effect). `app.evaluate_workflow_guard()` has real logic only for `always_true`; any other *registered* guard code fails closed with `guard_not_implemented` rather than silently passing — proven directly in `db:test` via a registered-but-unimplemented `never_true` guard that blocks a real transition attempt.
- **`entity_type`/`entity_id` on `app.workflow_instances`** are a polymorphic, application-validated reference — the same disclosed, sourced pattern `05_DATABASE_SCHEMA_WORKSTREAM.md` §4 established for `ticket_links` and `PLT-121`'s own `config_objects.scope_id` already reused.
- **Reachability/dead-end/cycle rules (§25).** Reachability and dead-end validation are real, enforced publish-time gates. "Cycle" is handled by making the reachability walk itself a bounded (≤50 hop) recursive walk — workflows may legitimately contain cycles (e.g. a revision/resubmission loop), so the validator itself must never loop forever, not that a cyclic workflow is rejected.
- **Definition admin and instance transition/view are separately permissioned** (§26): `app.check_config_object_authority()` (Supreme/`is_support_grant_authority`, `PLT-121`) governs the underlying `config_object`; `app.check_workflow_instance_authority()` (`has_active_tenant_membership` or Supreme) governs running an instance — a normal business action, not an administrative one.

## 3. Implementation (§20)

### 3.1 Hook allowlist and structural validation

`supabase/migrations/20260717140000_create_workflow_engine.sql`. `app.workflow_hooks` (code/hook_type/name/description registry, seeded with `always_true`/`noop`), `app.register_workflow_hook()` (idempotent, Supreme-only). `app.validate_workflow_definition(p_version_id)` reads the `states`/`initial_state`/`terminal_states`/`transitions` `config_items` off a workflow-typed `config_version` and raises a distinct, named exception per failure mode: `workflow_missing_states`, `workflow_invalid_initial_state`, `workflow_invalid_terminal_state`, `workflow_invalid_transition_from`/`_to`, `workflow_unknown_guard`/`_effect`, `workflow_unreachable_state` (bounded BFS from `initial_state`), `workflow_dead_end_state` (every non-terminal state must have ≥1 outgoing transition). `app.publish_workflow_definition()` runs this gate, then composes `PLT-121`'s own `app.publish_config_version()`.

### 3.2 Runtime instance/transition-history tables and lifecycle functions

`app.workflow_instances` (tenant-scoped, `config_version_id` FK binds the instance to the exact published snapshot at start time — a later republish creates a *new* `config_versions` row per `PLT-121`'s never-mutate-a-published-version discipline, so the binding is permanent; `unique(tenant_id, idempotency_key)`). `app.workflow_transition_history` (append-only: `start`/`transition`/`cancel`/`fail`). `app.start_workflow_instance()` (idempotent on `(tenant_id, idempotency_key)`, requires the target version `status='published'`, reads `initial_state`). `app.transition_workflow_instance()` (optimistic concurrency via `p_expected_current_state` → `stale_workflow_state` on mismatch; looks up the declared transition; evaluates its guard via `app.evaluate_workflow_guard()`; marks the instance `completed` the moment `to_state` is in `terminal_states`). `app.cancel_workflow_instance()` (`active` → `cancelled`). `app.get_workflow_instance_history()` (authority-gated read view-model, `SECURITY DEFINER` since `workflow_transition_history` carries no direct `authenticated` grant at all).

### 3.3 Access/audit/tests/docs

RLS: `workflow_hooks` is broadly `SELECT`-able platform reference data; `workflow_instances` gets a direct RLS `SELECT` policy for `authenticated` (`has_active_tenant_membership(tenant_id) or is_supreme_admin()`) — a deliberate departure from `PLT-121`'s "no direct grant" posture for draft config content, since a running instance is normal tenant business data, matching `PLT-113`'s posture for primary tables. `workflow_transition_history` has no RLS-governed `authenticated` grant at all; `app.get_workflow_instance_history()` is its sole read path. Every lifecycle mutation self-captures a canonical `app.capture_audit_event()` entry — no bespoke `*_history` table exists for hooks/definitions (only the genuinely new `workflow_transition_history` exists, for instance runtime events specifically). Per `ERR-2026-004`, the migration ends with its own `revoke execute on all functions in schema app from public;` before role-specific grants.

## 8. Errors / recovery / risks / issues

Two real defects found and fixed during authoring, both **before** the first `db:test` run:

1. A stray, prematurely-placed `comment on function app.evaluate_workflow_guard is 'placeholder';` statement appeared immediately before the real `create function app.evaluate_workflow_guard(...)` definition — `COMMENT ON FUNCTION` requires the function to already exist, so this would have failed migration application outright. Caught via self-review; removed.
2. `app.get_workflow_instance_history()` was initially written as a plain (non-`SECURITY DEFINER`) function, but `app.workflow_transition_history` carries no direct `authenticated` grant at all (it is the sole read path for that table) — calling it as `authenticated` would have hit "permission denied," the same class of defect `ERR-2026-004` found in `app.mask_email()` and `PLT-121` found in `app.resolve_config()`. Fixed by adding `security definer set search_path = app, pg_temp` before the first test run.

Two authoring mistakes were also caught and fixed while writing `scripts/db-tests/workflow.sql` itself, before it produced a clean run: a test ordering bug where the "regular org_user denied publish" assertion ran against stale (previously-invalid) `config_items` left over from an earlier failure-mode test rather than the intended valid example, and a plain-string-vs-`jsonb` cast/parenthesization mistake in the same file's dynamic JSON construction. Both fixed via direct edit before the suite was declared green.

No new issue opened. Real, disclosed `NOT_RUN`: live wiring of any of the 6 TS entry points into a REST/GraphQL/admin surface (no such surface exists yet); real (non-`always_true`) guard/effect implementations for any actual business rule (no business rule exists yet in this still-Platform-Core-only repository — the next capability to register one is also responsible for extending `app.evaluate_workflow_guard()`'s dispatch, per the migration's own header); `pnpm run test:e2e` (sandbox Playwright browser-binary revision skew, unrelated to this diff, same condition disclosed at `PLT-117..121`).
