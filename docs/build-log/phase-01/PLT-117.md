## 3. Implementation (§20)

### 3.1 Canonical brand tokens/asset types, validation, accessibility constraints (task 1)

`supabase/migrations/20260717090512_create_white_label.sql`. `app.hex_color_relative_luminance()`/`app.hex_color_contrast_ratio()` implement the real WCAG 2.2 relative-luminance/contrast formula in SQL (verified against the known black-on-white=21:1 reference pair in `scripts/db-tests/white-label.sql`). `app.validate_brand_tokens()` (only `primary`/`secondary`, each a `#RRGGBB` hex string — RPD-019's fixed color surface), `app.validate_brand_asset_url()` (HTTPS-only, safe character set — structurally rejects `javascript:`/`data:`/quote/angle-bracket injection vectors), and `app.validate_document_template_refs()` (a fixed, whitelisted key set; free-text values reject any `<`/`>`) are all wired as table `CHECK` constraints on `app.tenant_brand_versions` — a database guarantee, not an application-layer convention, matching `PLT-114`'s "column `REVOKE`, not a documented rule" discipline.

### 3.2 Versioned draft/publish/rollback model and evaluator/cache (task 2)

`app.tenant_brand_versions` mirrors `PLT-111`'s `app.role_versions` shape exactly: `draft -> published -> archived`, at most one draft and one published version per tenant at a time (partial unique indexes), publishing a new draft archives the tenant's previously published version (supersession). `app.create_tenant_brand_draft()` (idempotent), `app.set_tenant_brand_tokens()` (draft-only mutation; computes contrast as **preview evidence**, not yet enforced), `app.publish_tenant_brand_version()` (**enforces** a 4.5:1 WCAG AA contrast minimum against the fixed `#fafafa` neutral-50 background token when a primary color is set; a version with no primary color at all always publishes — exactly the "missing override uses the accessible default" alternative flow, Prompt 117 §22), `app.discard_tenant_brand_draft()` (draft -> archived without publishing), and `app.rollback_tenant_brand_version()` (clones a published/archived target's exact snapshot into a brand-new version and publishes it immediately — never mutates the target row, `PLT-111`'s `clone_role_version()` discipline; re-validates contrast at the target's own tokens rather than trusting a stored flag, since an archived row could have reached that state via discard, never having passed the publish gate). `server/queries/white-label.ts`'s `WhiteLabelBrandCache` mirrors `PLT-106`'s `EntitlementCache` explicit-invalidation shape, disclosed as a freshness optimization here rather than load-bearing correctness (a fresh publish/rollback always produces a new `version_id`, so a stale cache entry can never point at content that no longer matches its own key).

### 3.3 Shared shell/component token integration with safe fallbacks (task 3)

`app.evaluate_tenant_brand(p_tenant_id)`: `SECURITY DEFINER`, deliberately granted to `anon` as well as `authenticated`/`service_role` — a disclosed departure from this repository's default-deny grant posture, justified by RPD-019 branding being non-confidential presentation data that a tenant's login page must render *before* authentication. Returns the tenant's current published version (`source='tenant'`) when the tenant is `active` and has one, otherwise the accessible CargoGrid default (`source='default'`) — composed only from `DESIGN_SYSTEM.md` §2.1's one already-sourced, already-contrast-vetted `neutral-900` (`#171717`) reference value for both `primary`/`secondary`, undifferentiated, never a fabricated brand color (§2 above). No UI/shell consumes this yet (§2's disclosed scope boundary) — `server/queries/white-label.ts`'s `evaluateTenantBrand()` is the typed entry point a future portal-shell/login-page implementation calls.

### 3.4 Asset/security/contrast/cache/cross-tenant tests, docs, admin dependencies (task 4)

`scripts/db-tests/white-label.sql` (16 scenario groups, detailed in §4.1). Two real defects were found and fixed by actually running the tests, not merely written and assumed correct:

1. **A genuine ambiguous-column bug in `app.evaluate_tenant_brand()` itself.** The function's own `RETURNS TABLE (tenant_id uuid, ...)` clause declares an output column literally named `tenant_id`, which Postgres treats as an implicit variable in the function body — colliding with `app.tenant_brand_versions.tenant_id` in the internal `WHERE tenant_id = p_tenant_id` predicate (`ERROR: column reference "tenant_id" is ambiguous`). Fixed by qualifying every table reference inside the function body with an explicit alias (`t.id`, `v.tenant_id`). Caught by `db:test`, not by inspection — the query is syntactically valid SQL in isolation and would not have been flagged by `typecheck`/`lint`.
2. **A `plpgsql` composite-assignment syntax defect across seven of this test file's own assertions**, all using the invalid `v_var := (select * from table where ...)` form instead of the required `select * into v_var from table where ...;` (`ERROR: subquery must return only one column`) — fixed uniformly across the file. A pure test-authoring mistake, not a defect in the migration.

## 4. Tests (§27/§28/§29)

### 4.1 Real database tests (`scripts/db-tests/white-label.sql`, via `pnpm run db:test`)

16 scenario groups, using two tenants (a `tenant_admin` for each, a regular `org_user`, a global Supreme Admin):

`app.hex_color_contrast_ratio()` WCAG reference-pair sanity (black-on-white=21.0, identical colors=1.0, symmetric regardless of argument order, malformed hex raises); every `CHECK` constraint rejects its corresponding malformed input (non-hex token, out-of-scope token/template key, `javascript:`/non-`https` asset URL, angle-bracket template/sender-name text) at insert time; `app.create_tenant_brand_draft()` is idempotent and authority-gated (a regular `org_user` and a *different* tenant's `tenant_admin` are both denied; the tenant's own `tenant_admin` and Supreme Admin both succeed); `app.set_tenant_brand_tokens()` only mutates a draft, computes `contrast_validated` as informational preview evidence (a low-contrast draft still saves), and is authority-gated; `app.publish_tenant_brand_version()` rejects a nonexistent `version_id`, **rejects a low-contrast primary color outright** (the enforced accessibility gate — the version provably remains `draft` afterward, not silently degraded), and **always publishes a version with no primary color at all** (the default-theme case); publishing a second draft supersedes (archives) the tenant's prior published version, with exactly one published version at a time proven by count; `app.rollback_tenant_brand_version()` never mutates the target row, creates a brand-new version carrying the target's exact snapshot, publishes it immediately, and rejects rolling back a still-`draft` version; every one of the five lifecycle mutations produces its own `app.audit_logs` entry (Prompt 117 §18 — no bespoke `*_history` table exists for this capability, §3.2); `app.evaluate_tenant_brand()` returns the tenant's own published version when active-and-published, and falls back to the accessible default for a tenant that never published, does not exist, or is `suspended`; schema-privilege defense in depth — `authenticated`/`anon` hold zero direct `SELECT` on `app.tenant_brand_versions`, `anon` holds `EXECUTE` on the evaluator; cross-tenant isolation — evaluating one tenant never returns a different tenant's published primary color, proven with real published data on both sides; `app.discard_tenant_brand_draft()` archives a draft directly and rejects discarding an already-archived row; at most one draft per tenant at a time (partial unique index, proven by a raw `INSERT` violation).
