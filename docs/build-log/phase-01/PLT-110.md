## 2. Scope (§7/§11/§12)

`00_PLATFORM_CORE_WBS.md` row 7: allowed scope is "User/membership/invitation schema/migrations/service/contracts/tests/docs and bounded notification hook"; DB impact is "User profile, tenant/customer memberships, invitations, lifecycle, org assignments and uniqueness constraints." Forbidden: full role builder/portal (`PLT-111`, next), HR employee lifecycle, real user data, broad email provider integration (§12) — all deliberately absent here.

**Layering, disclosed explicitly (task 1's "define user versus auth identity versus tenant/customer membership lifecycle"):** `app.users` (this checkpoint) is a *profile* layered on top of two already-`VERIFIED` primitives, not a replacement for either — `app.tenant_user_identities` (`PLT-107`) proves an auth identity is linked to a tenant at all; `app.principal_memberships` (`PLT-108`) carries the four-layer context. `app.users` adds the business-facing richness (display name, org assignment, a richer invite/active/suspend/revoke lifecycle) neither of those tables carries, and every lifecycle transition here keeps the other two in sync rather than letting three tables drift.

**Disclosed constraint carried over from `PLT-107`:** invite-by-email-before-signup (no `auth_user_id` yet) is `NOT_RUN` — `app.tenant_user_identities` already requires an existing `auth.users` row via FK, so `app.users` (one layer above it) inherits the same constraint. This checkpoint models inviting an *already-known* auth identity into a new tenant, not a cold invite to someone with no account anywhere; a live GoTrue admin-invite flow would be needed to close this gap, and none exists (`ADR-0010`).

## 3. Implementation (§20)

### 3.1 User vs. identity vs. membership taxonomy (task 1)

See §2. `app.users` composite-FKs `(auth_user_id, tenant_id)` to `app.tenant_user_identities` — the same "no orphan membership" discipline `PLT-108` established for `app.principal_memberships`.

### 3.2 Invitation/membership/profile/org-assignment service (task 2)

`supabase/migrations/20260716102620_create_users.sql`: `app.invite_user()` composes `app.link_auth_identity()` (`PLT-107`) to guarantee the identity linkage exists, then idempotently creates (or returns) the profile — proven distinctly from a genuine duplicate-email conflict in §4.1. `app.resend_invitation()` extends a still-pending invite's expiry, rejected once the user is no longer `invited`. `app.reassign_user_org_unit()` validates the target `app.org_units` row belongs to the same tenant (`PLT-109`) and carries real optimistic concurrency (`expected_version`), matching `PLT-109`'s own discipline.

### 3.3 Auth activation/session revocation and downstream cleanup (task 3)

`app.transition_user_status()` is the single canonical transition function (`invited→active|revoked`, `active→suspended|revoked`, `suspended→active|revoked`, `revoked` terminal — the same matrix-and-trigger discipline every prior lifecycle table this session established) with two real integration behaviors inside the same transaction, both proven in §4.1, not merely documented:

1. **Last-critical-admin guard:** leaving `active` toward `suspended`/`revoked` is blocked if the identity holds the tenant's *only* active `tenant_admin` `principal_membership` (`PLT-108`) — Prompt 110 §23's explicit exception-flow requirement.
2. **Downstream cleanup on revoke:** the underlying `PLT-107` identity linkage is revoked (via `app.revoke_auth_identity()`), and every active `PLT-108` principal membership for that identity in that tenant is revoked too (via `app.revoke_principal_membership()`, one call per active membership) — real reuse of already-`VERIFIED` primitives, not new role/permission logic (§12 forbids a full role builder here).

**Real, disclosed `NOT_RUN`:** live session/token revocation against a GoTrue endpoint — no live Supabase project exists (`ADR-0010`, unchanged since `PLT-107`). This checkpoint's revocation is complete at the database layer (identity link + every membership), which is what a live session-revocation call would itself act on once wired.

### 3.4 Bulk-safe boundaries, tests, audit, docs, recovery (task 4)

No bulk/batch function exists (§12 "unapproved bulk restructuring" forbidden) — every mutation operates on exactly one user, one transition, one org reassignment at a time. `app.user_lifecycle_history` records `invite`/`resend_invite`/`activate`/`suspend`/`reactivate`/`revoke`/`cancel_invite`/`org_reassign` events — `cancel_invite` is computed distinctly from a post-activation `revoke` (both are `invited`/`active`→`revoked` transitions through the same function, but the history event type differs, proven in §4.1) — the same bounded, capability-scoped audit pattern `PLT-105`–`109` each established.

## 4. Tests (§27/§28/§29)

### 4.1 Real database tests (`scripts/db-tests/user-lifecycle.sql`, via `pnpm run db:test`)

13 scenario groups: idempotent invitation composing the underlying identity link (exactly 1 linkage row, not 2); a duplicate email within one tenant rejected even for a different identity; resend extends expiry and is rejected once no longer `invited`; activation syncs both `app.users` and the underlying `app.tenant_user_identities` row; a second `tenant_admin` is granted so the guard has something to compare against; suspend/reactivate for a non-last admin succeeds; **the last-critical-admin guard blocks suspending/revoking the tenant's only active `tenant_admin`**; **revoke propagates** — the identity link is revoked and every active principal membership is revoked too; terminal-state enforcement; a cancelled (never-activated) invitation records a distinct `cancel_invite` history event vs. a post-activation `revoke`; org reassignment validates tenant ownership and optimistic concurrency; a cross-tenant org-unit reassignment rejected; resending/transitioning/reassigning a non-existent user fails cleanly; the same anon/authenticated schema-privilege defense-in-depth `PLT-105`–`109` established.
