## 2. Scope (§7/§11/§12)

Prompt 125 §11 allowed scope: "Numbering schema/migrations/service/contracts/tests/docs and one representative hook," 5-15 files, 1-3 migrations. §12 forbidden: "Renumber historical records, client allocation, module-specific hard-coded formats."

- **No module-specific hard-coded format exists anywhere in this migration.** The one representative example (an `INV-{SCOPE_CODE}-{YYYY}-{SEQ}` invoice-shaped format) is built and proven entirely inside `scripts/db-tests/numbering.sql` — no format/definition row of any kind is seeded into the migration itself.
- **A numbering *definition* is not a new table family**, exactly like `PLT-122`/`123`/`124`'s definitions — it is `PLT-121`'s own `config_type_code='numbering'` config_object/config_version/config_items, reused directly.
- **The internal canonical ID remains stable; the display number is a separate, generated artifact** (§24). `app.numbering_allocations.id` is the real primary key any FK would reference; `formatted_number` is a rendered string with its own `unique(tenant_id, formatted_number)` constraint — the two are deliberately never conflated, the same `05_DATABASE_SCHEMA_WORKSTREAM.md` §2 dual-column discipline `PLT-124`'s own header cites.
- **Allocated numbers are never silently recycled** (§24). Voiding/releasing an allocation only changes its `status` — it never returns the consumed sequence value to the counter. Voiding an already-allocated number requires an explicit, non-empty reason (`numbering_void_reason_required`), the "explicit audited rule" §24 requires.
- **Format tokens are a real, bounded allowlist** (`{YYYY}`, `{YY}`, `{MM}`, `{DD}`, `{SEQ}`, `{SCOPE_CODE}`), not an arbitrary template/expression language — the same "no arbitrary executable code" guardrail `07_CONFIGURATION_ENGINE_WORKSTREAM.md` §11 states for every `config_type`. Exactly one `{SEQ}` token is required (never zero, never more than one).
- **Atomic, collision-safe allocation is a real database guarantee.** `app.allocate_numbering_seq()` is a single `INSERT ... ON CONFLICT ... DO UPDATE ... RETURNING` statement — Postgres's standard safe atomic-counter pattern. A genuine concurrent-load benchmark across multiple real database connections is disclosed `NOT_RUN` in this single-threaded SQL test sandbox; correctness under sequential concurrent-shaped calls (same key, repeated calls, idempotent replay) is proven directly instead.
- **Legacy bootstrap never renumbers historical records** (§19/§25). `app.bootstrap_numbering_counter()` seeds a counter's *last-used* value directly and structurally refuses to ever lower an existing counter (`numbering_counter_cannot_decrease`).

## 3. Implementation (§20)

### 3.1 Format-token validation and definition publish

`supabase/migrations/20260719110000_create_numbering_engine.sql`. `app.validate_numbering_definition(p_version_id)` reads `format`/`reset_period`/`padding` off a `numbering`-typed `config_version`, raising 4 distinct named exceptions (`numbering_missing_format`, `numbering_invalid_seq_token_count`, `numbering_unknown_token`, `numbering_invalid_reset_period`, `numbering_invalid_padding`). `app.publish_numbering_definition()` runs this gate then composes `PLT-121`'s own `app.publish_config_version()`. `app.format_numbering_value()` is the pure, side-effect-free token-substitution renderer, granted directly to `authenticated` for live format-preview use (§15).

### 3.2 Atomic counters and allocation lifecycle

`app.numbering_counters` (one row per `(config_version_id, scope_key, period_key)`, `next_seq` holds the last-allocated value). `app.allocate_numbering_seq()` is the single-statement atomic upsert-or-increment. `app.numbering_allocations` (`unique(tenant_id, formatted_number)` is the final collision guarantee; `unique(tenant_id, idempotency_key)` is the idempotency guarantee). `app.allocate_or_reserve_number()` is the shared core (authority + published-version check, idempotent short-circuit, seq allocation, format render), used by both `app.allocate_number()` (main flow, `status='allocated'` immediately) and `app.reserve_number()` (alternative flow, `status='reserved'`). `app.confirm_number_reservation()`/`app.release_number_reservation()`/`app.void_number_allocation()` round out the lifecycle — all terminal state changes, none ever frees a seq for reuse. `app.bootstrap_numbering_counter()` handles legacy import.

### 3.3 Access/audit/tests/docs

Allocation-time authority is deliberately looser than definition-management authority (§26: "services allocate for permitted records") — the same `has_active_tenant_membership()`-or-Supreme posture `PLT-122`/`123`'s own instance-level authority checks use. RLS: `app.numbering_allocations` gets a direct RLS-governed `SELECT` policy for `authenticated` (normal tenant business data); `app.numbering_counters` carries no direct grant at all (its own comment: "every real read path is through app.numbering_allocations or the dedicated RPCs"). Every lifecycle mutation self-captures a canonical `app.capture_audit_event()` entry. ## 8. Errors / recovery / risks / issues

One real defect found and fixed **via an actual failing `db:test` run**, not caught during self-review beforehand: the first draft of the representative example used the format `INV-{YYYY}-{SEQ}` (no `{SCOPE_CODE}` token) for the definition, then a later assertion allocated a number under a *different* `scope_key` (`branch-jkt` vs. `default`) expecting it to independently reach `seq=1` and succeed. Both counters correctly and independently reached `seq=1` — but because the format never referenced `{SCOPE_CODE}`, both renders produced the *identical* literal string (`INV-2026-000001`), and `app.numbering_allocations`' own real `unique(tenant_id, formatted_number)` constraint correctly rejected the second insert as a genuine collision. This was not a database bug — it was the collision-safety guarantee working exactly as designed against a test that hadn't actually differentiated its two scope keys in the rendered output. Fixed by changing the representative example's format to `INV-{SCOPE_CODE}-{YYYY}-{SEQ}` (a more realistic multi-branch invoice format in the first place) and updating every downstream assertion to match; the fix was verified by re-running the isolated test file directly against a disposable debug database before re-running the full suite. This incident is itself disclosed as a useful, real proof point: the "collision analysis" this migration's own header describes as happening at allocation time (not merely format validation time) is genuinely enforced, caught a real would-be collision in this checkpoint's own test authoring, and did not silently duplicate a number.

No new issue opened. Real, disclosed `NOT_RUN`: live wiring of any of the 9 TS entry points into a REST/GraphQL/admin-form surface (no such surface exists yet); a genuine multi-connection concurrent-load benchmark (single-threaded SQL test sandbox, correctness proven via sequential repeated/idempotent calls instead); `pnpm run test:e2e` (sandbox Playwright browser-binary revision skew, unrelated to this diff, same condition disclosed at `PLT-117..124`).
