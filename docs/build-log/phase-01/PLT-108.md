## 3. Implementation (§20)

### 3.1 Principal/layer taxonomy and memberships (task 1)

`supabase/migrations/20260716100825_create_principal_memberships.sql` adds `app.principal_memberships`: `layer` (`supreme_admin`/`tenant_admin`/`org_user`/`customer_user`, closed set via `check`), `tenant_id` (nullable — null only for `supreme_admin`), `customer_account_ref` (non-null only for `customer_user`). A single `principal_memberships_layer_scope_shape` check constraint enforces exactly which scope dimensions each layer may carry — proven to reject malformed rows at insert time in §4.1, not left to application code (Prompt 108 §20 task 1's "allowed context dimensions"). A composite foreign key `(auth_user_id, tenant_id)` → `app.tenant_user_identities(auth_user_id, tenant_id)` ties every tenant-scoped membership to an already-linked identity from `PLT-107` — a membership cannot outrun its own identity linkage. A partial unique index (`coalesce`-guarded so nulls do not defeat uniqueness) keeps at most one active row per `(identity, tenant, layer, customer_account_ref)`. `app.principal_membership_history` and `app.enforce_principal_membership_transition()` reuse the exact trigger-enforced `active→suspended→revoked`, `revoked` terminal pattern from `PLT-105`/`107`.

### 3.2 Server context resolver and typed contract (task 2)

`app.resolve_access_context(p_auth_user_id, p_tenant_id default null, p_customer_account_ref default null)` returns `app.access_context` — always exactly one resolved principal or a `RAISE EXCEPTION`, never a "resolved but ambiguous" shape. This is a deliberate departure from `PLT-106`'s `evaluate_entitlement` (which returns an `allowed`/`reason` decision record because UI callers need the reason even on denial): a context is an identity/scope fact, not a permission decision, so there is no meaningful "denied context" to hand back — Prompt 108 §24 ("Layer is not permission") makes returning a null-ish context exactly the "permit if inconclusive" anti-pattern §3's fail-closed discipline forbids elsewhere. Resolution order: (1) an unqualified request with a live `supreme_admin` grant always resolves globally, and alone; (2) otherwise an unqualified request auto-resolves only if the identity holds exactly one active membership, else raises `ambiguous_context`; (3) a tenant-qualified request requires the target tenant `canonical_status = 'active'` (else `inactive_tenant`) and the underlying `PLT-107` identity linkage to be `active`, not merely `invited` (else `inactive_identity_link`), before matching membership rows, with `customer_account_ref` as an explicit disambiguator when a customer holds multiple accounts in one tenant. `server/contracts/access-context/access-context.ts` (Zod schemas, closed `PrincipalLayerSchema` enum, row/composite parsers) and `server/queries/access-context.ts` (`resolveAccessContext()`, classifying every raised Postgres message into a typed `AccessContextResolutionErrorCode` — `no_active_membership`/`ambiguous_context`/`inactive_tenant`/`inactive_identity_link`/`no_active_membership_for_tenant`/`resolution_failed`/`invalid_response`) are the typed contract this task calls for.

### 3.3 Base guards and context propagation (task 3)

`resolveAccessContext()` **is** the base guard: it never returns a partial context, so every future call site either has a concrete `AccessContext` or has already thrown. `server/mutations/access-context.ts` (`grantPrincipalMembership()`/`revokePrincipalMembership()`) is the write side a future admin surface will call. **Real, disclosed `NOT_RUN`:** actual wiring of this guard into REST/GraphQL/server-action/job/file entry points — no such entry point exists yet anywhere in this repository (Phase 1 Platform Core has not reached `PLT-129`–`132`, API/Jobs, or `PLT-135`/`136`, the Admin portals). The guard is built and proven against injected structural clients today, exactly as `PLT-105`/`106`/`107` proved their service layers before any live route existed, and will be composed into those surfaces when they are built, per the WBS's own sequential dependency chain.

### 3.4 Cross-layer/tenant ambiguity/escalation tests and docs (task 4)

See §4. `principal_memberships_layer_scope_shape` and `resolve_access_context`'s three named failure codes are the escalation/ambiguity defenses; §4.1 exercises each one directly, including the attempted forgeries (malformed layer/scope combinations, reviving a revoked membership, resolving against an inactive tenant or a merely-invited identity).

## 4. Tests (§27/§28/§29)

### 4.1 Real database tests (`scripts/db-tests/access-context.sql`, via `pnpm run db:test`)

9 scenario groups: global `supreme_admin` resolution (always alone, never ambiguous with a tenant-scoped grant); idempotent `supreme_admin` grant (repeated grant returns the same row, never duplicates); an unqualified request auto-resolves when exactly one membership exists; **the same identity holding two tenant-scoped memberships is ambiguous without an explicit `tenant_id`** — fails closed, then resolves deterministically once disambiguated; `customer_user` with two accounts in one tenant is ambiguous without `customer_account_ref`, resolves once supplied; the layer/scope shape constraint rejects a `supreme_admin` row carrying a `tenant_id` and a `customer_user` row missing `customer_account_ref`, both at insert time; an inactive (suspended) tenant fails closed even with a valid membership; a merely-`invited` (not yet `active`) `PLT-107` identity linkage fails closed even with a granted membership; a revoked membership no longer resolves and cannot be revived (terminal-state enforcement); resolving for an identity with zero memberships fails closed; revoking a non-existent membership fails cleanly (`no_data_found`); the same anon/authenticated schema-privilege defense-in-depth `PLT-105`/`106`/`107` established.

### 4.2 TypeScript unit tests

`server/contracts/access-context/access-context.test.ts` (4 tests): row/composite mapping; the four-layer enum is closed (an unknown layer is rejected at both the row-parse and the mutation-input boundary); default-null `tenantId`/`customerAccountRef` on an unqualified resolve input.

`server/queries/access-context.test.ts` (5 tests): exact snake_case RPC params with explicit `null` defaults (never `undefined`, so the SQL function's own defaulting is never silently relied on); every named failure prefix (`ambiguous_context`, `inactive_tenant`, `inactive_identity_link`) classifies into its own typed code; an unrecognized database message falls back to `resolution_failed` rather than leaking a raw string as a code; a null/non-object success response is rejected as `invalid_response` rather than fabricating a context.

`server/mutations/access-context.test.ts` (4 tests): exact snake_case RPC params for grant/revoke; database errors (e.g. the scope-shape check, a non-existent membership) wrap into typed `PrincipalMembershipMutationError`.
