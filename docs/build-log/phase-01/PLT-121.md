## 3. Implementation (§20)

### 3.1 Config definition/type/scope/precedence/version/dependency/lifecycle model (task 1)

`supabase/migrations/20260717130000_create_configuration_engine.sql`. `app.config_types` (registry, seeded with the 10 sourced Phase-1 types), `app.config_objects` (CONFIG_SET — a scoped namespace, `scope_shape_check` `CHECK` constraint structurally enforces Tech Arch §13.4's 6-level precedence shape: global carries neither `tenant_id` nor `scope_id`, tenant carries only `tenant_id`, company/branch/role/user require both), `app.config_versions` (CONFIG_VERSION, draft/published/archived), `app.config_items` (CONFIG_ITEM, key/value/`canonical_ref`), `app.config_dependencies` (CONFIG_DEPENDENCY).

### 3.2 Draft/validate/simulate/publish/effective/snapshot/rollback service (task 2)

`app.create_config_draft()` (idempotent, auto-vivifies the `config_object` on first use — including a real concurrent-creation race guard, catching `unique_violation` and re-selecting rather than raising), `app.set_config_items()` (bulk-replace, structural safety via `app.validate_config_value()` — recursive, depth-bounded, injection-safe), `app.publish_config_version()` (supersedes the object's prior published version, **the dependency-cycle gate runs here** — §23: "cycle... blocks publish"), `app.discard_config_draft()`, `app.rollback_config_version()` (clones the target's exact item snapshot into a brand-new version and publishes it immediately, never mutating the target — extended one level from `PLT-117`/`119`'s pattern to also clone child `config_items` rows).

### 3.3 Evaluator/cache/invalidation and dependency/cycle/conflict detection (task 3)

**`app.resolve_config()` is the real 6-level precedence walk** (Tech Arch §13.4, verbatim: user → role → branch → company → tenant → global), returning the first level with a currently-effective published version — the most specific override always wins, proven directly with a real role-level-beats-tenant-level test. `app.detect_config_dependency_cycle()` is a real, bounded (≤20 hop) recursive-CTE cycle detector, proven against both a genuine A→B→C→A cycle (blocks publish) and a non-circular A→B→C chain (publishes fine) — not merely asserted safe. `app.verify_config_version_current()` is `EXC-CFG-001`'s concrete mechanism ("stale configuration version... require refresh/revalidation before submit") — re-resolves precedence right now and compares against a caller's previously-resolved `version_id`, proven to flip from `true` to `false` the instant a newer version publishes. `server/queries/config.ts`'s `ConfigResolutionCache` mirrors Tech Arch §9's caching contract (published-only, invalidated on publish/rollback).

### 3.4 Representative platform config, access/audit/tests/docs, adoption guide (task 4)

The 10 sourced config types (§2) are the representative platform config this checkpoint ships — no fabricated rule/approval content. Authority (§26): `app.check_config_object_authority()` — Supreme Admin for `scope_level='global'`, `PLT-115`'s `app.is_support_grant_authority()` for every other level, reused verbatim across `PLT-117..121` now. `server/contracts/config/` is created in this same slice per `07_*.md` §12's resolution of `ADR-CAND-ARCH-010` ("the contracts folder cannot wait past Phase 1").
