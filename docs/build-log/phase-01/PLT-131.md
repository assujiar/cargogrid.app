## 2. Scope (§7/§11/§12)

Prompt 131 §11 allowed scope: "Import/export framework schema/migrations/service/adapters/job/file/UI primitives/tests/docs," 5-15 files, 1-3 migrations. §12 forbidden: "Full domain imports/exports, real tenant files, synchronous heavy processing."

- **`app.jobs` is created here, not by a not-yet-built generic Background Job Framework.** Research confirmed no `app.jobs`/`app.import_export_jobs`/`app.import_staging_rows` table exists anywhere in this repository before this migration, and that Prompt 132 (Background Job Framework, `CG-S6-PLT-029`, row `029`) is itself `BLOCKED` on `PLT-131` (row `029`'s own dependency list names `131`), not the reverse. `app.jobs` therefore matches `05_DATABASE_SCHEMA_WORKSTREAM.md` line 110's exact Tech Arch §32.11 field list verbatim, extended with this repository's own standing `idempotency_key` convention and this checkpoint's own import/export-specific columns. `job_type` is deliberately `CHECK`-constrained to exactly `('import','export')` for this checkpoint's own real scope — Prompt 132 will widen it in its own migration when it adds further job types (never fabricating content beyond what this checkpoint actually needs). `locked_by`/`locked_until` are real, stored, correctly-typed columns with no live-worker wiring yet — that belongs to Prompt 132's own worker, disclosed `NOT_RUN`.
- **`import_export_jobs` is not a separate table** — `05_*.md` line 113 states explicitly that it is "a specialization row-type of `jobs`, sharing the same table with `job_type = 'import'|'export'`." `app.import_staging_rows` matches that same line's exact field list.
- **An import/export schema is a per-instance config registry**, the same `<domain>:<code>` pattern `PLT-124..130` all established — `app.register_import_export_schema()` mints a dedicated `import_export:<code>` config_type via `PLT-121`'s own `app.register_config_type()`. Column definitions are bounded structural shape (`key`/`label`/`required`/`data_type` from a real allowlist: `text`/`number`/`boolean`/`date`/`email`), never a fabricated business-field catalogue.
- **Row validation is structural, not a domain write.** `app.validate_staging_row()` checks required-ness and data-type shape only — it never inserts into a canonical business table, since none exists yet in Phase 1 Platform Core. `app.commit_import_job()`'s "commit" therefore means "finalize validation and mark this framework's own job complete," not "write to a real table" — matching §24's own "Domain adapter uses canonical service/validation, not direct unsafe writes." Commit is all-or-nothing by default (§24 "no partial corruption") unless the caller explicitly passes `p_allow_partial`.
- **Malware scanning is composed from `PLT-128`, not reimplemented.** `app.stage_import_rows()` refuses to stage any row from a source file whose `malware_scan_status` is not `clean` (RPD-032), reused via a direct `app.files` lookup.
- **Cooperative cancellation**, matching `08_API_INTEGRATION_WORKSTREAM.md` §9's own "cancelling/cancelled, cooperative not hard-kill" language — a `pending` job cancels immediately; an `in_progress` job moves to `cancelling` and requires `app.acknowledge_job_cancellation()` (the real function a future worker would call) to reach `cancelled`.
- **A real, tested, disclosed-partial formula-injection (CSV injection) sanitizer.** No `docs/architecture/*.md` document names this control anywhere (confirmed via grep) — `app.sanitize_formula_injection()` is this checkpoint's own disclosed construction (OWASP-recommended leading `=`/`+`/`-`/`@` quote-prefix mitigation), deliberately NOT wired into import-row validation (a value like `-5` is legitimate on import) but meant for a future export-writer to call per output cell.
- **Server-mediated for every write**, the design `PLT-128`/`129`/`130` established — every mutating function is `service_role`-only, so none needs `SECURITY DEFINER` except `app.preview_import_job()`, the one `authenticated`-facing read function (mirrors `PLT-129`'s `list_api_keys_for_tenant` precedent).

### 5.1 Out-of-band finding: `pnpm run test`'s own glob silently never executed 26 contract test files, across this entire session

While running this checkpoint's own `pnpm run test` gate, the reported count (655/655, expected 629 baseline + 26 net new) did not reconcile against a manual per-file count of the three new PLT-131 test files (36 `test()` calls actually present in them). Investigating the discrepancy found a real, pre-existing defect in `package.json`'s `test`/`test:coverage` scripts: `server/**/*.test.ts` (unquoted) is expanded by the invoking shell (`sh`/`bash`, both without `globstar` enabled in this non-interactive environment) *before* Node ever sees it, and without `globstar`, `**` degrades to matching exactly one directory level — not the recursive-any-depth match the pattern's author clearly intended. `server/mutations/*.test.ts` and `server/queries/*.test.ts` (one level under `server/`) matched by coincidence; every `server/contracts/<domain>/<domain>.test.ts` file (two levels under `server/`, `26` files across every capability built this entire session, including this checkpoint's own new `server/contracts/import-export/import-export.test.ts`) was silently never collected or executed by any `pnpm run test` invocation at any prior checkpoint (`PLT-105` onward). Every prior build log's "`pnpm run test`: PASS" gate result was accurate for the tests it actually ran, but that set of tests was smaller than every prior checkpoint believed.

**Fix (both parts applied and verified this checkpoint):**
1. `package.json`: quoted each glob (`'server/**/*.test.ts'`, etc.) in the `test`/`test:coverage` scripts so the pattern reaches Node's own test-runner glob engine unexpanded, which correctly resolves `**` recursively regardless of shell/`globstar` state.
2. Running the corrected glob surfaced exactly one real, previously-undetected regression: `server/contracts/white-label/white-label.test.ts`'s `"rejects a data: URI logo asset (injection defense)"` test (written at `PLT-117`) failed, because `SetTenantBrandTokensInputSchema`'s own `logoAssetUrl`/`emailLogoAssetUrl` fields (`server/contracts/white-label/white-label.ts`) were declared as bare `z.string().nullable()` rather than the already-existing, already-correct `BrandAssetUrlSchema` (a regex mirroring the database's own `app.validate_brand_asset_url()` https-only, injection-safe URL shape) — `BrandAssetUrlSchema` was correctly wired into `DocumentTemplateRefsSchema`'s nested letterhead-URL fields at `PLT-117` but was overlooked for the two top-level logo fields in the same file. Fixed by pointing both fields at `BrandAssetUrlSchema` directly. The database-layer `CHECK` constraint (`tenant_brand_versions_logo_url_check`) was never at risk — this was purely a missing fail-fast/defense-in-depth layer at the TypeScript contract boundary, confirmed via `scripts/db-tests/white-label.sql`'s own already-passing coverage of the DB-level constraint.

All 25 other previously-unexecuted contract test files (`access-context`, `api`, `approval`, `audit-trail`, `auth`, `config`, `custom-domain`, `document`, `entitlement`, `field-access`, `form`, `localization`, `master-data`, `notification`, `numbering`, `org-hierarchy`, `rbac`, `role-permission`, `status`, `support-access`, `tenant`, `user-lifecycle`, `workflow`, `api-key-webhook`) passed cleanly on their first-ever real execution — no other regression was found. Full corrected baseline: 843/843 (96 test files total, verified via `find . -name '*.test.ts'` under `scripts/`, `server/`, `lib/`, `tests/`), which becomes this session's new accurate `pnpm run test` baseline going forward.

## 8. Errors / recovery / risks / issues

**Two real defects were found and fixed this checkpoint — one in this checkpoint's own new code, one a significant pre-existing, session-wide gap discovered while running this checkpoint's own mandatory `pnpm run test` gate:**

1. **This checkpoint's own db-test authoring, not the migration itself:** the first draft of the `app.validate_staging_row` delta-counting scenario group used an empty-string value (`ref: ''`) to probe the "missing required value" path, and incorrectly asserted it should be rejected — but `app.validate_staging_row`'s required-ness check only treats a JSON-null or genuinely-absent key as missing, not an empty string (a defensible, disclosed design choice: "required" means "key present," matching this checkpoint's own "structural-only, never a domain business rule" validation philosophy). The test was corrected to omit the key entirely rather than supply an empty string; the migration itself required no change. The idempotent-safe delta-counting logic itself — the scenario given the closest scrutiny this checkpoint, given its similarity to bug classes found earlier this session — proved correct on the first `db:test` run once the test fixture was fixed.

2. **Out-of-band, session-wide, pre-existing (§5.1):** `package.json`'s `test`/`test:coverage` scripts' unquoted `**` globs silently never executed any `server/contracts/<domain>/<domain>.test.ts` file (26 files, every capability built this entire session) because the invoking shell has no `globstar` enabled in this environment, degrading `**` to a single directory level. Fixed by quoting the globs so Node's own test-runner glob engine resolves them recursively instead. Running the corrected glob surfaced exactly one real regression it had been hiding: `server/contracts/white-label/white-label.ts`'s `SetTenantBrandTokensInputSchema.logoAssetUrl`/`emailLogoAssetUrl` fields were bare `z.string().nullable()` rather than the already-existing, already-correct `BrandAssetUrlSchema` (an https-only, injection-safe regex mirroring the database's own `app.validate_brand_asset_url()`, already correctly wired into the same file's `DocumentTemplateRefsSchema` for its nested letterhead-URL fields at `PLT-117` — simply overlooked for the two top-level logo fields). Fixed by pointing both fields at `BrandAssetUrlSchema`. The database `CHECK` constraint was never at risk; this was a missing TypeScript-layer fail-fast/defense-in-depth guarantee only. All 25 other previously-unexecuted contract test files passed cleanly on their first-ever real execution. This finding and both fixes are disclosed here rather than silently folded into "gates passed" — `pnpm run test` reporting a clean, complete result is itself one of this checkpoint's own mandatory gates, and a broken gate silently under-testing is exactly the "fabricated evidence" failure mode this build's own governance forbids.

No new issue opened. Real, disclosed `NOT_RUN`: live wiring of any of these RPCs into an actual file-upload UI, a real streaming CSV/Excel parser, or a real signed-URL export-download flow (no such surface exists anywhere in this repository yet — Prompt 131 §15 itself defers full-module UI to a later task); `locked_by`/`locked_until` distributed worker-locking semantics and any live retry scheduler (Prompt 132, Background Job Framework, owns the actual worker); `app.sanitize_formula_injection()` being called by any real export-file writer (no live file-generation step exists yet, the same "algorithm real, live generation deferred" pattern `ADR-0011`/`ADR-0012` already established); `pnpm run test:e2e` (sandbox Playwright browser-binary revision skew, unrelated to this diff, same condition disclosed at `PLT-117..130`).
