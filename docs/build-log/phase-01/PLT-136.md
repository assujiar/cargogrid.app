## 3. Scope (§7/§11/§12) — bounded the same way `PLT-135` was, for the same disclosed reason

Prompt 136 §11 allowed scope: "Supreme portal shell/routes/components/privileged adapters/tests/docs and bounded workflows," 5-15 files, 1-3 migrations. §12 forbidden: "Direct client service-role/database, unreviewed destructive bulk action, domain feature admin."

- **Guard is simpler than the Tenant Admin portal's own** — Supreme Admin membership is tenant-independent. `app.resolve_access_context(auth_user_id, null)`'s own migration comment states a live Supreme Admin grant "always resolves first and alone" when `p_tenant_id` is omitted — no tenant-slug/RLS lookup step exists in `lib/portal/supreme-admin-guard.ts` at all, unlike `tenant-admin-guard.ts`'s three-step resolution.
- **The global tenant list uses the RLS-scoped client directly, not service-role** — confirmed by direct inspection of `app.has_active_tenant_membership()`'s own body (`... or app.is_supreme_admin(p_auth_user_id)`), a Supreme Admin's `authenticated` session already satisfies `app.tenants`'s own `tenants_select_own_tenant` RLS policy for every row, tenant-independent. No service-role bypass is needed for this specific read path — the service-role client is still used, exactly once, for the guard's own `resolve_access_context` call (that RPC's grant is `service_role`-only, unchanged from `PLT-135`'s own finding).
- **Bounded to shell + one read-only child slice (global tenant list), not the full control plane** (§12's "no domain feature admin," and equally no "unreviewed destructive bulk action"). Package/subscription management, platform config, support-grant administration, audit-log browsing, and system health — all real, already-shipped or partially-shipped backend capability — are **not** built here, mirroring `09_UX_DESIGN_SYSTEM_WORKSTREAM.md` §3's own route map (`app/(supreme)/supreme/tenants`, `.../subscriptions`, `.../system` are named as separate route segments, not one page) and §14's atomic backlog discipline (management-workflow UI is sequenced as its own later slice behind "Portal shells").
- **No re-authentication/impact-preview/confirmation machinery is built** (Prompt 136 §16/§20 task 2's own requirements) — deliberately, since this checkpoint ships **zero mutating action**. Building that UI now, with nothing real to gate, would be exactly the unresolved-placeholder pattern this build's own governance forbids; it is the responsibility of whichever future slice adds the first real Supreme-scoped mutation (e.g. a tenant status transition) to build the guard alongside it.
- **RPD-022 (Supreme Admin's absolute-CRUD exception) is disclosed structurally, not just in this document** — `app/(supreme)/supreme/layout.tsx` renders a real, persistent `Banner` (new `components/ui/banner.tsx` primitive) on every page in this portal, quoting the same disclosure `AGENTS.md`'s own "Supreme Admin risk rule" requires, satisfying Prompt 136 §18's "disclosing Supreme can also alter audit itself" and §24's "must not pretend immutable/tamper-proof evidence" as a real, unavoidable render, not documentation alone.
- **The shared `app/(public)/login` page/action (`PLT-135`) is extended, not forked** — the organization field is now optional: a blank submission redirects to `/supreme` instead of `/{slug}/admin`. This is not a new login surface (`app/(public)/` remains the one shared, portal-agnostic entry point) — the guard on each portal, unchanged, is still the real authorization boundary; picking the wrong redirect target here only costs an extra guard-denied render, never grants anything.
- **Zero new migration, zero new schema** — Prompt 136 §13's "No direct client DB" is honored: the tenant list reads `app.tenants` (`PLT-105`) directly, no new table/column/function.
