## 3. Scope (§7/§11/§12) — including two real, disclosed gaps this checkpoint's own kickoff found

Prompt 135 §11 allowed scope: "Tenant Admin portal shell/routes/components/tests/docs and exact bounded workflow adapters," 5-15 files, 1-3 migrations. §12 forbidden: "Supreme/Customer/domain portals, direct DB access, all-admin-pages mega task."

- **Gap found #1 — the "Design-system foundation" atomic slice was never executed as its own capability.** `docs/architecture/09_UX_DESIGN_SYSTEM_WORKSTREAM.md` §14's own atomic backlog sequences "Design-system foundation" (`components/ui/` primitives, token mechanism implementation) as a *separate* slice that "Portal shells" depends on — but no row `105`–`134` in the execution index built it; `docs/standards/DESIGN_SYSTEM.md` §8 confirms "No `components/ui/`, `lib/`, or token file exists yet." Reconciled the same way `00_PLATFORM_CORE_WBS.md` §3 already resolved the identical question for `server/contracts/`: `components/ui/` is created incrementally by whichever capability needs it first, not a blocking separate prerequisite (that §3 text explicitly lists `components/ui/` among the allowed Phase-1 paths with no capability ID of its own). This checkpoint builds only the one primitive it actually consumes (`components/ui/button.tsx`), not `09_*.md` §4.1's full inventory (tables/forms/dialogs/timeline/upload/chart primitives) — those are built by whichever later slice first needs them.
- **Gap found #2 — CargoGrid's own base brand identity (primary/secondary color, logo, typography family) is an explicitly open Product/Design decision** (`docs/standards/DESIGN_SYSTEM.md` §3: "No hex color, logo, or typography family... exists anywhere in `docs/blueprint/**`... inventing one here would be exactly the kind of unauthorized product decision this build's governance has consistently refused to make"). This is the first checkpoint to actually hit that blocking point (every prior UI-adjacent checkpoint was backend-only). Resolved by building the entire shell from the **already-decided** neutral/semantic token set only (`docs/standards/DESIGN_SYSTEM.md` §2.1) — `--color-primary`/`--color-secondary` are explicitly mapped to `neutral-900`/`neutral-600` as a disclosed placeholder (`app/globals.css`'s own header comment), never a fabricated "brand blue." The portal is fully functional and accessible today; only these two CSS variable values need to change once Product/Design resolves §3.
- **`app.resolve_access_context` (PLT-108) is `service_role`-only, not `authenticated`-granted** — confirmed by direct inspection of its migration's own grant statement, not assumed. The guard therefore uses two Supabase clients: the RLS-scoped `authenticated` client (session lookup, tenant-by-slug lookup via `app.tenants`'s own `tenants_select_own_tenant` RLS policy) and a service-role client (`lib/supabase/service-role.ts`, server-only, never imported by a Client Component) for the one privileged RPC call — the same "explicit actor, service-role execution" architecture every backend capability this session already established, now extended to the UI layer for the first time.
- **No new migration, no new schema** — Prompt 135 §13's own "No direct client DB... new portal-specific schema is prohibited unless separately justified" is honored structurally: the Users list reads `app.users_directory` (`PLT-114`/`116`'s own field-masked, tenant-scoped view), reused directly, zero new table/column/function.
- **Bounded to shell + one read-only child slice (Users list), not the full admin surface** (§12's "no all-admin-pages mega task"). Roles/hierarchy/entitlements-view/configuration/branding management UI — all real, already-shipped backend capability (`PLT-109..111`, `117`, `121`) — are deliberately **not** built here; `docs/architecture/09_UX_DESIGN_SYSTEM_WORKSTREAM.md` §14's own atomic backlog places "Configuration Studio UI"/"White-label Studio" as separate slices *depending on* "Portal shells," not part of it — this checkpoint's own scope call is consistent with that already-ratified sequencing, not an invented narrowing.
- **A minimal, real `app/(public)/login` page is included** — the portal guard's only entry point; `app/(public)/` is a shared, portal-agnostic route group (not Supreme/Customer-portal-specific), so building the minimum sign-in path here is the enabling prerequisite for any portal to be reachable at all, not scope creep into another portal.
- **File count (26 new/changed files, including `package.json`/lockfile/config) exceeds the "5-15" guideline** — disclosed as inherent to being the first UI-bootstrapping capability in this repository (Next.js `app/` router, Tailwind v4, Radix copy-in, two Supabase client factories all land for the first time here), the same class of outsized "foundational" scope `PLT-105` (first migration) and `PLT-121` (first shared engine) already had disclosed.

## 4. Implementation (§20)

### 4.1 Design-system foundation (minimal, per §3's gap-#1 reconciliation)

### 4.2 Supabase client wiring

`lib/supabase/server.ts` (RLS-scoped `authenticated` client, `@supabase/ssr`'s documented `getAll`/`setAll` cookie pattern, reuses `lib/auth/session-cookie-options.ts`'s already-decided attributes, `PLT-107`). `lib/supabase/service-role.ts` (service-role client, server-only, no cookies — see §3's `resolve_access_context` finding).

### 4.3 Guard and portal shell

`lib/portal/tenant-admin-guard.ts` (pure, five-outcome discriminated union: `unauthenticated`/`tenant_not_found_or_not_member`/`tenant_suspended`/`forbidden`/`allowed`; deliberately does not distinguish "tenant doesn't exist" from "not a member" — no tenant-enumeration signal, the same posture `app.resolve_tenant_by_domain` already established). `lib/portal/tenant-admin-guard-deps.server.ts` (real wiring), `lib/portal/resolve-tenant-admin-access.server.ts` (`React.cache()`-memoized per request so the layout and every nested page share one guard resolution). `app/(tenant)/[tenantSlug]/admin/layout.tsx` (the shell — nav chrome, denied-state page, redirect-to-login for unauthenticated). `app/(tenant)/[tenantSlug]/admin/page.tsx` (Home, minimal, no dead buttons). `app/(tenant)/[tenantSlug]/admin/users/{page,loading}.tsx` (the one bounded child slice) plus `server/queries/portal-users.ts` (paginated `app.users_directory` read, `pageSize` clamped to `[1,100]`).

### 4.4 Login

`app/(public)/login/{page,actions}.tsx` — email/password/organization-slug form, a real `signInWithPassword` Server Action, post-login redirect re-validated through `lib/auth/redirect-allowlist.ts` (`PLT-107`) as defense in depth even though the target is server-constructed, not user-supplied.

### 4.5 Real defect found and fixed during this checkpoint's own gate verification

Running `pnpm run lint` after a real `next build` (to prove the app actually compiles, not just typechecks) surfaced that `eslint-config-next`, consumed directly as a flat-config array (not through the deprecated `next lint` CLI wrapper that auto-injects this), does **not** exclude `.next/`/`playwright-report/`/`test-results/` on its own — `eslint .` began linting Next's own minified production bundle and a stray Playwright HTML-report bundle as if they were source, producing thousands of nonsensical "React Hook called outside a component" errors against variable named `a2` in minified code. Fixed by adding an explicit `ignores` entry to `eslint.config.js` (load-bearing, not decorative — proven by re-running `next build` then `pnpm run lint` again, clean). This is a real, previously-latent gap that could not surface before this checkpoint, since no prior task ever ran `next build` (no `app/` code existed to build).
