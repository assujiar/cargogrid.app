## 2. Scope (§7/§11/§12)

Prompt 129 §11 allowed scope: "API key/webhook schema/migrations/service/auth/delivery/tests/docs/runbook," 5-15 files, 1-3 migrations. §12 forbidden: "Live third-party endpoints without authority, plaintext keys, generic provider integration hub."

- **API key scopes can only narrow what the issuing actor's own role already holds, never widen it** (`08_API_INTEGRATION_WORKSTREAM.md` line 106). A scope string is shaped `<resource_module_code>:<action>`, `app.permissions`'s own natural key (`PLT-111`) reused directly. `app.create_api_key()` calls `app.evaluate_permission()` (`PLT-112`) for every requested scope, composing the existing RBAC evaluator rather than re-implementing a parallel authority check — proven with a real role/permission/assignment chain in the db-test, not merely asserted.
- **A raw API key is never stored** — only `key_hash` (`digest(raw_key, 'sha256')`) and `key_prefix` persist; the raw value returns exactly once from `app.create_api_key()`/`app.rotate_api_key()`. **Rotation is overlap-window-based** (`08_*.md` line 107), never an instant cutover — the old key's continued validity is bounded to a caller-supplied window (0–10080 minutes), never extending an already-sooner expiry.
- **A webhook signing secret cannot be a one-way hash** (unlike an API key) — HMAC verification is symmetric, so the platform's own delivery worker needs the raw secret to *compute* an outgoing signature. `secret_value` is stored directly but the table carries zero `authenticated` grant; the tenant receives it exactly once, the same "shown once" contract implemented via access control rather than hashing (the pragmatic reading of §13's "secret refs," disclosed not assumed).
- **`ADR-0011` resolves `ADR-CAND-ARCH-018`'s signature/timestamp/auto-disable sub-questions**: HMAC-SHA256 signing, a 5-minute timestamp tolerance, a 10-consecutive-failure auto-disable threshold — the candidate's own recommendation. `app.compute_webhook_signature()`/`app.verify_webhook_signature()` are genuinely provable without any live HTTP call (pure cryptographic computation), reusing `pgcrypto`'s `hmac()`/`digest()`/`gen_random_bytes()`, already unconditionally enabled since `PLT-105`, zero new extension dependency.
- **Webhook delivery itself is never real** — no live HTTP client exists anywhere in this repository (disclosed `NOT_RUN`). `app.record_webhook_delivery_attempt()` is the real, tested, bounded adapter interface a future delivery worker would call; this migration never calls it against a fabricated HTTP response itself.
- **A bounded, disclosed-partial SSRF check** — `app.validate_webhook_url()` requires `https://` and rejects a literal `localhost`/loopback/private/link-local IPv4/IPv6 host at *registration* time. No architecture document specifies SSRF controls beyond naming the dimension (confirmed absent from every `docs/architecture/*.md` search this checkpoint performed); this cannot defend against DNS-rebinding-based SSRF, which requires live DNS resolution at actual delivery time — left to the not-yet-built delivery worker.
- **A webhook event type is a Supreme-registered registry entry** (`app.webhook_event_types`), mirroring `PLT-120`'s registry-not-enum discipline, but mints no `app.config_types` entry (no versioned definition content exists for a subscribable code, unlike `PLT-124..128`'s definitions). Zero real domain event type seeded.
- **This entire capability is server-mediated only** (the design `PLT-128` established) — every mutating function is `service_role`-only. Consequently none of them needs `SECURITY DEFINER` except the two authenticated-facing listing functions, which need it to read `service_role`-only tables while excluding `key_hash`/`secret_value` from their result shape.
- **Scope boundary**: `rate_limit_per_minute` is real and stored, but its *enforcement* is disclosed `NOT_RUN` — no live API gateway exists yet (Prompt 130, `BLOCKED`). Inbound webhook receiving (this platform receiving FROM an external provider) is a different direction than this capability's outbound tenant-subscribed delivery and is left to its own future task.

## 3. Implementation (§20)

### 3.1 API key lifecycle

`supabase/migrations/20260719150000_create_api_key_webhook_primitives.sql`. `app.api_keys` (hashed metadata only). `app.create_api_key()` — authority, name/scope-format/rate-limit/expiry validation, per-scope `evaluate_permission()` check, `cgk_`-prefixed raw key generation, returns the raw key once. `app.rotate_api_key()` — overlap-window bounded rotation (0 = immediate revoke). `app.revoke_api_key()` — idempotent. `app.authenticate_api_key()` — the real authentication entry point (disclosed `NOT_RUN` as a live HTTP consumer), lazily flips a past-expiry row to `expired`. `app.api_key_has_scope()`, `app.list_api_keys_for_tenant()` (`SECURITY DEFINER`, excludes `key_hash`).

### 3.2 Webhook endpoint/event-type lifecycle

`app.webhook_event_types` (registry) + `app.register_webhook_event_type()`. `app.webhook_endpoints`/`app.webhook_subscriptions`. `app.validate_webhook_url()` (the SSRF check). `app.register_webhook_endpoint()` — validates URL + every event_type_code, returns the raw secret once. `app.rotate_webhook_secret()`, `app.disable_webhook_endpoint()` (idempotent), `app.reenable_webhook_endpoint()` (resets the failure counter), `app.list_webhook_endpoints_for_tenant()` (`SECURITY DEFINER`, excludes `secret_value`).

### 3.3 Delivery, signing, audit, and RLS

`app.webhook_deliveries`/`app.webhook_delivery_attempts`. `app.queue_webhook_delivery()` — idempotent per (tenant, endpoint, idempotency_key), payload validated via `app.validate_config_value()` (`PLT-121` reused), fans out to every active subscribed endpoint, an empty subscriber set is a normal non-error outcome. `app.record_webhook_delivery_attempt()` — the bounded delivery adapter; exponential backoff on transient failure, `dead_letter` at `max_attempts`, endpoint auto-disable at 10 consecutive failures (`ADR-0011`), a single success resets the counter; applies the `'success'`/`'failure'` audit-result mapping proactively (`PLT-127`'s own lesson). `app.compute_webhook_signature()`/`app.verify_webhook_signature()` — real HMAC-SHA256 + 5-minute tolerance. Every lifecycle mutation self-captures a canonical `app.capture_audit_event()` entry. `app.webhook_event_types` is broadly readable RLS; every other table carries zero `authenticated` grant at all (matches `app.notification_delivery_attempts`' precedent). Per `ERR-2026-004`, the migration ends with its own explicit `revoke execute on all functions in schema app from public;`.

## 8. Errors / recovery / risks / issues

One real defect found and fixed via an actual failing `pnpm run db:test` run (not self-review): `app.rotate_api_key()` and `app.rotate_webhook_secret()` both declared `returns table (id uuid, ...)`, which auto-declares `id` as a PL/pgSQL variable in that function's scope — their internal `select * into v_old from app.api_keys where id = p_key_id`-shaped lookups then raised "column reference \"id\" is ambiguous" the moment the db-test actually exercised rotation. This is the identical bug class `PLT-117` found in `app.evaluate_tenant_brand()` (a `RETURNS TABLE` output column shadowing a same-named real table column) — recognized immediately from that precedent and fixed the same way, by table-qualifying every affected bare `id` reference (`app.api_keys.id`/`app.webhook_endpoints.id`) rather than renaming the output column (preserving the exact `id`-first field ordering every other list/create function in this migration already uses).

No new issue opened. Real, disclosed `NOT_RUN`: live wiring of any of the sixteen TS entry points into a REST/GraphQL/admin-UI surface (no such surface exists yet, Prompt 130 still `BLOCKED`); a real external HTTP delivery worker or API-gateway authentication middleware (no such infrastructure exists anywhere in this repository — `app.authenticate_api_key()`/`app.record_webhook_delivery_attempt()` are the real, tested entry points a future one would call); `rate_limit_per_minute` enforcement against real request volume (stored and validated, not yet enforced — no live gateway exists); `pnpm run test:e2e` (sandbox Playwright browser-binary revision skew, unrelated to this diff, same condition disclosed at `PLT-117..128`).
