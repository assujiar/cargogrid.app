## 2. Scope (§7/§11/§12)

`00_PLATFORM_CORE_WBS.md` row 10: allowed scope is "Approved RLS helper/policy migrations/grants/tests/docs and tenant-key indexes"; DB impact is "Enable/retain RLS, tenant-aware keys/FKs/indexes, explicit operation policies and least-privilege grants." Forbidden: disabling RLS, broad grants/service bypass, unrelated domain tables, applied migration edits (§12) — every table already had RLS *enabled* since its own migration (`PLT-105`–`112`); this checkpoint never disables it anywhere, adds no business-domain table, and edits zero already-applied migration file.

This is stage 7 of the 8-stage access flow (`docs/architecture/06_RLS_RBAC_WORKSTREAM.md` §3: "7 RLS enforcement at database"). **The pivotal, disclosed change this checkpoint makes:** `authenticated` has held **zero** `USAGE` grant on the `app` schema since `PLT-105` — every prior "defense in depth" db-test proved `anon`/`authenticated` both hit `insufficient_privilege` *before RLS was ever evaluated*. This migration is the one that finally grants `authenticated` a real, narrow, RLS-governed path — the first checkpoint where RLS policies become reachable by an actual authenticated session rather than a theoretical policy nothing could ever exercise.

**Disclosed scope boundaries (Prompt 113 §4: "INITIAL policies," not full production authorization):**
- **SELECT only.** `authenticated` receives no `INSERT`/`UPDATE`/`DELETE` grant on any table in this migration. Every write in this repository flows through an already-`VERIFIED`, `service_role`-only plpgsql function (`app.provision_tenant`, `app.invite_user`, `app.assign_role`, ...). Granting direct table writes to `authenticated` today would let a session bypass those functions' idempotency, lifecycle-transition, and cross-table-sync logic entirely. Writing `INSERT`/`UPDATE`/`DELETE` RLS policies with no matching grant to exercise them would be untestable busywork, not real coverage — proven negatively in §4.1 instead (a direct `INSERT` attempt as `authenticated` still hits `insufficient_privilege`).
- **Primary tenant-scoped tables only** — `app.tenants`, `app.tenant_entitlements`, `app.tenant_entitlement_overrides`, `app.tenant_user_identities`, `app.principal_memberships`, `app.org_units`, `app.users`, `app.roles`, `app.role_versions`, `app.role_assignments` (10 tables). Every capability's own bounded `*_history` audit table is deliberately left `service_role`-only here — `PLT-116` (Audit Trail Foundation) is the more coherent single decision point for a canonical audit-read policy across all seven of them at once, rather than seven ad hoc one-off decisions now. Global, non-tenant-scoped catalogues (`app.entitlement_modules`/`features`/`packages`, `app.permissions`) are out of this checkpoint's scope entirely — no live UI consumes them yet.

## 3. Implementation (§20)

### 3.1 Inventory and policy matrix (task 1)

See §2's table list above and §4.1's coverage — every table carrying a direct `tenant_id` column (or, for `app.role_versions`, one derivable through its parent `app.roles` row) gets exactly one `SELECT` policy using the same reusable predicate.

### 3.2 Stable helper functions, grants, explicit policies (task 2)

`supabase/migrations/20260716105512_create_rls_tenant_policies.sql`: `app.is_supreme_admin()` and `app.has_active_tenant_membership()` are `STABLE`, non-recursive, `SECURITY DEFINER` functions (the search-path-injection footgun closed explicitly via `set search_path = app, pg_temp`, a disclosed, correct mitigation, not an oversight) — `SECURITY DEFINER` specifically because `authenticated` is never granted direct `SELECT` on `app.tenant_user_identities`/`app.principal_memberships` themselves, so the helper must run with elevated privilege to answer the membership question while the caller only ever sees a boolean. Every policy below reuses these two functions rather than duplicating the membership predicate ten times.

**A real bug found and fixed during authoring, not left in:** the first draft of the `app.principal_memberships` policy used `tenant_id is null or app.has_active_tenant_membership(tenant_id)` — this would have let *any* authenticated caller see *every other identity's* global `supreme_admin` row (since `tenant_id is null` is true for all of them, regardless of whose row it is), leaking who is a platform Supreme Admin. Fixed to `(tenant_id is null and auth_user_id = auth.uid()) or (tenant_id is not null and app.has_active_tenant_membership(tenant_id))` — proven correct in §4.1 (a Supreme Admin sees their own global row; an ordinary member sees zero of another identity's).

### 3.3 Tenant-aware constraints/indexes, safe service/job/migration paths (task 3)

No new index was needed — every table in scope already leads its indexes with `tenant_id` from its own defining migration (`PLT-105`–`111`), confirmed by inspection rather than assumed. `service_role`'s existing `BYPASSRLS` grant (established at `PLT-105`) is unchanged and unaffected — proven still working in §4.1's final scenario group.

### 3.4 Cross-tenant/API/job/report/file tests, performance evidence (task 4)

## 4. Tests (§27/§28/§29)

### 4.1 Real database tests (`scripts/db-tests/rls-tenant-policy.sql`, via `pnpm run db:test`)

10 scenario groups, using a real simulated authenticated session (`set local role authenticated; set local request.jwt.claims to '{"sub": "...", "role": "authenticated"}';` — the exact mechanism a real Supabase/PostgREST deployment uses, not a stand-in): a tenant member sees only their own tenant's row via `app.tenants`; the same member sees only their own tenant's `org_units`/`users`; the same member sees only their own tenant's `roles`/`role_versions` (proving the derived-through-`roles` join policy works); **a Supreme Admin sees rows across every tenant**, not just one; **the Supreme Admin's own global `principal_memberships` row is visible to itself but invisible to an ordinary member** (the exact fix from §3.2, proven negatively); an identity with zero tenant memberships anywhere sees zero rows across every RLS-governed table; `authenticated` has no write grant anywhere — a direct `INSERT` attempt still hits `insufficient_privilege`; `anon` remains fully denied at the schema-privilege layer (this checkpoint only ever widened `authenticated`, never `anon`); `service_role` is unaffected, still bypassing RLS entirely as before.

### 4.2 Regression test updated (task 5's "compare baseline/post-change evidence"; Prompt 113 §29)

`scripts/db-tests/tenant-lifecycle.sql`'s own `PLT-105` "defense in depth" test asserted `authenticated` hits `insufficient_privilege` on `app.tenants` — now stale, since this checkpoint's own migration legitimately grants `authenticated` real `SELECT` access. Updated in place (not weakened — the new assertion still requires zero visible rows for a session with no JWT claims set, RLS now doing the denial instead of the schema-privilege layer) with a comment explaining the change and pointing to `rls-tenant-policy.sql` for the full authenticated-session coverage.
