## 2. Scope (§7/§11/§12)

Prompt 126 §11 allowed scope: "Form/custom-field engine schema/migrations/service/renderer/tests/docs and isolated example," 5-15 files, 1-3 migrations. §12 forbidden: "Domain forms/pages, arbitrary code/SQL, uncontrolled EAV, full builder portal."

- **No domain form exists anywhere in this migration.** The one representative example (a small "Vendor Onboarding" custom-field set: `tax_id`, `is_preferred_vendor`, a conditionally-visible sensitive `credit_note`) is built and proven entirely inside `scripts/db-tests/form.sql` — no form/field row of any kind is seeded into the migration itself.
- **EAV query explosion is structurally avoided** (§13, explicitly named as a forbidden pattern). `app.custom_field_values` holds exactly one row per `(tenant, entity_type, entity_id)`, with every field's value packed into a single `values jsonb` column — never one row per field. Reading or writing an entity's custom fields is always a single-row operation, proven directly (an update to an existing entity stays at exactly one row).
- **A form *definition* is not a new table family**, exactly like `PLT-122`/`123`/`124`/`125`'s definitions — it is `PLT-121`'s own Configuration Engine, reused directly. `app.register_form()` mints a dedicated `form:<code>` config_type per form (the same "registry, not enum" composition `PLT-124`'s Status Engine established, for the identical structural reason: the single shared seeded `'form'` type can host at most one tenant-scoped object).
- **No arbitrary code, ever** (§16/§24). Field types and validators are both real, bounded allowlists. A field's visibility `condition` is a bounded, structural `{field_code, operator, value}` comparison — never an expression language — and may only reference an *earlier*-declared field, which makes a condition cycle structurally impossible by construction rather than requiring a separate cycle detector.
- **`file` is deliberately not in the field-type allowlist.** No file-storage plumbing exists anywhere in this repository yet.
- **Sensitive custom-field values get a real, narrower read gate** (§18). A values row with no sensitive field is readable by any active tenant member; a row with at least one `sensitive: true` field additionally requires being the original submitter or holding the same definition-admin-grade authority `app.check_config_object_authority()` already grants for managing the form itself — proven with a real third-party regular `org_user` correctly denied.
- **Values remain interpretable under their historical definition** (§19/§25). `app.custom_field_values.config_version_id` binds every stored value set to the exact published definition snapshot active when captured, the same "never renumbers/reinterprets history" discipline `PLT-125`'s numbering bootstrap established.

## 3. Implementation (§20)

### 3.1 Form registry and structural validation

`supabase/migrations/20260719120000_create_form_custom_field_builder.sql`. `app.form_registry` (registry, code + a dedicated `form:<code>` config_type minted per form). `app.register_form()` — idempotent, Supreme-Admin-only, audited. `app.validate_form_definition(p_version_id)` reads the single `fields` config_item (a jsonb array in declaration order) and raises 9 distinct named exceptions covering duplicate codes, missing labels, disallowed types, missing `select`/`multiselect` options, disallowed/oversized validators, and forward/self-referencing conditions. `app.publish_form_definition()` runs this gate then composes `PLT-121`'s own `app.publish_config_version()`.

### 3.2 Submission-time validation and safe value storage

`app.evaluate_field_condition()` is the bounded structural condition evaluator. `app.validate_custom_field_values(p_config_version_id, p_values)` is the real, exhaustive submission-time gate: rejects any key not a declared field code, enforces `required` only while a field's own condition currently makes it visible, type-checks every JSON value shape against its field's declared type (including `select`/`multiselect` membership in the field's own `options`), and enforces every declared validator. `app.custom_field_values` reuses `PLT-121`'s own `app.validate_config_value()` directly as its `values` column `CHECK` constraint — the exact recursive depth/size/injection-safety gate every prior `config_items` row already trusts, composed rather than reimplemented.

### 3.3 Lifecycle functions, access, and audit

`app.set_custom_field_values()` — idempotent on `(tenant_id, idempotency_key)`, requires a published definition, runs the real validation gate *before* ever writing (proven: an invalid submission leaves zero rows), upserts the single current-values row per entity. `app.get_custom_field_values()` — the sensitive-field-gated read (`SECURITY DEFINER`, since it must read `app.config_items` internally to determine sensitivity). `app.check_custom_field_authority()` is the same looser instance-level `has_active_tenant_membership()`-or-Supreme posture `PLT-122`/`123`/`125`'s own instance-level checks use. RLS: `app.custom_field_values` gets direct RLS-governed `SELECT` for `authenticated` (normal tenant business data); the finer sensitive-field distinction — which ordinary table-level RLS cannot express per-field — is enforced entirely inside `app.get_custom_field_values()`, the same "the function itself is the access-control boundary" posture `PLT-116`'s `audit_logs` and `PLT-122`'s `workflow_transition_history` already use. Every lifecycle mutation self-captures a canonical `app.capture_audit_event()` entry — no bespoke `*_history` table. Per `ERR-2026-004`, the migration ends with its own `revoke execute on all functions in schema app from public;` before role-specific grants.
