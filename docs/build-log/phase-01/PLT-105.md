## 4. Tests (Â§27/Â§28/Â§29)

### 4.1 Real database tests (`scripts/db-tests/tenant-lifecycle.sql`, executed via `pnpm run db:test`)

Run against a real, disposable Postgres database (`scripts/db-tests/run.sh` â€” drop/create, apply every `supabase/migrations/*.sql` in order, run every `scripts/db-tests/*.sql`, drop again). Every assertion is a `DO` block that `RAISE`s on failure:

1. **Idempotent provisioning:** a second `provision_tenant()` call with the same `idempotency_key` (deliberately passing a *different* slug/name) returns the original row unchanged; exactly 1 row exists; exactly 1 history row exists (the retry does not append a duplicate).
2. **Duplicate slug, different key:** rejected with `unique_violation`.
3. **Full canonical lifecycle:** `provisioning â†’ active â†’ suspended â†’ active â†’ terminated`, asserting `activated_at`/`deactivated_at`/`terminated_at`/`record_version` at each step and the corresponding history row.
4. **Terminal-state enforcement:** any transition attempted out of `terminated` is rejected (`check_violation`).
5. **Invalid (non-adjacent) transition:** `active â†’ provisioning` is rejected.
6. **Legal hold:** termination is rejected while `legal_hold = true`; succeeds immediately once cleared.
7. **Non-existent tenant:** `transition_tenant_status()` on an unknown id raises `no_data_found`, not a silent no-op.
8. **Access control (defense in depth):** `anon`/`authenticated` (created as real Postgres roles mirroring Supabase's own role model) are denied at the schema-privilege layer (`insufficient_privilege`, never reach RLS); `service_role` has explicit access and sees every provisioned tenant.

All 8 scenario groups pass against Postgres 16 locally this checkpoint; CI runs the identical script against `postgres:17` (Tech Arch's target version) via the new `db` job's service container.
