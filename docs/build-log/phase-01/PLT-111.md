## 3. Implementation (¬ß20)

### 3.1 Permission catalogue and role/version taxonomy (task 1)

`supabase/migrations/20260716103445_create_roles_permissions.sql`: `app.permissions` (see ¬ß2), `app.roles` (a stable, tenant-scoped role identity ‚Äî title/description are tenant-configurable, "permission semantics remain canonical" per Prompt 111 ¬ß24), `app.role_versions` (the actual, immutable-once-published permission snapshot).

### 3.2 Versioned draft/validate/publish/clone/archive and binding model (task 2)

`app.create_role_version()` is idempotent (at most one draft per role, enforced by a partial unique index, not just application discipline). `app.set_role_version_permissions()` replaces a draft's full binding set, rejected once the version is no longer `draft` ‚Äî proven in ¬ß4.1 that publishing genuinely makes bindings immutable. `app.publish_role_version()` archives the role's previously published version on publish (supersession, the same pattern `PLT-106`'s `assign_entitlement()` established) ‚Äî a version's bindings are never mutated after publish (Prompt 111 ¬ß22). `app.clone_role_version()` copies a published/archived version's bindings verbatim into a fresh draft; cloning a draft itself is rejected (nothing stable exists yet to copy, proven in ¬ß4.1).

### 3.3 Assignment eligibility, impact/dependency checks, cache invalidation (task 3)

`app.assign_role()` validates: the target version is `published` (not `draft`/`archived`); the role belongs to the requesting tenant (cross-tenant rejected); the assignee holds an active `app.users` profile in that tenant (`PLT-110` dependency, real not assumed); and ‚Äî the concrete "no self-escalation" mechanism (Prompt 111 ¬ß16/¬ß23) ‚Äî **an actor may not assign to themselves a role version carrying any `protected` permission**, proven directly in ¬ß4.1 alongside its two necessary counter-cases (the same protected version assigned to a *different* user succeeds; a non-protected version may still be self-assigned). The actor's real identity (`p_actor_auth_user_id`) is a distinct parameter from the free-text `p_granted_by` audit label every other function in this session already uses ‚Äî a deliberate, disclosed choice so self-escalation cannot be spoofed via a text field. **Cache invalidation** is deferred: no query layer yet caches role/permission data (unlike `PLT-106`'s `EntitlementCache`), because no RBAC evaluator (`PLT-112`) exists yet to need one ‚Äî there is nothing to invalidate before there is something that reads and caches a decision.

### 3.4 Builder contract, tests, audit, downstream RBAC integration (task 4)

`server/contracts/role-permission/`, `server/queries/role-permission.ts` (catalogue/role lookup), `server/mutations/role-permission.ts` (the 8-function lifecycle, each with unit tests) are this checkpoint's "builder view model" ‚Äî Prompt 111 ¬ß15 explicitly defers the full portal page to `PLT-135`. `app.role_lifecycle_history` records `role_created`/`version_drafted`/`permissions_set`/`published`/`cloned`/`archived`/`assigned`/`revoked` ‚Äî the same bounded, capability-scoped audit pattern `PLT-105`‚Äì`110` each established.
