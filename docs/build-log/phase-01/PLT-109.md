## 3. Implementation (§20)

### 3.1 Node types, allowed relationships, lifecycle, canonical codes (task 1)

`supabase/migrations/20260716101726_create_org_units.sql`: `app.org_units` (`unit_type` closed to the four types, `code` unique per tenant regardless of type, `status` active/inactive). `app.enforce_org_unit_parent_shape()` is a single trigger enforcing three things at once, proven separately in §4.1: the allowed-parent-type matrix (§2 above), cross-tenant parent rejection, and the cycle guard (a node can never be moved under one of its own descendants).

### 3.2 Cycle-safe hierarchy service and query model (task 2)

Ancestry is a materialized `path uuid[]` column (root-first ancestor ids, self excluded) plus a `depth` integer, maintained by `app.recompute_org_unit_path()` on every insert/parent change — chosen specifically so ancestor/descendant reads never need a recursive CTE (Prompt 109 §17: "avoid recursive N+1"). A GIN index on `path` makes `path @> array[node_id]` (descendant lookup) and cycle-detection both indexed operations, not sequential scans.

### 3.3 CRUD/move/deactivate with dependency/conflict/concurrency checks (task 3)

`app.create_org_unit()` is idempotent on `(tenant_id, code)` — a repeated create with matching type/parent returns the original row; a repeated create with the *same code but a different type/parent* raises `org_unit_code_conflict` (a genuine reuse, not a safe retry — proven distinctly in §4.1). `app.move_org_unit()` relocates a node and cascades path/depth to every descendant in one bounded `UPDATE ... WHERE path @> array[p_id]` (proven against a 3-level-deep tree in §4.1, not just the moved node itself). `app.rename_org_unit()` and `app.set_org_unit_status()` round out CRUD. All three mutating functions (move/rename/status) take `p_expected_version` and raise `org_unit_version_conflict` on a stale value — real optimistic concurrency (Prompt 109 §14's explicit "concurrency" requirement), not merely a passively-incrementing `record_version` column as in `PLT-105`–`108`. Deactivation is blocked while any active child exists (`org_unit_has_active_children`) — nodes are never hard-deleted (Prompt 109 §24; there is no delete function at all), only transitioned to `inactive`, and only once dependents are cleared first.

### 3.4 Scope ancestry helpers, tests, audit, docs (task 4)

`app.org_unit_ancestor_ids()`/`app.org_unit_descendant_ids()` are direct `path`-array reads (§3.2). `app.org_unit_history` records `create`/`move`/`rename`/`status_change` events with before/after fields — the same bounded, capability-scoped append-only audit pattern `PLT-105`–`108` each established, distinct from the generic `audit_log` table `PLT-116` will own later.

## 4. Tests (§27/§28/§29)
