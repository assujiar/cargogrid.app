## 3. Implementation (§20)

### 3.1 Event taxonomy/schema, required context, sensitivity/redaction, retention (task 1)

`supabase/migrations/20260716113048_create_audit_trail.sql`: `app.audit_logs` (`correlation_id`, `tenant_id` nullable for platform-wide events, `actor_auth_user_id`/`actor_label`, `action`, `resource_type`/`resource_id`, `result` — `check`-constrained to `success`/`failure` — `reason`, `before_value`/`after_value` jsonb, `occurred_at`, `legal_hold`/`legal_hold_reason`). `app.redact_audit_payload()` reproduces `scripts/observability/logger.ts`'s `SENSITIVE_KEY_PATTERN` key-name list (`docs/standards/OBSERVABILITY_STANDARDS.md` §4) in SQL — a deliberate, disclosed duplication (a SQL function and a TS module cannot share one source), applied unconditionally inside `app.capture_audit_event()` so a caller that forgets to pre-redact still cannot persist a raw secret/PII value.

### 3.2 Transaction-safe capture/outbox/failure handling, shared instrumentation API (task 2)

`app.capture_audit_event()` is the single real write path — every capability that wants an audit trail entry calls this, never a raw `INSERT` (the established "explicit named-parameter function" discipline since `PLT-105`). Deliberately **not** idempotent/deduplicated — proven in §4.1 that two calls with identical arguments produce two distinct rows, since silently collapsing genuinely repeated actions into one row would under-report, the opposite of an audit trail's purpose. No outbox/async-dispatch mechanism was built — `event_logs` (the table architecturally scoped to that role) is out of this checkpoint (§2); `app.capture_audit_event()` itself is a synchronous, transaction-safe `INSERT` that either commits with its caller's transaction or rolls back with it, which is the correct "transaction-safe" behavior for a compliance trail (a captured event must never survive if the action it describes did not).

### 3.3 Permission-aware query/export with pagination and legal-hold semantics (task 3)

`app.query_audit_logs()`/`app.export_audit_logs()`: keyset-paginated (`docs/architecture/05_*.md` line 125: "keyset mandatory for... `audit_logs`"), authority-gated (reuses `PLT-115`'s `app.is_support_grant_authority()` verbatim: Supreme Admin or the target tenant's own active `tenant_admin`), and — the concrete form of Prompt 116 §26's "privileged access itself audited" — **each call unconditionally captures its own invocation** (`action='query_audit_logs'`/`'export_audit_logs'`) before returning any row, proven in §4.1 to be unbypassable by a caller that discards the result. `legal_hold`/`legal_hold_reason` are settable only through `app.supreme_admin_mutate_audit_log()` (§3.4) — no other mutation path exists.

**A deliberate design departure from this repository's usual RLS-policy pattern, disclosed here:** `app.audit_logs` itself carries **no grant to `authenticated` at all** — the table is fully `service_role`-only, like every `*_history` table's current scope boundary since `PLT-113`. A raw RLS `SELECT` policy (every prior capability's default) cannot make "the read itself is audited" structurally unbypassable — only routing every read through a function that always self-logs can, so that is the mechanism chosen here instead.

### 3.4 Representative platform events, completeness/redaction/privileged-risk tests, docs (task 4)

**RPD-022 made concrete**, since Supreme Admin has no literal Postgres database role to attach an "except Supreme Admin" RLS policy to (Supreme Admin is an app-level `principal_memberships` row, `PLT-108`, and every write in this repository already flows through explicit `service_role`-executed functions): `app.supreme_admin_mutate_audit_log()`/`app.supreme_admin_delete_audit_log()` each check `app.is_supreme_admin()` structurally and, on success, immediately capture their own before/after evidence back into `app.audit_logs` — test #9's exact mechanism, extended to delete as well as update. Scoped to the fields an operator would realistically need to correct/annotate (`reason`, `legal_hold`, `legal_hold_reason`), not a raw arbitrary-column `UPDATE` — RPD-022 describes Supreme Admin's authority, not a mandate that every column be independently exposed through one RPC.

**Representative platform-event integration** (§20 task 4, literally): `PLT-115`'s own kill switch, `app.revoke_support_access()` — an already-`VERIFIED`, genuinely privileged action — is extended via `create or replace function` (same signature, one new `app.capture_audit_event()` call appended) to also leave a canonical `app.audit_logs` entry alongside its existing capability-scoped `app.support_access_events` row. This is the second time this session has used an additive `create or replace function` on an already-shipped function (`PLT-115` itself did this once, to `app.has_active_tenant_membership()`) — proven in §4.1 not to change `revoke_support_access()`'s own return value or existing `support_access_events` behavior, only to add the new audit entry.

### 3.5 A real, empirically-found bug in PLT-114, found and fixed during this checkpoint's authoring

Not part of this checkpoint's own planned scope, but disclosed in full rather than silently worked around: **`app.users_directory` (`PLT-114`) never actually enforced tenant isolation.** Root cause, confirmed empirically: the view is `security_invoker = false` (Postgres's default, needed so the view owner's rights can read the raw, `REVOKE`d-from-`authenticated` `email` column internally) — but per Postgres's own documented semantics, a non-`security_invoker` view's row-level-security posture is the *owner's*, not the *invoker's*, and this repository's migration-applying/view-owning role has `BYPASSRLS` (confirmed directly: `select rolbypassrls from pg_roles where rolname = current_user` → `t`). The practical effect: querying `app.users_directory` returned literally every tenant's users to any authenticated caller, silently ignoring `PLT-113`'s own `users_select_own_tenant` RLS policy on the underlying table entirely.

**How it surfaced:** this checkpoint's own `scripts/db-tests/audit-trail.sql` happens to sort alphabetically *before* `PLT-114`'s `field-record-access.sql` in `scripts/db-tests/run.sh`'s glob-ordered execution, so it now creates real cross-tenant `app.users` rows earlier in the shared `db:test` run than before — enough foreign-tenant data existed for the first time to make the leak observable as a wrong row count (`saw 7` instead of the expected `4`) in an assertion that had *always* been correct in its expectation, just never previously exercised against enough foreign data to catch the underlying defect.

**Fix**, applied via `create or replace view` in this same migration: added an explicit `where app.has_active_tenant_membership(u.tenant_id)` predicate directly inside the view's own query, rather than relying on RLS propagation through the view (proven unreliable). `has_active_tenant_membership()`'s default `p_auth_user_id` parameter reads `auth.uid()`, which reflects the real querying session's JWT claims regardless of the view's owner-rights execution context — `BYPASSRLS` affects RLS *policy* enforcement only, not ordinary SQL predicate evaluation, so this restores correct per-caller tenant scoping without reintroducing the original email-column-privilege problem `PLT-114` solved. Disclosed, bounded limitation: this predicate evaluates `auth.uid()`, which is `null` for a `service_role` session (no JWT claims GUC ever set for it) — `service_role` therefore sees zero rows through this view specifically, a deliberate and harmless scope choice since `service_role` already has, and retains, full unfiltered direct access to `app.users` itself for any use case that needs it. A permanent, self-contained regression test (creating its own extra foreign tenant, not depending on incidental cross-file ordering) was added to `scripts/db-tests/field-record-access.sql` (`PLT-114`'s own test file, since that is the artifact whose contract this proves) so the fix can never regress silently again.

No new issue opened.
