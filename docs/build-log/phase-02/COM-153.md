## 3. Implementation

### 3.1 Scope boundaries (disclosed, task 1)

Recorded in the migration's own header comment, restated here:

1. **Two axes, not one.** `app.quotations.status` (draft/submitted/cancelled) keeps meaning submission lifecycle only. A new `approval_status` (not_required/pending/approved/rejected) is the approval-routing outcome, its own column â€” a property of the exact version, per Prompt 153 Â§24 ("approval always binds one exact quote version").
2. **Thresholds are a bespoke, tenant-wide versioned policy table (`app.quotation_approval_rules`)**, not a second Configuration Engine instantiation â€” the direct precedent is `COM-150`'s own `app.margin_rule_versions` (draft/published/archived, `supersedes_version_id`, exactly one published row per tenant via a partial unique index). Margin/discount/value are covered; customer/service/organizational thresholds are a real, larger feature not built in this bounded slice (mirrors `COM-150`'s own "tenant-wide only" disclosure).
3. **The approval *routing* definition (pattern/steps/approvers) is instantiated, not rebuilt.** It is the already-`VERIFIED` `PLT-123` Approval Engine's own `config_type_code='approval'` object (itself `PLT-121`'s Configuration Engine, reused), scoped `tenant` â€” exactly one per tenant in this bounded slice. Publishing it uses the already-existing generic Platform RPCs (`app.create_config_draft`/`app.set_config_items`/`app.publish_approval_definition`) directly â€” no new SQL, and no dedicated Commercial UI to author routing steps is built here (the same "ships with zero authoring UI" precedent `PLT-123` itself set).
4. **`app.submit_quotation` is where routing happens**, widened via `CREATE OR REPLACE` (never editing `COM-151`/`152`'s own migration files) â€” resolving approval requirement and, if crossed, opening the routed request happens inside the same transaction as the draftâ†’submitted move. No published threshold rule â‡’ `not_required`, auto-approved. A threshold crossed with no published routing definition fails closed (`approval_definition_not_configured`), never silently letting the quotation through.
5. **`app.decide_quotation_approval_step` is the one domain-specific sync wrapper** this migration adds over the Approval Engine â€” the engine itself is entity-agnostic and cannot write back to `app.quotations`. Delegation/escalation need no such wrapper (neither changes a request's final outcome) â€” the API layer calls the engine's own `app.create_approval_delegation`/`app.revoke_approval_delegation`/`app.escalate_approval_step` directly, unwrapped.
6. **"Request revision" is not a fourth decision verb.** The Approval Engine's own `approval_decisions.decision` CHECK is `approved`/`rejected` only, not widened here for one call site. An approver requesting revision decides `rejected` with a reason; `app.create_quotation_revision` (`COM-152`, already built) is the real "fresh governed approval path" mechanism â€” the new revision starts `approval_status='not_required'` by column default (never inheriting a rejected verdict).
7. **Legacy submitted quotations are never retroactively approved.** Every pre-existing `app.quotations` row defaults to `approval_status='not_required'` purely because it predates this capability â€” not backfilled to `'approved'`. Whether a pre-existing submitted quotation should be acceptance-eligible is `COM-154`'s own reconciliation policy to define.

### 3.2 Schema â€” `supabase/migrations/20260724270000_create_commercial_quotation_approval.sql`

New table `app.quotation_approval_rules` (mirrors `app.margin_rule_versions`' own versioning shape exactly): `min_margin_pct`/`max_discount_pct`/`min_value_amount` (each optional, at least one required by CHECK), draft/published/archived, `supersedes_version_id`, partial-unique-published-per-tenant index. `app.create_quotation_approval_rule_version` (`COM:Create`-gated) / `app.publish_quotation_approval_rule_version` (`COM:Approve`-gated, archives the prior published rule) mirror `app.create_margin_rule_version`/`app.publish_margin_rule_version` line for line.

Widens `app.quotations` (never edits `COM-151`/`152`'s own files): adds `approval_status` (CHECK'd `not_required`/`pending`/`approved`/`rejected`), `approval_request_id` (FK to `PLT-123`'s own `app.approval_requests`), `approval_rule_version_id` (FK to the new rules table, the exact rule snapshot evaluated), `approval_required_reasons` (`text[]`, reason codes only).

`app.evaluate_quotation_approval_requirement(quotation_id)` â€” the one deterministic, explainable "does this need approval" decision. Reads the tenant's published rule (none â‡’ not required); margin check uses `min(quotation_lines.margin_pct_snapshot)` across sourced lines; discount check computes the header's own effective `discount_amount/subtotal_amount * 100`; value check compares `total_amount` directly. Returns `(required, reasons, rule_version_id)` â€” reason codes only, never a dollar figure, mirroring `app.get_quotation_submission_readiness`'s own (`COM-151`) masking-independent-safety discipline.

### 3.3 Widened / new functions

`CREATE OR REPLACE FUNCTION app.submit_quotation` (identical signature) â€” after the existing readiness gate passes, calls the evaluator; not required â‡’ single `UPDATE` sets `status='submitted'`, `approval_status='approved'` in one transaction; required â‡’ looks up the tenant's published `config_type_code='approval'` config version, fails closed (`approval_definition_not_configured`) if none, else calls `PLT-123`'s own `app.request_approval` (idempotency key = the quotation's own id) and sets `approval_status='pending'`/`approval_request_id` in the same `UPDATE`.

`app.decide_quotation_approval_step(request_step_id, decision, actor, label, reason)` (new) â€” validates the bound request's `entity_type='quotation'` (`not_a_quotation_approval` otherwise), calls the unmodified `app.decide_approval_step`, then re-reads the request: `approved`/`rejected` â‡’ syncs `app.quotations.approval_status`; still mid-routing â‡’ no sync (stays `pending`).

`CREATE OR REPLACE VIEW app.quotations_directory` â€” appends the four new columns at the end (reason codes/references only, visible to any record-scoped viewer regardless of `COM:View cost`/`View selling price`, the same `get_quotation_submission_readiness` precedent).
