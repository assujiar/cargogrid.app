## 3. Implementation

### 3.1 Scope boundaries (disclosed, task 1)

Recorded in the migration's own header comment, restated here:

1. **"Send"/"resend" are one function, not two.** Calling `app.send_quotation_for_acceptance` again while an active token exists revokes it (`superseded_by_resend`) and mints a fresh one — database-enforced via a partial unique index (`quotation_acceptance_tokens_quotation_active_unique`), never two live tokens. Deliberately not idempotent on a literal shared secret — `PLT-129`'s own precedent already established that a raw secret is returned exactly once and never replayed.
2. **Token expiry is bound to the quotation's own `validity_to`**, never a separately invented policy — a token can never outlive the quote it grants a decision over. Lazily flipped `active`→`expired` on read/decide, the exact mechanism `app.authenticate_api_key` already established.
3. **No scheduled expiry/reminder job.** `PLT-132`'s Background Job Framework has no live worker anywhere in this repository — lazy expiry-on-read is the real, sufficient mechanism, the same choice `PLT-129` already made.
4. **No email/SMTP delivery is dispatched.** No real notification provider is wired anywhere in this repository. `app.send_quotation_for_acceptance` mints and returns the real, secure, hashed token/link; the seller relays it through their own channel until a real provider capability exists.
5. **No rate limiting / IP throttling on the public decision route.** No live deployment or edge/gateway infrastructure exists yet — replay protection instead comes from the token's own atomic single-use consumption, the same "stale record fails safely" pattern `app.decide_approval_step` already established.
6. **Customer identity is bearer-token possession, not a verified account.** No customer login/account exists yet (`COM-155`). `decided_by_name`/`decided_by_title`/`decided_by_email` are the customer's own typed attestation; `ip_address`/`user_agent` are best-effort, supplied by the calling route where lawfully available, never fabricated.

### 3.2 Schema — `supabase/migrations/20260724280000_create_commercial_quotation_customer_acceptance.sql`

New `app.quotation_acceptance_tokens` (mirrors `app.api_keys`: `token_hash` one-way sha256, raw value returned exactly once, never stored; status `active`/`consumed`/`revoked`/`expired`; at most one active row per quotation via a partial unique index). New `app.quotation_customer_decisions` (append-only, `unique(quotation_id)` — one final decision per exact quotation version; `decision` CHECK `accepted`/`rejected`; mandatory `reason` when rejected). Widens `app.quotations` with `customer_decision`/`customer_decision_at` (a third, independent axis alongside `status`, COM-151, and `approval_status`, COM-153).

`SECURITY DEFINER` functions calling `pgcrypto` (`gen_random_bytes`/`digest`) use `search_path = app, public, pg_temp` rather than the usual bare `app, pg_temp` — `pgcrypto` installs into `public`; every table/function reference in this migration's own bodies is already fully schema-qualified, so this is a real, narrower-than-`PLT-129`'s-own-precedent restriction (`app.create_api_key` sets no `search_path` override at all), not a regression of the `ERR-2026-004` discipline. Disclosed explicitly in the migration header.

### 3.3 Functions

`app.send_quotation_for_acceptance(quotation_id, recipient_contact_id, channel, actor, label)` — `COM:Edit` + record access gated (mirrors `app.submit_quotation`'s own authority check via a new `app.check_quotation_send_authority` helper); requires `status='submitted'`, `approval_status` in `('approved','not_required')`, `is_current=true`, no existing customer decision, `validity_to` not yet passed. `app.revoke_quotation_acceptance_token(token_id, actor, label, reason)` — explicit seller-initiated revoke, mandatory reason. `app.get_quotation_for_customer_decision(raw_token)` — the public, unauthenticated read (no `security definer`, no `auth.users` session exists for this caller, mirrors `app.authenticate_api_key`'s own posture); never raises for an expired/revoked/consumed token, only for a genuinely unknown one — the caller renders `token_status` as a real UI state. `app.record_quotation_customer_decision(raw_token, decision, decided_by_name, decided_by_title, decided_by_email, reason, ip_address, user_agent)` — the one customer-facing decision write; atomic single-use token consumption (`update ... where status = 'active'`) is the real replay guard; syncs `app.quotations.customer_decision`/`customer_decision_at`; audits with `actor_auth_user_id = null` (no `auth.users` identity exists for a customer) and `actor_label` = the customer's own typed name/email.

### 3.4 Service layer and UI (task 4)

`server/contracts/quotation/quotation.ts` gained the two new fields; new `server/contracts/quotation/quotation-acceptance.ts` (token/customer-view schemas, send/revoke/decision input schemas — `token_hash` never appears anywhere in this contract). `server/queries/quotation-acceptance.ts` — `listQuotationAcceptanceTokens` (authenticated, RLS-scoped, `token_hash` never selected) and `getQuotationForCustomerDecision` (public, called with a service-role client). `server/mutations/quotation-acceptance.ts` — `sendQuotationForAcceptance`/`revokeQuotationAcceptanceToken`/`recordQuotationCustomerDecision`.

Seller-side UI: `CustomerAcceptancePanel` (send history table, revoke action) + `SendAcceptanceForm` (recipient/channel picker; displays the raw link exactly once immediately after a successful send — never persisted, never retrievable again) added to the Quotation Detail page. ## 8. Errors / recovery / risks / issues

Three real defects found and fixed during authoring, all caught before this checkpoint declared itself green: (1) two `plpgsql` functions whose `RETURNS TABLE` declared an output column literally named `quotation_id` produced "column reference is ambiguous" against bare `quotation_id` references inside their own bodies (a genuine Postgres/plpgsql name-shadowing bug, not a logic error) — fixed by table-aliasing every such reference (`d.quotation_id`, `t.quotation_id`); (2) `gen_random_bytes`/`digest` (pgcrypto, installed in `public`) were unreachable from two `SECURITY DEFINER` functions whose `search_path` excluded `public` — fixed by widening those two functions' `search_path` to `app, public, pg_temp` (disclosed in the migration header as a real, narrower-than-the-existing-`PLT-129`-precedent restriction, not a regression); (3) `app.record_quotation_customer_decision`'s expiry check only handled the case where the token's `expires_at` had *just* passed (lazily flipping status in the same call) but not the case where a prior read (`app.get_quotation_for_customer_decision`) had *already* flipped it to `expired` in the database — a genuine `db:test` failure (`token_not_active` returned instead of `token_expired`) — fixed by adding an explicit `if v_token.status = 'expired'` branch after the lazy-flip check, covering both paths uniformly. Zero Critical/High-severity issue remains open. Zero regression to any prior capability (full `db:test`/`test`/`next build` re-run, all green).
