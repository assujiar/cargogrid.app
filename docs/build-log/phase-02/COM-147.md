## 3. Implementation

### 3.1 Scope boundaries (disclosed, task 1)

Prompt 147 §13 says "account/prospect links" — the canonical customer Account entity does not exist yet (`COM-155`, Customer and Account Conversion, has not run). `app.opportunities` therefore references `app.prospects` (`prospect_id`, not null) plus a forward-compatible, currently-unpopulated `account_ref` column `COM-155` is expected to backfill once accounts exist — never a second competing customer identity.

Service/cargo/lane/SLA "requirements" (§13/§25) are captured as a bounded jsonb snapshot (`requirements`) rather than a formal Operations service-catalogue integration, since no canonical service-type/cargo-type/lane master exists yet (those belong to Phase 3 Operations, which has not started). `app.get_opportunity_costing_readiness` checks the presence of well-known keys within this snapshot, not a validated master reference — matching the same disclosed-boundary pattern COM-144's `billing_address` and COM-145's deferred site/address entity already established.

The opportunity stage machine is domain-native (its own `stage` column plus a dedicated transition function), matching every prior Commercial capability's own precedent, rather than integrating with the generic Configurable Status Engine (`PLT-124`) — no earlier Commercial capability established that integration, and doing so now would be a materially larger architecture decision than this bounded task warrants. Named as a disclosed boundary for a future ADR, not a silent omission.

### 3.2 Schema — `supabase/migrations/20260723210000_create_commercial_opportunity_management.sql`

`app.opportunities` (canonical opportunity: prospect reference, stage/probability, bounded requirements snapshot, value/currency, next action, close reason, clone lineage). `app.opportunity_stage_history` (explicit, queryable stage-transition history alongside `app.audit_logs`, matching Prompt 147 §13's own "stage history" naming). `app.has_view_selling_price` (a new field-masking gate for the seeded, protected `COM:View selling price` permission — never previously consumed by any Commercial capability). `app.opportunities_directory` (a field-masked projection of `app.opportunities`).

### 3.3 Field masking and the security_invoker constraint found during authoring (task 4)

This capability reuses PLT-114's established field-masking pattern (`app.has_view_personal_data`/`app.users_directory`). The first draft of `app.opportunities_directory` used `security_invoker = true` (matching COM-146's own stricter row-visibility design). This failed with a real `permission denied for table opportunities` the first time an actor without `COM:View selling price` queried it: Postgres checks column privileges against the *invoker* for every column a view's defining query references, even one only ever read inside a `CASE` branch that resolves to `null` at runtime — an invoker-mode view cannot read a column-REVOKEd field at all, masked or not, since the privilege check happens at parse/plan time, not per-row at execution time.

Fixed by matching PLT-114's own proven technique exactly: `security_invoker` defaults to `false`, so the view runs as its owner and can read the REVOKEd `value_amount`/`value_currency`/`probability` columns internally while still deciding, per row, what to reveal. Unlike `app.users_directory` (which has no row-level filter of its own), `app.opportunities_directory` adds its own explicit `app.can_access_record(...)` `WHERE` clause — necessary because `security_invoker = false` also means the base table's RLS policy does not apply to the view's own read. This is a deliberate, disclosed strengthening over the bare PLT-114 pattern, not a critique of it (out of this task's scope to alter).

The database guarantee matches PLT-114's own proven technique exactly: `revoke select on app.opportunities from authenticated` followed by `grant select (<explicit column list omitting value_amount/value_currency/probability>) on app.opportunities to authenticated` — a bare column-level `REVOKE` cannot carve an exception out of a broader table-level `GRANT`.

### 3.4 Opportunity lifecycle (tasks 2/4)

`app.create_opportunity` (always creates a new row — no external idempotency key exists, unlike `app.capture_lead`; starts at `stage=qualifying`, `probability=10`). `app.update_opportunity` (blocked once `stage` is `won`/`lost`; setting a new/changed `value_amount`/`value_currency` additionally requires `COM:View selling price` — an actor cannot set a sensitive figure they are not entitled to see). `app.transition_opportunity_stage` (the one stage-transition entry point; closing to `won`/`lost` requires `COM:Approve` and a non-empty reason — "managers move/close by policy," Prompt 147 §26 — any other move requires `COM:Edit`; probability defaults deterministically per stage — 10/30/60/100/0 — unless the caller overrides it). `app.clone_opportunity` (always creates a new row, never idempotent — cloning the same source twice is a legitimate distinct action; resets stage/probability to the create defaults, copies requirements/value as a mutable, explicitly reviewable starting point). `app.get_opportunity_costing_readiness` (a fixed, deterministic, non-tenant-configurable data-completeness check, mirroring `COM-144`'s `app.get_prospect_conversion_readiness`).

### 3.5 Extending the shared polymorphic resolver (disclosed extension, not required by the prompt)

`app.resolve_commercial_record_ref` (`COM-145`) is extended with an `'opportunity'` branch, and the `contact_links`/`activities` polymorphic `related_type` `CHECK` constraints are extended (additive `ALTER TABLE ... DROP/ADD CONSTRAINT`, never editing `COM-145`'s applied migration) to accept it. This was not explicitly requested by Prompt 147, but is exactly the extension point that resolver's own documentation names as the correct way to grow it — a small, low-risk, clearly-scoped addition that lets sellers use the same Contact/Activity Timeline UI against opportunities, avoiding a parallel "opportunity-specific" contact/activity mechanism later.

### 3.6 Service layer and UI (task 4)

`server/contracts/opportunity/opportunity.ts` (Zod), `server/queries/opportunity.ts` (reads through `app.opportunities_directory`, never the base table directly, so masking is never accidentally bypassed), `server/mutations/opportunity.ts`. New route `app/(tenant)/[tenantSlug]/commercial/opportunities/` (server-paginated table + create form) and `opportunities/[opportunityId]/` (detail: stage history, costing readiness, requirements/value/stage/clone actions, with the value-edit form hidden entirely — not just its current value — when `valueMasked` is true). The Contact/Activity Timeline is embedded on the Opportunity Detail page exactly as on Lead/Prospect Detail, now that `'opportunity'` is a valid `related_type`.

Disclosed UI scope boundary: a server-paginated table, not a drag/drop Kanban board — a full board UI is deferred; stage moves remain fully real, tested mutations, exercised from the Opportunity Detail page's own stage-select control.

### 3.7 A second real defect found and fixed: `RelatedType`/route-segment fallout from extending the resolver

Extending `contact_links`/`activities` to accept `'opportunity'` required two TypeScript-layer follow-ups that would otherwise have silently under-served the new capability: (1) `server/contracts/contact/contact.ts`'s `RELATED_TYPES`/`RelatedTypeSchema` only listed `lead`/`prospect` — extended to include `opportunity`. (2) `app/(tenant)/[tenantSlug]/commercial/_shared/activity-actions.ts`'s four Server Actions each hardcoded `relatedType === "lead" ? "leads" : "prospects"` when computing the `revalidatePath` target — this ternary would have silently revalidated `/prospects/{id}` instead of `/opportunities/{id}` for any activity logged against an opportunity, a real (if minor) staleness bug caught before it shipped. Fixed with a small `relatedTypeRouteSegment()` switch covering all three related types.
