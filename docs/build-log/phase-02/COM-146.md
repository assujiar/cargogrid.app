## 3. Implementation

### 3.1 Scope boundary (disclosed, task 1)

Prompt 146 §13 asks the plan/target/forecast layer to reference "canonical leads/prospects/opportunities." The Opportunity entity does not exist yet — `00_COMMERCIAL_WBS.md` §3 row 5 places Opportunity Management at Prompt 147, *after* this task, depending on `143..146`. This checkpoint scopes its pipeline and metric definitions to Lead and Prospect only — the only canonical revenue-track records that exist at this checkpoint. Every new polymorphic surface (`app.commercial_pipeline_view`'s two `UNION ALL` branches, `app.pipeline_outcomes`'s `related_type` check) is additive and designed to extend to `'opportunity'` once `147` lands, not a parallel scheme to reconcile later.

Prompt 146 §13 also names a fifth entity noun, "category," alongside plan/target/forecast/win-loss reason, without otherwise defining it. This checkpoint interprets it as a tenant-managed reference list for grouping sales targets by funnel stage for reporting (`app.pipeline_categories`) — a bounded, disclosed interpretation, not a hidden guess, and explicitly not the deferred Configuration Engine rule evaluator (`ADR-CAND-ARCH-014`/`015`, still open per `PLT-121` §25).

No canonical selling-value/currency field exists on any Commercial record yet (Quotation Builder, `COM-151`, is the first capability to introduce one) — every sales-target metric in this checkpoint is therefore count-based (`leads_captured`, `leads_qualified`, `prospects_created`, `prospects_disqualified`), not revenue-based. This is a real, disclosed scope boundary, not an oversight.

### 3.2 Schema — `supabase/migrations/20260723180000_create_commercial_sales_pipeline.sql`

`app.pipeline_categories` (tenant-managed reference list). `app.sales_plans` (versioned, effective-dated, organization-unit-scoped; `draft`→`published`→`archived`; a published plan is never edited in place — `app.publish_sales_plan`'s `p_supersedes_plan_id` parameter atomically archives the prior published plan and links the new one, the "versioned plan" mechanism). `app.sales_targets` (one of the four count-based metrics above, per-seller when `owner_user_id` is set, org-unit-wide otherwise; only editable while the parent plan is `draft`). `app.forecast_snapshots` (append-only, point-in-time capture of a target's reconciled actual plus an optional reasoned manual override). `app.win_loss_reasons` (tenant-managed reference list, `won`/`lost`-scoped). `app.pipeline_outcomes` (additive win/loss categorization event against a lead or prospect via `app.resolve_commercial_record_ref`, `COM-145` — never mutates the source row; a new call supersedes the prior current row via an `is_current`/`superseded_by_id` chain, preserving the full override history).

### 3.3 Governed pipeline read-model (task 3, security design point 1)

`app.commercial_pipeline_view` unions `app.leads` and `app.prospects` into one `(stage, org_unit_id, owner_user_id)` shape, declared `security_invoker = true`. Every query against it runs with the calling role's own privileges, so the existing `leads_select_scoped`/prospect RLS policies filter rows exactly as they would for a direct drill-down query — "aggregates respect the same row permissions as drill-down records" (Prompt 146 §16) holds **by construction**, not by re-implementing the same predicate a second time. `app.get_pipeline_summary` is deliberately `SECURITY INVOKER` (no elevation) for the same reason; its `p_org_unit_id` parameter is a business drill-down filter (descendant-scoped via the new `app.pipeline_scope_org_unit_ids` wrapper), never itself the security boundary.

`app.pipeline_scope_org_unit_ids` wraps `app.org_unit_descendant_ids` (`PLT-109`, service_role-only) the same narrow-`SECURITY DEFINER`-wrapper pattern `COM-143`'s `app.lead_record_scope_org_unit_ids` already established for the sibling `app.org_unit_ancestor_ids` — deliberately a *different* function for a *different* query direction (descendant-plus-self, "records at or below this org unit," vs. ancestor-plus-self, "can an actor positioned above see this record"), not a duplicate.

### 3.4 Forecast reconciliation (task 3, security design point 2)

`app.compute_sales_metric_count` is the one reconciliation implementation shared by `app.get_sales_target_actual` (live drill-down) and `app.capture_forecast_snapshot` (point-in-time capture). Unlike the read-model above, it unavoidably runs `SECURITY DEFINER` (it must write audit events and span an org-unit-descendant scope beyond RLS's per-row row-owner check), so it explicitly re-states the identical `app.can_access_record(...)` filter per candidate row rather than trusting RLS — the one place in this migration where the predicate is deliberately restated, and only ever in this one function, reusing `app.can_access_record` and `app.lead_record_scope_org_unit_ids` directly rather than re-deriving new logic.

### 3.5 Sales plan lifecycle and win/loss categorization (tasks 2/4)

`app.create_sales_plan`/`app.publish_sales_plan` (overlap-checked against other published plans in the same organization scope)/`app.archive_sales_plan`. `app.create_sales_target`/`app.update_sales_target` (draft-plan-only). `app.capture_forecast_snapshot` (an override value requires a non-empty override reason, enforced at both the `CHECK` constraint and application layers). `app.create_pipeline_category`/`app.update_pipeline_category`, `app.create_win_loss_reason`/`app.update_win_loss_reason`, `app.record_pipeline_outcome` (validates the reason's own `outcome` matches the requested outcome, structurally denies a cross-tenant reason).

### 3.6 Real defect found and fixed during authoring: `app.can_access_record`'s NULL-owner gap

`app.sales_plans`/`app.sales_targets` are the first tables in this repository whose `owner_user_id` can legitimately be `NULL` (an org-unit-wide plan/target has no single owning user). `app.can_access_record`'s original body (`PLT-114`) compared `p_owner_user_id = p_auth_user_id` directly — when `p_owner_user_id` is `NULL` this yields SQL `NULL`, not `false`, and `false OR null` is `NULL` rather than `false`. Every existing caller's `if not app.can_access_record(...)` guard silently *skipped* its denial branch on a `NULL` result (`if not null` is `NULL`, which a plpgsql `IF` treats as not-true, i.e. it does not raise) — meaning a `NULL` owner comparison could silently **grant** access instead of denying it. No caller before this checkpoint ever passed a `NULL` owner_user_id, so this was latent, never exercised, and never previously caught by any test; it surfaced immediately as a real, reproducible `db:test` failure (a sibling-team outsider was not denied access to a team-scoped target) the first time this checkpoint's own test suite exercised the null-owner path. Fixed via `CREATE OR REPLACE FUNCTION app.can_access_record` in this migration (the applied `PLT-114` migration file itself is never edited) — the owner comparison is now deterministically boolean (`p_owner_user_id is not null and p_owner_user_id = p_auth_user_id`), and the full `OR` expression is wrapped in a defense-in-depth `coalesce(..., false)`.
