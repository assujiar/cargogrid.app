## 3. Implementation

### 3.1 Schema (task 1) — `supabase/migrations/20260723090000_create_commercial_lead_management.sql`

`app.leads` — the first business-domain table in this repository: tenant-scoped, source taxonomy (`manual`/`import`/`api`/`referral`/`campaign`/`integration`), status machine (`new`→`contacted`/`qualified`/`disqualified`/`merged`/`converted`), computed `normalized_email`/`normalized_phone`/`duplicate_fingerprint` (trigger-maintained, never entered directly), `score`/`score_explanation`/`score_version`, `owner_user_id`/`org_unit_id` for record-scope, optimistic `record_version`, and `merged_into_id` lineage (never deleted, mirroring `app.master_records`' own pattern). One additive `app.permissions` seed row (`Assign`/`COM`) — `COM` never previously needed that action.

### 3.2 Duplicate detection and idempotent capture (tasks 1/2)

`app.normalize_lead_email`/`app.normalize_lead_phone`/`app.compute_lead_duplicate_fingerprint` (pure, immutable) feed a `before insert or update` trigger. `app.capture_lead` is idempotent on `(tenant_id, source, external_reference)` for import/api/integration sources (including under a concurrent-insert race, handled via `unique_violation` re-select); manual/referral/campaign captures always succeed and never silently block on a duplicate — `app.find_duplicate_leads` surfaces candidates separately, tenant-scoped and fail-closed on missing membership (never a probe vector).

### 3.3 Scoring (task 3)

`app.compute_lead_score` — deterministic, rule-based, capped at 100, versioned (`score_version=1`). No AI/ML call anywhere (Phase 9's own exclusive ownership of learned scoring, `01_MODULE_DEPENDENCY_MAP.md` line 147). Informs workflow only — no gate anywhere blocks a low-scored lead from being contacted/qualified (Prompt 143 §24).

### 3.4 Record-scope (task 3, §16) — the first real composition of two proven-but-unexercised Platform Core primitives

`app.can_access_record` (`PLT-114`) and `app.org_unit_ancestor_ids` (`PLT-109`) were both real and tested since Platform Core but had **no business-domain table to exercise them against** until this checkpoint. A lead's own org unit plus every ancestor (branch/department/company) is passed as the "shared scope" set — whoever sits at the tenant's root org unit already sees every lead beneath it, satisfying the full "own/team/department/branch/company" scope ladder with zero new mechanism. A narrow `SECURITY DEFINER` wrapper, `app.lead_record_scope_org_unit_ids`, avoids widening `org_unit_ancestor_ids`' own `service_role`-only grant — the same bounded-wrapper pattern `PLT-114`'s `has_view_personal_data` already established (found by a real `permission denied for function org_unit_ancestor_ids` failure during db-test authoring, not anticipated in advance — see §8).

### 3.5 Assignment, qualify/disqualify, merge (tasks 2/3)

`app.assign_lead` — optimistic-concurrency-checked (`p_expected_version`), gated on both `COM:Assign` entitlement and current record-scope. `app.transition_lead_status` is the shared core `qualify_lead`/`disqualify_lead` both wrap; `disqualify_lead` validates a non-empty reason before ever calling it, with the DB `CHECK` constraint as a second, independent line of defense. `app.merge_leads` marks the duplicate `status='merged'` (never deleted), structurally denies a cross-tenant merge, and requires record-scope access to *both* leads.

## 8. Errors / recovery / risks / issues

Three real defects found and fixed during this checkpoint's own authoring (not anticipated in advance, disclosed per this repository's standing discipline):

1. **Org-unit type mismatch**: `app.create_org_unit` has no `'team'` unit type (only `company`/`branch`/`department`/`business_unit`) — the db-test's own fixture setup was corrected to use `'department'` for the two sales-team nodes.
2. **`app.org_unit_ancestor_ids` grant gap**: granted `service_role`-only since `PLT-109` (never previously called from an `authenticated`-evaluated RLS policy). Fixed by adding a narrow `SECURITY DEFINER` wrapper (`app.lead_record_scope_org_unit_ids`) rather than widening `PLT-109`'s own original grant decision — found via a real `permission denied for function org_unit_ancestor_ids` failure, not anticipated.
3. **`SupabaseClient.rpc()`/`.from()` type mismatch**: a hand-rolled `RpcClient`/`TableClient` interface (the pattern `server/mutations|queries/master-data.ts` already used) does not structurally accept a real `SupabaseClient`, since `.rpc()`/`.from()` return thenable `PostgrestFilterBuilder`s, not strict `Promise`s — found via a real `tsc` failure once this checkpoint's own UI code passed a real client (the first time any capability in this repository has). Fixed by typing `LeadMutationRpcClient`/`LeadQueryRpcClient` as `Pick<SupabaseClient, "rpc">` and `listLeads`/`getLeadById` as `Pick<SupabaseClient, "from">`, matching `server/queries/portal-users.ts`'s own already-correct convention. `master-data.ts`'s own latent version of this same class was left unchanged (out of this task's scope — it has no real caller yet either) but is worth a future capability's own attention if it ever gains one.

Zero Critical/High-severity issue. Zero regression to any Platform Core capability (full `db:test`/`test`/`next build` re-run, all green).
