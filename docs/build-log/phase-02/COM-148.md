## 3. Implementation

### 3.1 Scope boundaries (disclosed, task 1)

This is a bounded Commercial cost-request shell over one canonical opportunity — explicitly NOT full Procurement RFQ/sourcing/PO/vendor onboarding (Prompt 148 §24; that remains Phase 6 Procurement's own scope). Three boundaries disclosed up front: (1) "configuration versions" (§24) — no Commercial capability has integrated with the Configuration Engine (`PLT-121`) yet; the only version this migration pins is the source opportunity's own `record_version`. (2) Vendor identity is a free-text `vendor_ref`, never a reference into a real vendor master — Procurement Phase 6 owns vendor onboarding, and `ADR-0015` already reserves rate/cost-lookup ownership for `COM-149` (the very next capability); this migration builds the request/response shell `COM-149` will feed live rate data into, it does not perform automated lookup. (3) SLA is a plain `due_at` field with a read-time `is_overdue`/`is_expired` computation — no background job/notification dispatch (`PLT-127`/`132`) is wired, matching the same generic-engine-integration deferral every prior Commercial capability has disclosed.

### 3.2 Schema — `supabase/migrations/20260724090000_create_commercial_costing_request.sql`

`app.costing_requests` (one per `(tenant, opportunity, source_opportunity_version)` — idempotent creation; `requirements_snapshot` is a point-in-time copy of the opportunity's own requirements, never a live re-typed reference). `app.costing_request_components` (the requested line items). `app.costing_responses` (a componentized cost response from an internal or vendor source; `total_amount` is always server-computed as the sum of its own components, never a client-supplied figure). `app.costing_response_components` (one priced amount per requested component per response).

### 3.3 A new Commercial-scoped permission: `('View cost', 'COM', ...)`

The seeded permission catalogue (`PLT-111`) already has a `'View cost'` action, but only for the `FIN` and `PRC` modules — permissions are scoped per `resource_module_code`, so those existing rows grant nothing for module `'COM'`. This was found the hard way during test authoring: a fixture granting `'View cost'` to a Commercial role silently selected zero permission rows (the `array(select id from app.permissions where resource_module_code = 'COM' and action = 'View cost')` subquery simply returned nothing, no error), so `app.has_view_cost` returned `false` for an actor the test intended to have the permission. Fixed by inserting a new, Commercial-scoped `('View cost', 'COM', 'sensitive', true)` catalogue row — the same technique `COM-143` used to add `'Assign'` for `'COM'` after the fact, a new row, never an edit to any existing one.

### 3.4 Field masking (task 4, reusing the COM-147 lesson directly)

`app.has_view_cost` mirrors `COM-147`'s `app.has_view_selling_price` exactly. `app.costing_responses_directory` is `security_invoker = false` **from the start** — the previous checkpoint's own build log (`COM-147.md` §3.3) already proved that an invoker-mode view cannot read a column-REVOKEd field at all, even inside a masking `CASE`, since Postgres checks column privileges against the invoker for every referenced column at parse time, not the runtime branch outcome. No need to rediscover that here. The view adds its own explicit `app.can_access_record(...)` row filter (the same deliberate strengthening over `PLT-114`'s bare `app.users_directory` that `COM-147` introduced). Component-level cost detail (`app.costing_response_components`) has no "masked but visible" state at all — an actor either holds `COM:View cost` and sees every row, or is denied the entire table via RLS; this is a disclosed, simpler design than mirroring the top-level masking pattern a second time for a table whose only content is cost figures.

### 3.5 Lifecycle functions (tasks 2/3)

`app.request_costing` (idempotent on `(tenant, opportunity, source_opportunity_version)`; reuses `app.get_opportunity_costing_readiness` — `COM-147` — directly rather than re-deriving a second completeness check, and inherits that function's own access check for free). `app.assign_costing_request` (optimistic concurrency). `app.submit_costing_response` (requires both `COM:Edit` and `COM:View cost` — an actor cannot enter a cost figure they are not entitled to see back, mirroring `COM-147`'s value-write gate; rejects a component id that does not belong to the request; moves the parent request to `responded` on its first response). `app.revise_costing_request` (creates a new request pinned to the opportunity's *current* version/requirements, marks the source `superseded` — never edited in place, so linked response evidence remains reachable; raises `no_new_version` if the opportunity has not actually changed, since the unique constraint would otherwise silently return the unchanged source row, the wrong outcome for an explicit "revise" action). `app.cancel_costing_request` (mandatory reason, terminal-state guarded).

### 3.6 Service layer and UI (task 4)

`server/contracts/costing/costing.ts` (Zod), `server/queries/costing.ts` (reads through `app.costing_responses_directory`, never the base table directly), `server/mutations/costing.ts`. The Opportunity Detail page (`COM-147`) gained a Costing Requests section (list + a bounded "Request costing" form supporting up to 3 initial line items — disclosed UI scope boundary, a fully dynamic add/remove component list is deferred) and a new top-level route `app/(tenant)/[tenantSlug]/commercial/costing-requests/[requestId]/` (components, field-masked responses with their own itemized components when unmasked, assign/respond/revise/cancel actions). Disclosed UI scope boundary: no standalone costing-request queue/list view across all opportunities in this bounded slice — requests are reached via their owning opportunity's detail page.

### 3.7 A second class of real defect found and fixed: cross-file audit-log-count fragility, twice over

This capability's own setup legitimately needs a qualified lead converted to a prospect (to attach a realistic opportunity), and its own scenarios legitimately create three opportunities. Since `commercial-costing-request.sql` sorts alphabetically *before* both `commercial-lead-management.sql` and `commercial-opportunity-management.sql`, its real (not unnecessary) `qualify_lead`/`create_opportunity` calls broke those two files' own bare, unscoped `count(*) from app.audit_logs` "exactly N" assertions — the same class of fragility `COM-145` first found and worked around by removing an *unnecessary* call. This time the extra calls are legitimate, so the durable fix is different: **both fragile assertions were corrected to scope by their own file's tenant_id** (`... and tenant_id = v_tenant1`), rather than relying on a bare global count across the entire shared disposable database every db-test file runs against. This closes the fragility permanently for both files rather than deferring it to whichever future capability's fixtures happen to sort alphabetically earlier next.
