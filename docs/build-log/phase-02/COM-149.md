## 3. Implementation

### 3.1 Scope boundaries (disclosed, per ADR-0015 and Prompt 149 §24)

`ADR-0015` is the controlling document: Commercial (this task) adds **exactly one** new migration on top of `PLT-120`'s master-data foundation — one child table (`app.vendor_rate_versions`, keyed by `master_record_id references app.master_records(id)`, `master_type_code` constrained to `'vendor_rate'`) carrying the structured rate-version fields the generic `attributes` jsonb cannot index efficiently, plus read views — **never** a second, independently-writable rate master. Write authority for the interim period (Phase 2 through Phase 5, before Procurement ships) reuses `app.is_support_grant_authority` unchanged — no parallel authority model was invented. Three further boundaries disclosed up front: (1) `service_type`/`mode`/`equipment_type` stay free text (mirrors `COM-147`/`148`'s own disclosed boundary — no Operations service-catalogue master exists yet); (2) `app.search_vendor_rates` is a bounded, single-lane comparison lookup (≤200 rows, no cursor pagination) — not a paginated full master-data browse (`app.search_master_records`, `PLT-120`, already serves that broader purpose); (3) selecting a rate for one costing request (`app.select_vendor_rate`) always snapshots the exact rate detail at selection time into `app.rate_selections` — a later edit or supersede of the source `vendor_rate_versions` row never silently reprices an already-issued quote, the same no-reentry snapshot rule every prior Commercial capability follows.

### 3.2 Schema — `supabase/migrations/20260724150000_create_commercial_rate_cost_lookup.sql`

`app.vendor_rate_versions` (the one child detail table `ADR-0015` mandates; lane/mode/equipment/weight/volume breaks, currency/base/minimum amounts, surcharge components jsonb, `approval_status` state machine `pending_approval → approved/rejected`, `approved → withdrawn`, and supersede-on-revise → `superseded`). `app.rate_selections` (a point-in-time snapshot of the rate detail selected for one costing request — never a live reference; `is_adhoc=true` is a governed escape hatch that always requires a non-empty `override_reason`).

### 3.3 The `app.v_active_vendor_rates` view is enhanced, not duplicated

`PLT-120` already seeded `app.v_active_vendor_rates` as `select * from app.master_records where master_type_code='vendor_rate' and canonical_status='active'` — a master-identity-only filter, not yet validity/approval-aware (its own build log disclosed exactly this: "empty until Phase 6 actually writes real vendor_rate records"). This checkpoint drops and recreates that same view name (its column shape changes entirely — a `CREATE OR REPLACE VIEW` cannot do that — and nothing has consumed the seeded definition yet, so a clean drop+recreate is safe) so it now joins `app.master_records` with `app.vendor_rate_versions` and filters to `approval_status='approved'` and currently within its effective window, composed on a new `app.vendor_rate_versions_directory` view so the masking logic exists in exactly one place.

### 3.4 Field masking (reuses `app.has_view_cost`, `COM-148`, directly — no new permission)

`app.vendor_rate_versions_directory` mirrors `COM-148`'s `app.costing_responses_directory` shape exactly: `security_invoker = false` **from the start** (the `COM-147`/`148` lesson — an invoker-mode view cannot read a column-`REVOKE`d field at all, even inside a masking `CASE`, since Postgres checks column privileges against the invoker for every referenced column at parse time; no need to rediscover this a third time). Unlike leads/opportunities/costing requests, vendor rates are **tenant-wide reference data**, not per-owner/org-unit-scoped — this view's own row filter is `app.has_active_tenant_membership(tenant_id) or app.is_supreme_admin()` (mirroring `app.master_records`' own RLS shape), not `app.can_access_record`. `app.rate_selections_directory`, by contrast, *is* scoped via `app.can_access_record` through the owning costing request (mirrors `app.costing_responses_directory`'s join-through pattern), since a rate selection belongs to one costing request's own record-scope.

### 3.5 Lifecycle functions

`app.create_rate_version` (gated by `app.is_support_grant_authority`, reused unchanged per `ADR-0015`; idempotent get-or-create of the vendor/lane identity row via `app.create_master_record`, PLT-120; an optional `p_supersedes_version_id` folds "revise" into "create" — the new row inherits the source's `master_record_id` and the source is immediately marked `superseded`, the same pattern every prior Commercial capability's revision functions use). `app.approve_rate_version` / `app.reject_rate_version` (mandatory reason) / `app.withdraw_rate_version` (mandatory reason, approved-only) — all `is_support_grant_authority`-gated, optimistic concurrency. `app.search_vendor_rates` (requires ordinary `COM:View`; a design correction made mid-authorship — see §8 — queries the base tables directly with an explicit-actor cost mask rather than composing on the auth.uid()-keyed directory view). `app.select_vendor_rate` (requires both `COM:Edit` and `COM:View cost`, mirroring `COM-148`'s `app.submit_costing_response` dual gate; snapshots the selected rate — or an ad-hoc entry — into `app.rate_selections`; selecting a non-approved rate, or an ad-hoc entry, always requires a non-empty `override_reason`).

### 3.6 Service layer and UI

`server/contracts/rate/rate.ts` (Zod; a distinct `RateVersionRecordSchema`/`RateVersionRecord` exists alongside `RateVersionSchema`/`RateVersion` — see §8, defect 1), `server/queries/rate.ts`, `server/mutations/rate.ts`. New route `app/(tenant)/[tenantSlug]/commercial/rates/` (list: active + pending-approval tables, a create form) and `app/(tenant)/[tenantSlug]/commercial/rates/[rateVersionId]/` (detail + approve/reject/withdraw actions panel). The Costing Request Detail page (`COM-148`) gained a "Rate selections" section and a `SelectRateForm` (catalog-pick or ad-hoc, with a mandatory override reason for the ad-hoc path). Disclosed UI scope boundary: the Rates list has no lane/service filter or comparison sort — the fully tested `app.search_vendor_rates` RPC is ready for a richer lookup UI, but this bounded slice ships the two tables plus create/review-by-detail-page, matching every prior Commercial list page's own bounded-first-iteration convention.

## 8. Errors / recovery / risks / issues

Three real defects found and fixed during this checkpoint's own authoring:

1. **`app.create_rate_version`/`approve`/`reject`/`withdraw` return `app.vendor_rate_versions` base-table rows, which have no `vendor_code`/`vendor_name` (those live only on the joined `app.master_records` row the directory view adds) and no `cost_masked` flag** — the mutation TypeScript layer's first draft tried to parse every RPC result through the same `RateVersionSchema` (directory shape) used for reads, and failed real unit-test assertions with `expected string, received undefined` for `vendorCode`/`vendorName`. Fixed by introducing a distinct `RateVersionRecordSchema`/`parseRateVersionRecord` (the base-table shape) for mutation return values, keeping `RateVersionSchema` (directory shape, `vendor_code`/`vendor_name`/`cost_masked` present) exclusively for read-path functions — a genuine structural difference, not merely masking, so a single shared schema was the wrong design.

2. **`app.v_active_vendor_rates`'s pre-existing `PLT-120` db-test assertion broke** once this checkpoint's own real, approved `vendor_rate_versions` rows existed: `scripts/db-tests/master-data.sql`'s own scenario asserted "at least one active vendor_rate row in the view" and queried a `code` column that no longer exists on the redefined view. Root-caused to the disclosed, intentional behavior change in §3.3/§6. Fixed by rewriting that assertion to expect 0 rows *for its own tenant* (the view is genuinely empty for a fixture that only seeds a master-identity row, never a priced version) — and, since `scripts/db-tests/commercial-rate-cost-lookup.sql` sorts alphabetically *before* `master-data.sql` and legitimately creates real approved rows under its own, different tenant, the fix was scoped by `tenant_id` from the start rather than a bare global count — proactively applying the `COM-145`/`COM-148` cross-file audit-log-count-fragility lesson to a *new* class of shared-view assertion, rather than waiting to rediscover the same class of bug a third time.

3. **`app.search_vendor_rates` silently returned zero rows when composed on the auth.uid()-keyed `app.v_active_vendor_rates` view** — the function takes an explicit `p_actor_auth_user_id` parameter (matching every other Commercial function's convention) and gates via `app.evaluate_permission` with that explicit actor, but its first draft then queried `app.v_active_vendor_rates`, whose own tenant-visibility/cost-masking predicates read the *session's* `auth.uid()` by default, not the function's own explicit parameter. Found via a genuine `db:test` failure (a fully-entitled outsider's search legitimately returning 0 rows) traced to a Postgres custom-placeholder-GUC quirk in the local test harness (`request.jwt.claims` reverting to an empty string rather than a clean unset value across sequential `DO` blocks, then failing `current_setting(...)::json` outright) — but the underlying design issue is real regardless of that harness quirk: any caller of this function without a live session JWT (e.g., a future service-role-context caller) would silently see zero results. Fixed by rewriting `app.search_vendor_rates` to query `app.vendor_rate_versions`/`app.master_records` directly, applying the `approved`+currently-effective filter and cost masking (`app.has_view_cost(tenant_id, p_actor_auth_user_id)`, explicit actor) inline — correct regardless of calling context, and not a duplication of business logic since it is the same filter/masking the views already apply, just parameterized explicitly.

Zero Critical/High-severity issue remains open. Zero regression to any prior capability (full `db:test`/`test`/`next build` re-run, all green, including the one corrected sibling test file).
