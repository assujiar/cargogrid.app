## 3. Implementation

### 3.1 Schema ‚Äî `supabase/migrations/20260723120000_create_commercial_prospect_lifecycle.sql`

`app.prospects` ‚Äî governed pre-customer identity, unique on `lead_id` ("one conversion idempotency key yields one prospect outcome," Prompt 144 ¬ß24). Contact fields are a **snapshot** taken at conversion time (`contact_name`/`email`/`phone` copied from the lead row, never a live FK) ‚Äî the no-reentry binding rule. `app.leads` gains an additive `converted_prospect_id` column plus a `CHECK` pairing it with `status='converted'`.

### 3.2 Conversion (tasks 1/2) ‚Äî main and alternative flows

`app.convert_lead_to_prospect` ‚Äî the main flow (¬ß21): idempotent on `lead_id` (a repeated call, including under a concurrent-insert race, returns the original prospect), requires the source lead be `status='qualified'`, requires record-scope access to the lead, and requires `COM:Create`. `app.link_lead_to_existing_prospect` ‚Äî the alternative flow (¬ß22): links a qualified lead to an existing, accessible prospect instead of creating a new one, requiring access to *both* records and denying a cross-tenant link structurally. Both set the lead's own `status='converted'`/`converted_prospect_id`.

### 3.3 Duplicate detection and conversion readiness (tasks 1/3)

`app.find_duplicate_prospects` mirrors `app.find_duplicate_leads`'s own tenant-scoped, fail-closed design exactly. `app.get_prospect_conversion_readiness` is a real-time, deterministic evaluator over a **fixed** rule set (legal name, tax id, billing address, at least one contact identifier) ‚Äî disclosed as bounded and non-tenant-configurable, not the deferred Configuration Engine rule evaluator (`ADR-CAND-ARCH-014`/`015`, still open per `PLT-121` ¬ß25).

### 3.4 Disqualify/archive/merge (task 3)

`app.disqualify_prospect` (mandatory reason, both application- and `CHECK`-enforced) and `app.archive_prospect` (no reason ‚Äî a distinct "no longer pursuing" transition, not a commercial-fit judgment) both require `active` status and record-scope access. `app.merge_prospects` mirrors `app.merge_leads` exactly (lineage preserved via `merged_into_id`, cross-tenant merge structurally denied, re-merge rejected).

### 3.5 Record-scope reuse (no new mechanism)

Reuses `app.lead_record_scope_org_unit_ids` (`COM-143`) directly for `app.prospects`' own RLS policy and every mutation's in-body scope check ‚Äî its body is generic (ancestors + self of an org unit) despite the lead-specific name; a byte-identical second wrapper would be exactly the duplicate utility `AGENTS.md` forbids. Renaming it would require editing `COM-143`'s own applied migration, which this repository's immutable-migrations convention forbids.

### 3.6 Service layer and UI (task 4)

`server/contracts/prospect/prospect.ts` (Zod), `server/queries/prospect.ts`, `server/mutations/prospect.ts` ‚Äî same `Pick<SupabaseClient, "rpc"|"from">` convention `COM-143` corrected to. The Lead Detail page (`COM-143`) gained a "Convert to prospect" action, wired to `app.convert_lead_to_prospect` and redirecting to the new prospect's detail page ‚Äî the one bounded mutating entry point this checkpoint's own UI needs, matching the same "core management, not an all-actions mega task" scoping already used.

## 8. Errors / recovery / risks / issues

Two real defects found and fixed during this checkpoint's own authoring:

1. **Migration statement order**: the `app.leads` `ALTER TABLE ADD COLUMN converted_prospect_id uuid references app.prospects (id)` was originally written before `CREATE TABLE app.prospects`, causing a real `relation "app.prospects" does not exist` failure on first apply. Fixed by reordering ‚Äî create the table first, then add the referencing column and its `CHECK` constraint afterward.
2. **Ambiguous `text[] || text` operator resolution**: `app.get_prospect_conversion_readiness`'s `v_missing := v_missing || 'tax_id'` raised a real `malformed array literal` error ‚Äî Postgres resolved the untyped string literal against the `anyarray || anyarray` overload rather than `anyarray || anyelement`. Fixed with an explicit `::text` cast on each appended literal.

Zero Critical/High-severity issue. Zero regression to any prior capability (full `db:test`/`test`/`next build` re-run, all green).
