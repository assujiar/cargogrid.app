## 3. Implementation

### 3.1 Scope boundaries (disclosed, migration header)

1. **No AR/GL/payment balance of any kind is modeled or invented** (Prompt 157 §24). `app.credit_check_snapshots` has no exposure/balance/outstanding column at all — the check evaluates only `approved_limit_amount` (or a valid override) against the one requested transaction amount. Phase 4 Finance is the real, disclosed, un-built consumer/producer of any such signal; this check's own stable shape is the "stable Finance integration contract" Prompt 157 §14 asks for.
2. **Governance routing reuses `PLT-121`'s single, generic `config_type_code='approval'` config object directly** — unlike `COM-153`'s own *conditional* (threshold-crossed) routing, every credit profile request is *unconditionally* routed: Prompt 157 §24's "customer creation does not imply credit approval" means credit governance is never skipped.
3. **"MFA for privileged approvers" reuses `PLT-115`'s own reauth-freshness mechanism** rather than inventing a second, unproven pattern — applied to every privileged-approver action this capability adds: deciding a routed request, hold, release, and creating an override. No live challenge UI exists; the account-detail forms capture a checked-attestation timestamp client-side, which the server independently re-validates for freshness on every call, never trusted blindly.
4. **Credit limit/override/check amounts are masked via the already-established `COM:View selling price`** (`COM-147`), not a new permission — the same reuse choice `COM-156` already made for contract pricing.
5. **One credit profile "in flight" at a time per account** — database-enforced via a partial unique index; a fresh request after rejection/expiry links back via `supersedes_profile_id`.
6. **Expiry is lazy-flipped on read/check**, the same mechanism `app.quotation_acceptance_tokens` (`COM-154`) already established.
7. **Legacy credit data migration is not applicable** — greenfield, no live environment.

### 3.2 Schema — `supabase/migrations/20260724310000_create_commercial_credit_commercial_control.sql`

New `app.credit_profiles` (one row per requested/active credit relationship; `status` requested/active/held/expired/rejected; `supersedes_profile_id` self-FK for a fresh request after history). New `app.credit_profile_overrides` (a bounded, reasoned, always-expiring exception — never a silent permanent limit change). New `app.credit_check_snapshots` (the deterministic, immutable, append-only evidence record).

`app.request_customer_credit_profile` — `COM:Create`-gated, unconditionally opens a routed `entity_type='credit_profile'` request via `app.request_approval`, failing closed (`approval_definition_not_configured`) if the tenant has never published a routing definition. `app.decide_credit_profile_approval_step` — wraps `app.decide_approval_step` (unchanged) and syncs the profile only once the bound request reaches a final state (`active` with `approved_limit_amount` set on approval; `rejected` with a reason otherwise); reauth-freshness gated. `app.hold_credit_profile`/`app.release_credit_profile` — `COM:Approve` + reauth-freshness gated, mandatory hold reason. `app.create_credit_override` — `COM:Approve` + reauth-freshness gated ("elevated approval"), mandatory reason, mandatory future expiry. `app.check_customer_credit` — the deterministic pre-conversion check: lazily flips an expired profile in the same transaction, evaluates hold/not-active/currency-mismatch/limit against the effective limit (a currently-valid override, if any, else the approved limit), persists an immutable snapshot, and raises no structural exception for a legitimate business outcome — every call produces a real, pinnable row. Its own returned row is masked per-caller (`COM:View selling price`) directly in the function body (the same "mask the function's own output" technique `app.get_effective_customer_price`, `COM-156`, and `app.search_vendor_rates`, `COM-149`, already established) rather than gating the whole function, so an ordinary `COM:View` holder still sees the real outcome (Prompt 157 §26) without the raw figures.

### 3.3 Service layer and UI

`server/contracts/credit/credit.ts` (schemas/parsers for `CreditProfile`/`CreditProfileOverride`/`CreditCheckResult`). `server/queries/credit.ts` — masked-directory reads plus an approval-inbox composition (`getCreditProfileApprovalOverview`/`listCreditProfileApprovalInboxForActor`) mirroring `server/queries/quotation-approval.ts` (`COM-153`) exactly, filtered to `entity_type='credit_profile'`. `server/mutations/credit.ts` — thin wrappers around the six RPCs; `checkCustomerCredit` lives here (not in queries) since it always persists a new snapshot row as its real side effect.

UI: a `CreditPanel` embedded on the Account Detail page (`COM-155`'s own route) — profile status/masked amounts, `RequestCreditForm` (no live profile), `CreditApprovalDecisionForm` (eligible-approver decision, with the reauth-freshness checkbox), `HoldReleaseForm`/`OverrideForm` (active/held), `CreditCheckForm` (the eligibility widget). New `commercial/credit-approvals/` inbox route mirroring `commercial/approvals/` (`COM-153`) — a routing list only, deciding happens on the account page. `commercial/layout.tsx` gained a "Credit Approvals" nav link.
