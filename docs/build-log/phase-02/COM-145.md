## 3. Implementation

### 3.1 Scope boundary (disclosed, task 1)

Prompt 145 §13 names "address/site reference" alongside contacts/activities. This checkpoint **defers a standalone canonical site/address entity** to whichever later capability first needs a durable, reusable address master against a real account/customer row (`COM-155`/`156`) — `app.prospects` (`COM-144`) already carries its own `billing_address` snapshot for its own narrow need, and building a second interim site model before any capability needs multi-site tracking risks the exact "two masters to reconcile later" duplication this repository's binding rules forbid. Disclosed here, not silently narrowed.

### 3.2 Schema — `supabase/migrations/20260723150000_create_commercial_contact_activity_management.sql`

`app.contacts` (canonical, reusable person identity — dedupe via a new person-identity fingerprint, email/phone normalization reused directly from `app.normalize_lead_email`/`app.normalize_lead_phone`, `COM-143`, rather than re-authoring identical logic). `app.contact_links` (polymorphic contact↔record relationship, roles `primary`/`billing`/`technical`/`decision_maker`/`other`, one contact may hold several roles on the same record). `app.activities` (unified `call`/`email`/`meeting`/`visit`/`follow_up`/`task` timeline, polymorphic target, optional contact).

### 3.3 Shared polymorphic resolver (task 1)

`app.resolve_commercial_record_ref(related_type, related_id)` is the one shared lookup every polymorphic table this phase adds reuses (extended by an `IF/ELSIF` per `related_type`, never a second competing resolver) — resolves tenant/owner/org_unit for a `lead` or `prospect` row, raising on an unknown type or a dangling reference.

### 3.4 Contact reuse and linking (tasks 1/2)

`app.create_contact` always creates a new row (no external-system idempotency key exists for a manually-entered contact, unlike `app.capture_lead`) — duplicate review is the caller's own responsibility via `app.find_duplicate_contacts` before calling it (main/alternative flow, §21/§22). `app.link_contact_to_record` is idempotent on `(contact_id, related_type, related_id, role)` including under a concurrent-insert race, requires record-scope access to *both* the contact and the target record, structurally denies a cross-tenant link, and requires `COM:Edit`. `app.unlink_contact_from_record` removes the relationship without touching the contact itself.

### 3.5 Activity lifecycle (tasks 1/3)

`app.log_activity` covers both already-happened activities (`status='completed'`, `completed_at` set directly) and to-do activities (`status='scheduled'`, `due_at` set) — the caller chooses the initial status. `app.complete_activity`/`app.reschedule_activity`/`app.cancel_activity` are optimistic-concurrency-checked and require `scheduled` status plus record-scope access. Every activity is append-oriented business history (Prompt 145 §24) — never deleted, only transitioned; `app.capture_audit_event`'s own before/after payload is the "corrections preserve before/after evidence" mechanism.

### 3.6 Record-scope reuse (no new mechanism)

All three new tables reuse `app.can_access_record` (`PLT-114`) and `app.lead_record_scope_org_unit_ids` (`COM-143`) directly for RLS and in-body mutation checks — contacts/activities carry their own `owner_user_id`/`org_unit_id` (set at creation, a deliberate simplification over per-relationship-derived visibility, disclosed rather than silently assumed). `app.contact_links` uses a coarser tenant-membership-only read policy (a disclosed trade-off — a bare link row reveals only that some contact is linked to some record, not the record's own content, which remains separately RLS-gated).

### 3.7 Service layer and UI (task 4)

`server/contracts/contact/contact.ts` (Zod), `server/queries/contact.ts`, `server/mutations/contact.ts`. New route `app/(tenant)/[tenantSlug]/commercial/contacts/` (directory with a create form; read-only detail — link management remains a real, tested service-layer capability without a dedicated button in this bounded slice, matching precedent). A new shared, non-route `_shared/` folder holds `activity-timeline.tsx`/`activity-actions.ts`, embedded in **both** the Lead Detail (`COM-143`) and Prospect Detail (`COM-144`) pages, since `app.activities` links polymorphically to either — avoids forking the timeline UI per record type.
