## 3. Implementation

### 3.1 Scope boundaries (disclosed, task 1)

Exact, versioned, explainable margin/markup calculation over a selected cost snapshot (`app.rate_selections`, `COM-149`) and selling inputs. Four boundaries disclosed up front: (1) money stays PostgreSQL `numeric` end to end ‚Äî never binary floating point (Prompt 150 ¬ß24) ‚Äî every amount/percentage column is `numeric`, and every arithmetic step in `app.calculate_margin` uses explicit `round(..., 2)`. (2) Margin rules are tenant-wide only in this bounded slice (no per-org-unit rule, unlike `COM-146`'s own org-scoped `app.sales_plans`) ‚Äî exactly one published `app.margin_rule_versions` row per tenant at a time. (3) FX/multi-currency conversion is out of scope ‚Äî `app.calculate_margin` requires the selling currency to exactly match the pinned cost snapshot's currency and fails closed (`mixed_currency`) otherwise. (4) This is the calculation engine and its own governed override, not the Quotation Builder itself (`COM-151`, the next capability, is expected to be the first real consumer that issues a customer-facing quote).

### 3.2 Schema ‚Äî `supabase/migrations/20260724180000_create_commercial_margin_calculation.sql`

`app.margin_rule_versions` (a versioned, tenant-wide minimum-margin-percentage policy; `draft`‚Üí`published`‚Üí`archived`; `app.publish_margin_rule_version`'s `p_supersedes_version_id` archives the prior published rule, mirroring `app.publish_sales_plan`, `COM-146`, exactly ‚Äî at most one published rule per tenant, enforced by a partial unique index). `app.margin_calculations` (one deterministic calculation per `(rate_selection, sell inputs)`, pinning the exact cost snapshot and the exact rule version/minimum/rounding in effect at calculation time ‚Äî a later edit to either never silently reprices or re-judges a past calculation, Prompt 150 ¬ß24; a recalculation for the same `rate_selection_id` creates a new row and marks the prior one `is_current=false`/`superseded_by_id`, mirroring `app.pipeline_outcomes`'s own `is_current`/`superseded_by_id` chain, `COM-146`).

### 3.3 Reuses, not duplicates

`app.rate_selections` (`COM-149`) is the one cost-snapshot source (never a second cost input path). `app.has_view_cost` (`COM-148`) masks cost/margin/markup figures; `app.has_view_selling_price` (`COM-147`) masks sell-side figures ‚Äî no new permission is seeded. `COM:Approve` gates both publishing a margin rule and overriding a below-threshold result, mirroring `COM-146`'s `app.publish_sales_plan`/`COM-147`'s close-to-won gate exactly (a governance action, not an ordinary edit). Every masked view is `security_invoker = false` from the start (the `COM-147`/`148`/`149` lesson, not rediscovered a fourth time), with its own explicit `app.can_access_record(...)` row filter through the owning costing request ‚Äî the same join-through pattern `app.costing_responses_directory`/`app.rate_selections_directory` already established.

### 3.4 Field masking ‚Äî two independent dimensions

`app.margin_calculations_directory` masks cost/margin/markup figures behind `COM:View cost` and sell/discount/net-sell figures behind `COM:View selling price` independently ‚Äî an actor can hold either, both, or neither. `threshold_outcome`/override state stay visible to anyone with record access (governance status, not a money figure). `app.margin_rule_versions` (the policy config itself) is tenant-wide visible without masking, mirroring `app.win_loss_reasons`/`app.master_records`'s own RLS shape ‚Äî a minimum-margin-percentage policy is not itself a specific deal's financial figure.

### 3.5 Lifecycle functions (tasks 2/3)

## 8. Errors / recovery / risks / issues

One real defect found and fixed in this checkpoint's own migration during authoring, and one real, pre-existing latent defect found and fixed in `COM-149`'s own db-test file:

1. **`app.calculate_margin`'s recalculation-supersede logic initially updated the prior current row's `superseded_by_id` to the new row's id *before* inserting that new row** ‚Äî a foreign-key violation, since `superseded_by_id` must reference a row that already exists. Found via a genuine `db:test` failure on the very first recalculation scenario. Fixed with the correct three-step order: (1) mark the prior current row `is_current=false` (no `superseded_by_id` yet, so the unique partial index on `(rate_selection_id) where is_current` is never violated by the new insert), (2) insert the new row, (3) only now set the prior row's `superseded_by_id` to the new row's real id.

2. **A pre-existing, latent cross-file test fragility in `COM-149`'s own `scripts/db-tests/commercial-rate-cost-lookup.sql`**, exposed for the first time by this checkpoint's own fixture: that file's rate-version/rate-selection lookups (`select * into v_rate from app.vendor_rate_versions where origin_lane = 'Jakarta' and destination_lane = 'Surabaya' and mode = 'FCL'`, and similar) carried no `tenant_id` filter ‚Äî safe only as long as no other capability's fixture ever reused the same lane/mode combination in the same shared disposable database. This checkpoint's own margin-calculation fixture legitimately creates a rate under the identical `Jakarta`/`Surabaya`/`FCL`/`ocean_freight` combination, and once `commercial-margin-calculation.sql` runs before `commercial-rate-cost-lookup.sql` (alphabetical test-file order), the latter's unscoped lookups became genuinely ambiguous ‚Äî manifesting as a `db:test` failure where a real, valid tenant_admin was reported to lack authority, because the fetched row actually belonged to a *different* tenant entirely. Root-caused via direct data inspection (not assumed) after ruling out a session-state/RLS explanation, since each db-test file runs in its own fresh `psql` connection (no GUC leakage across files is possible, unlike the intra-file `request.jwt.claims` quirk found at `COM-149`). Fixed durably ‚Äî the same class of fix `COM-145`/`148` established for unscoped audit-log counts, applied here to unscoped entity lookups instead: every affected query in `commercial-rate-cost-lookup.sql` (rate-version/rate-selection lookups across five scenario groups) now scopes by `tenant_id = v_tenant1`.

Zero Critical/High-severity issue remains open. Zero regression to any prior capability (full `db:test`/`test`/`next build` re-run, all green, including the one corrected sibling test file).
