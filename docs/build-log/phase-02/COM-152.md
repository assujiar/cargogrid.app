## 3. Implementation

### 3.1 Scope boundaries (disclosed, task 1)

Five boundaries disclosed up front, each also recorded in the migration's own header comment:

1. **"issued"/"accepted" status values do not exist yet.** Prompt 152 §23/§24 name an "accepted lock" exception, but no capability (`COM-153` Quotation Approval, `COM-154` Customer Acceptance) has run to ever produce an "approved"/"sent"/"accepted" status. This checkpoint does not invent those enum values ahead of a real producer — the lock rule is bounded to what is real today: a version may be revised only while it `is_current` and its `status <> 'cancelled'`. The extension point for `COM-153`/`154` to widen this once real approved/accepted states exist is named in the migration's own header, not silently assumed.
2. **"Supersede" is not a separate user-invoked action** — it is the internal side-effect `app.create_quotation_revision` performs on the prior current row, mirroring `app.calculate_margin`'s own supersession chain (`COM-150`).
3. **"Compare" is computed in the service layer**, not a third SQL function — `server/contracts/quotation/quotation-diff.ts`, pure and unit-tested. Two already field-masked directory reads already carry every access decision the diff needs; there is no real database-side rule left to enforce a third time.
4. **RPD-022 (Supreme Admin absolute CRUD) is disclosed, not newly implemented** — Supreme Admin's existing service-role authority is unaffected by these normal-role lock checks, and no version is claimed tamper-proof or immutable.
5. **Legacy data migration is not applicable** — greenfield table, no live environment.

### 3.2 Schema — `supabase/migrations/20260724240000_create_commercial_quotation_versioning.sql`

Widens `app.quotations` (never edits `COM-151`'s own migration file, per `AGENTS.md`): adds `root_quotation_id` (the version-1 row's own id, backfilled and then set `NOT NULL`), `version_number` (monotonic per root), `is_current` (true for exactly one row per root — a real partial-unique-index guarantee, `quotations_root_current_unique`, mirroring `app.margin_calculations_current_unique`), `superseded_by_id`, `revision_reason`. Drops `COM-151`'s `unique(tenant_id, quote_number)` and replaces it with `unique(tenant_id, quote_number, version_number)` — multiple version rows now legitimately share one `quote_number`. Adds `unique(root_quotation_id, version_number)` (Prompt 152 §25: "Version sequence/root uniqueness... are database-enforced").

### 3.3 Widened functions (never edited, `CREATE OR REPLACE FUNCTION`/`VIEW` with identical signatures — the same technique `COM-149` used on `app.can_access_record` and `COM-132` used on three `PLT-131` functions)

`app.create_quotation_draft`/`app.clone_quotation` now set `root_quotation_id`/`version_number=1`/`is_current=true` on the first row of a new quote/clone (a clone is always a brand-new root, never a revision — a deliberately distinct concept from versioning). `app.add_quotation_line`/`app.remove_quotation_line`/`app.update_quotation_terms`/`app.submit_quotation` now additionally require `is_current = true` — a superseded historical version can never be directly edited even though its own `status` column might still literally read `draft` (it was revised before ever being submitted). `app.get_quotation_submission_readiness` gained a `not_current_version` reason. `app.quotations_directory` gained the five new columns, appended at the end (the only shape `CREATE OR REPLACE VIEW` structurally allows).

### 3.4 The core versioning function (tasks 2/3)

`app.create_quotation_revision(p_source_quotation_id, p_reason, p_actor_auth_user_id, p_actor_label)` serves both flows identically: `p_source_quotation_id` may be the current version (the ordinary "revise" main flow) or any historical version of the same root (the "restore as new draft" alternative flow). `COM:Edit` + record access gated; mandatory non-empty `p_reason`. Concurrency: `select ... from app.quotations where root_quotation_id = ... and is_current for update` — real row-level locking (not merely optimistic concurrency) on the root's current version, so a genuinely concurrent second call blocks until the first commits, then naturally proceeds against the fresh current row (correct sequential behavior, not corruption) rather than racing. Copies header and every line from the source into a new row (`version_number = max(root)+1`), marks the prior current row `is_current=false`/`superseded_by_id` (the exact three-step insert-then-link order `COM-150`'s own `app.calculate_margin` established, to satisfy the partial unique index before the new row exists).

### 3.5 Service layer and UI (task 4)

`server/contracts/quotation/quotation.ts` gained the five new fields; `server/contracts/quotation/quotation-diff.ts` (new — `diffQuotationVersions`, pure, correlates lines by sourcing `marginCalculationId` when present, else `(lineType, description)`, disclosed as a best-effort match, not a claimed perfect semantic diff). `server/queries/quotation.ts` gained `listQuotationVersions`; `server/mutations/quotation.ts` gained `createQuotationRevision`. The Quotation Detail page (`COM-151`) gained: a Version column in the header; line/term/submit editing gated on `status === "draft" && isCurrent` (restated at the UI layer — the real gate is server-side); `VersionHistory` (every version of the root, oldest first, with View/Compare links); `RevisionForm` (mandatory reason, label switches between "Create new revision"/"Restore as new draft" depending on which version is being viewed, same underlying action); `ComparisonPanel` (renders `?compareWith=<versionId>`'s computed diff).
