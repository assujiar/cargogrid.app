## 3. Implementation

### 3.1 Scope boundaries (disclosed, task 1)

Six boundaries disclosed up front, each also recorded in the migration's own header comment:

1. **No canonical customer/account entity exists yet** (`COM-155` has not run) — `app.quotations` references `app.opportunities`/`app.prospects` directly, the same boundary `COM-147` already established for `app.opportunities`.
2. **No canonical address master exists yet** — `customer_snapshot` is a bounded `jsonb` snapshot copied from `app.prospects`' own already-real `legal_name`/`trade_name`/`billing_address`/`contact_name`/`contact_email`/`contact_phone` columns at creation time, not an invented address master.
3. **The Configurable Numbering Engine (`PLT-125`) is not adopted.** No Commercial capability (`143`–`150`) has adopted it either — it requires a published per-tenant numbering configuration version nothing in this repository seeds yet. `app.next_quotation_number()` is a real, atomic, tenant-scoped monotonic counter (`QTN-YYYY-NNNNNN`), stable and never reused, satisfying Prompt 151 §24's "canonical identity is stable" without pulling in machinery no sibling capability has wired up either.
4. **Line editing is add/remove, not in-place numeric edit.** `app.add_quotation_line`/`app.remove_quotation_line` cover the real editing need; changing a line's numbers is done by removing and re-adding it.
5. **Autosave is an explicit, optimistically-concurrent save action**, not real-time autosave-on-keystroke — no realtime/websocket infrastructure exists anywhere in this repository yet. Every mutation carries `p_expected_version`; a stale concurrent edit fails closed (`stale_version`).
6. **Document generation/private signed-URL preview is `BLOCKED`.** No asset-upload/storage pipeline exists anywhere in this repository (confirmed during this same session's prior CargoGrid Design System Expansion task, `docs/design-system/07_GAP_ANALYSIS_AND_ROADMAP.md` §4). `document_ref` is a forward-compatible, unpopulated column; no PDF is generated.

Revisioning/versioning (monotonic version numbers, supersession, issued/accepted locks — Prompt 152's own scope) and approval routing (threshold-driven approver resolution — Prompt 153's own scope) are explicitly not built here, per Prompt 151 §24's own "revisions are handled by Prompt 152."

### 3.2 Schema — `supabase/migrations/20260724210000_create_commercial_quotation_builder.sql`

`app.quotation_number_counters`/`app.next_quotation_number()` (atomic tenant-scoped number allocation, boundary 3 above). `app.quotations` (root — `quote_number`, `opportunity_id`, `source_opportunity_version` pin, `prospect_id`, `contact_id`, `customer_snapshot`, `currency`, `validity_from/to`, `terms`, four server-computed aggregate money columns, `status` `draft`/`submitted`/`cancelled`, `cloned_from_id`, `document_ref`, `submitted_at/by`). `app.quotation_lines` (typed lines — `line_type` `service`/`surcharge`/`fee`/`discount`, `margin_calculation_id` optional sourcing link, quantity/unit_price/discount_pct/tax_pct, four server-computed money columns, `cost_amount_snapshot`/`margin_pct_snapshot` copied from the sourcing margin calculation at add-time). `app.recalculate_quotation_totals()` is the single place the header's aggregate money columns are ever written, always derived from the current line set.

### 3.3 Reuses, not duplicates

`app.opportunities`/`app.prospects`/`app.contacts`/`app.margin_calculations` are read directly, never copied into a second table. `app.has_view_cost`/`app.has_view_selling_price` (`COM-148`/`147`) mask exactly as every prior Commercial capability's directory view does — no new permission-catalogue row. `app.lead_record_scope_org_unit_ids` (`COM-143`) is the record-scope resolver, reused verbatim. `app.capture_audit_event`/`app.evaluate_permission`/`app.can_access_record` (Platform Core) are unchanged.

### 3.4 Field masking — two independent dimensions

`app.quotations_directory` masks the header's four aggregate money columns behind `COM:View selling price` (`sell_masked`) — every other column (customer snapshot, terms, validity, status) is always visible to a record-scoped viewer, since this is the customer-offer header itself, not a cost figure. `app.quotation_lines_directory` masks sell-side line fields (`unit_price`/`discount_pct`/gross/discount/tax/total) behind `COM:View selling price` and the cost/margin snapshot (`cost_amount_snapshot`/`margin_pct_snapshot`) behind `COM:View cost` independently — mirrors `app.margin_calculations_directory`'s own two-dimension shape exactly.

### 3.5 Lifecycle functions (tasks 2/3)

`app.create_quotation_draft` (`COM:Create` + record access on the opportunity; builds `customer_snapshot` from the prospect; pins `source_opportunity_version`). `app.clone_quotation` (the "clone a prior draft with explicit origin" alternative flow, Prompt 151 §22 — copies header and every line into a brand-new draft, never mutates the source). `app.add_quotation_line`/`app.remove_quotation_line` (`COM:Edit`, draft-only, optimistic concurrency, exact `numeric` arithmetic with `round(..., 2)` at every money-producing step, mixed-currency rejected against a sourcing margin calculation). `app.update_quotation_terms` (`COM:Edit`; `terms` keys are a real, bounded allowlist — `payment_terms`/`incoterm`/`notes` — server-rejects any other key, Prompt 151 §24's "whitelisted variables... no arbitrary code"). `app.get_quotation_submission_readiness` (read-only; structural pass/fail + reason codes only, never a dollar figure — checks no-lines, zero-total, missing contact, expired validity, a stale opportunity snapshot, and a stale/superseded sourcing margin calculation). `app.submit_quotation` (`COM:Edit`; calls the readiness check internally and fails closed with the exact blocking reasons — `submission_not_ready` — never a bare rejection).

### 3.6 Service layer and UI (task 4)

`server/contracts/quotation/quotation.ts` (Zod; `CustomerSnapshotSchema`/`QuotationTermsSchema` map the bounded jsonb shapes), `server/queries/quotation.ts` (directory reads + the read-only readiness RPC), `server/mutations/quotation.ts` (the six write RPCs, unmasked since the caller already demonstrated authority). New route `app/(tenant)/[tenantSlug]/commercial/quotations/` (tenant-wide list) and `.../quotations/[quotationId]/` (the builder: header, lines table with per-line remove, add-line form sourcing from the opportunity's own current margin calculations, terms/contact form, submission-readiness panel with submit/clone actions). The Opportunity Detail page (`COM-147`) gained a "Quotations" section + `CreateQuotationForm`, mirroring `COM-148`'s own `RequestCostingForm` pattern exactly.

## 4. Tests

### 4.1 Real database tests (`scripts/db-tests/commercial-quotation-builder.sql`, via `pnpm run db:test`)

Setup (two tenants, a company/branch/team-a/team-b hierarchy, a manager holding `COM:Create/Edit/Approve/View/View selling price` but no `View cost`, a rep holding `COM:Create/Edit/View/View selling price/View cost`, a sibling-team outsider, a full opportunity→costing-request→rate-selection→margin-calculation chain, a published margin rule, a contact). `create_quotation_draft` (authority/record-access gated, cross-tenant opportunity rejected, invalid currency/validity rejected, `customer_snapshot` populated from the real prospect row, `quote_number` stable and unique across two drafts from the same opportunity). `add_quotation_line`/`remove_quotation_line` (draft-only, record-access gated, exact decimal line/header totals verified against concrete numbers, optimistic concurrency, a manually-added line's cost snapshot stays null). `update_quotation_terms` (unknown terms key rejected, a currency change conflicting with an existing sourced line's currency rejected). `get_quotation_submission_readiness`/`submit_quotation` (blocks on missing contact, becomes ready once satisfied, blocks on a deliberately staled opportunity snapshot via `app.update_opportunity`, submit fails closed with the exact reasons, succeeds once ready, and locks further line edits once submitted). `clone_quotation` (copies header and every line, authority/record-access gated). `app.quotations_directory`/`app.quotation_lines_directory` masking (both dimensions independently, base-table columns withheld from direct select, sibling-team outsider denied via record-scope). Full tenant-scoped audit trail. **One real test-authoring defect found and fixed during this checkpoint** (not a migration defect): the initial test file picked "the first quotation for this customer" via `order by created_at asc limit 1`, which is non-deterministic when two quotations are created within the same `do $$ ... $$` block (Postgres freezes `now()` for the whole transaction, so both rows share an identical `created_at`) — fixed by ordering on `quote_number` instead, which is guaranteed distinct and allocation-ordered by the atomic counter regardless of timestamp ties. All 41 db-test files pass together, zero regression to any prior capability.
