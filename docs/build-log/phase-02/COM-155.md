## 3. Implementation

### 3.1 Architecture decision — `ADR-0018`

Prompt 155 §20 task 1 asks to resolve "account vs. customer" naming and ownership — this is `ADR-CAND-ARCH-012`, open since Phase 2 kickoff. Wrote `docs/adr/ADR-0018-canonical-account-entity-shape-and-ownership.md`: a flat-column `app.accounts` table (Option 1), **not** built on `PLT-120`'s Master Data Engine (Option 2 — rejected: `register_master_type`/`create_master_record`'s Supreme-Admin/tenant-admin-grant authority model is the wrong shape for an ordinary `COM:Approve`-gated workflow action, and every sibling Commercial entity already succeeds with the flat-column pattern), **not** two separate `app.accounts`/`app.customers` tables (Option 3 — rejected: no real requirement drives the split yet). `customer_status` is the resolved account/customer distinction — one entity, one status field, not two tables. Updated `docs/adr/README.md`'s §6 index and §5.2 candidate table (`ADR-CAND-ARCH-012`: `BLOCKED` → `ACCEPTED → ADR-0018`) and the §1 count-reconciliation footnote (11 resolved/16 open → 12 resolved/15 open). `pnpm run docs:check` confirms ADR-index coherence.

### 3.2 Scope boundaries (disclosed, migration header + `ADR-0018`)

1. **No per-field masking on `app.accounts`.** Prompt 155 §16 asks to "restrict legal/tax/billing/credit fields" — attempted a new `COM:View billing` permission action and hit a hard wall: `app.permissions_action_check` (`supabase/migrations/20260716103445_create_roles_permissions.sql`) is a fixed, closed 19-action enum with no billing-shaped action; widening it is an architecture-level change out of this bounded task's scope. Resolved instead by enforcing the restriction one layer up — the `COM:Approve` gate on `app.convert_quotation_to_account` itself (a governance-weighted, hard-to-reverse action) — the same "tenant-wide, not masked" posture `app.margin_rule_versions`/`app.quotation_approval_rules` already established. `app.accounts` carries no `*_directory` view; `SELECT` is granted directly on the base table, mirroring `app.quotation_approval_rules`'s own precedent of skipping a view entirely when nothing needs masking.
2. **No credit field, no site/address master.** No credit-limit/credit-hold column exists on `app.accounts` (deferred — `COM-157` Credit and Commercial Control owns this). `billing_address` stays `jsonb`, mirroring `app.prospects.billing_address`'s own shape — `COM-145`'s own prior deferral of a dedicated site/address master restated as still-deferred.
3. **No merge-accounts function.** `app.accounts.merged_into_id`/`status='merged'` are structural columns for a future dedup/merge capability; no merge RPC exists yet in this bounded slice.
4. **No legacy data migration / backfill.** A pre-existing quotation with no `app.account_conversions` row simply has never been converted — no retroactive account is fabricated.
5. **Idempotent, concurrency-safe create-or-link**, not lock-based: `unique(quotation_id)` on `app.account_conversions` is the real idempotency guarantee; a `unique_violation` on `app.accounts`'s own partial unique fingerprint index (a genuine race between two concurrent conversions of two duplicate quotations) is caught and gracefully re-resolved as "link to the winning row," never an error.

### 3.3 Schema — `supabase/migrations/20260724290000_create_commercial_customer_account_conversion.sql`

New `app.accounts` (`legal_name` not null, `trade_name`, `tax_id`, `normalized_legal_name`/`normalized_tax_id`/`duplicate_fingerprint` reusing `COM-144`'s own normalization functions, `billing_address` jsonb, `customer_status` active/inactive, `parent_account_id` self-FK for subsidiaries, `source_prospect_id`, `status` active/merged + merge columns, owner/org-unit/record-version/audit columns). Database-enforced anti-duplication: partial unique index `accounts_tenant_fingerprint_active_unique on (tenant_id, duplicate_fingerprint) where status='active'`. New `app.account_conversions` (`unique(quotation_id)` — one final conversion outcome per quotation, ever; `outcome` created/linked_existing; `duplicate_candidate_ids` audit trail of what was offered at conversion time).

`app.find_duplicate_accounts` (tenant-scoped preview, fails closed on missing membership) and `app.get_account_conversion_readiness` (read-only: `ready`/`blocking_reasons`/`duplicate_candidate_ids`, reason codes only, never a dollar figure) are the pre-conversion read RPCs. `app.convert_quotation_to_account` is the one atomic, `COM:Approve`-gated write: idempotent (early-returns the existing account for an already-converted quotation, before any authority re-check — a retry is always safe); requires `customer_decision='accepted'`; creates a new account (computing the duplicate fingerprint, catching a concurrent `unique_violation` as a link-to-winner) or links an explicitly chosen `target_account_id`; re-links every one of the source prospect's existing `app.contact_links` rows onto the resulting account via `app.resolve_commercial_record_ref`'s widened `'account'` branch (contacts are reused, never re-created); backfills `app.opportunities.account_ref` (closing `COM-147`'s own disclosed forward-reference); captures one audit event on the success path only (the idempotent early-return path leaves no new audit trace, proven in the db-test's exact-count assertion).

`app.contact_links.related_type`'s CHECK constraint widened (`lead`/`prospect`/`opportunity` → `+account`) via `DROP`/`ADD CONSTRAINT`, the same mechanism every prior capability in this chain already used — no prior migration file edited.

RLS: `accounts_select_scoped`/`account_conversions_select_scoped` use `has_active_tenant_membership(tenant_id) or is_supreme_admin()` — tenant-wide visibility, not record-scoped, matching `app.margin_rule_versions`/`app.quotation_approval_rules`'s own precedent (verified directly: a sibling-team outsider with no ownership stake sees every account in the tenant). `select` granted to `authenticated, service_role`; `insert/update/delete` `service_role`-only; `execute` on the three new functions to `authenticated, service_role`.
