## 3. Risk-matrix design correction found during authoring (disclosed, not hidden)

A first draft used a multiplicative likelihood×impact score with fixed thresholds; running `pnpm run threat-model:check` against the real 25-entry register produced **zero `critical`-ranked entries** — including for threats (e.g. the malware-scan-bypass gate, `THR-011`) that `docs/architecture/10_TESTING_WORKSTREAM.md` §5.2 already rates Critical severity regardless of likelihood. A multiplicative score with `likelihood=medium` capped every "critical impact" threat at `high`, contradicting the already-`VERIFIED` source catalogue's own severity framing. Fixed by replacing the score with an explicit lookup table (`RISK_MATRIX`, `scripts/security/threat-model.ts`) where `critical` impact reaches `critical` rank once likelihood is at least `medium` — re-verified: the register now computes 4 `critical`, 11 `high`, 9 `medium`, 1 `low`, consistent with `10_*.md`'s own severity labels for the entries that map onto its catalogue. This is exactly the kind of design defect this checkpoint's own "risk ranking is reproducible" validation rule (§25) exists to catch, and it caught it before commit.
