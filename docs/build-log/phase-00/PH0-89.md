## 3. Design decisions

1. **`eslint-plugin-import` for boundary enforcement — existing toolchain, not a new choice.** Already a transitive dependency of `eslint-config-next` (confirmed via `node -e 'import("eslint-config-next")...'` showing `plugins: [... 'import' ...]`, `PH0-085`'s install). Promoted to a direct `devDependency` (pinned to the exact already-resolved version, `2.32.0`) so `eslint.config.js` can reference `import/no-restricted-paths` directly — satisfies §20 task 3 ("using existing toolchain where possible") without adding a genuinely new dependency choice; not treated as an ADR-worthy decision (no alternative was evaluated — this is direct application of already-present tooling, unlike `ADR-0002`/`ADR-0003`'s real either/or choices).
2. **Two automated boundary rules, not the full pairwise domain matrix.** `docs/architecture/01_MODULE_DEPENDENCY_MAP.md` §3.3's full domain-to-domain edge table is large, most domain files don't exist yet, and several edges are semantic exceptions (e.g. the `COM→OPS` read-only revenue-snapshot) a path-based rule cannot express. Automated only the two invariants that are both fully evidenced and mechanically expressible: platform-kernel-never-imports-domain, and no-domain-imports-CPT/REP. The rest is documented (`CODING_STANDARDS.md` §2) for code review until real file structure makes a bounded, non-guessed rule addition possible.
3. **Inert-until-real-paths-exist, same pattern as `ADR-0001`.** Both `import/no-restricted-paths` zones reference `lib/`, `server/`, `app/(tenant)/`, `app/(customer)/` paths that don't exist yet — configured now so they fire the moment Phase 1 creates them, never requiring a retrofit pass.
4. **`SELECT *` and raw service-role-key-access bans are enforceable today**, unlike the boundary rules — no folder gate needed, so proven against real fixtures this checkpoint (§5).

## 5. Real bugs found and fixed during authoring (not left open)

1. **`eslint-plugin-import` was not directly importable.** First attempt to lint a fixture failed with `ERR_MODULE_NOT_FOUND: Cannot find package 'eslint-plugin-import'` — pnpm's strict, non-hoisted `node_modules` (the exact property `ADR-0002` chose pnpm *for*) correctly refused to let my own `eslint.config.js` import a package that was only a transitive dependency, not a declared one. Fixed by adding it as a direct `devDependency` at the exact already-resolved version (no version drift).
2. **ESLint flat config silently discards, not merges, a repeated rule key across config objects.** Two separate config objects each setting `no-restricted-syntax` (one for `SELECT *`, one for the service-role-key ban) resulted in only the *later* object's selector taking effect — proven with `eslint --print-config`, which showed the `SELECT *` selector entirely missing once the second object was added. Fixed by combining both selectors into one `no-restricted-syntax` rule array in a single config object. This is a real, non-obvious ESLint flat-config semantics gap (documented inline in `eslint.config.js` itself so it isn't rediscovered the hard way again).
3. **The `SELECT *` rule's own error message triggered itself.** The message text `"No \`SELECT *\` (NFR-PERF-002...) — select exact columns."` literally contains the substring "SELECT" + a space + "*", which is exactly what the rule's regex selector matches — `pnpm run lint` failed on `eslint.config.js` itself. Fixed by rephrasing the message to avoid the literal pattern ("No wildcard column SELECT … select exact columns instead of every column"), with an inline comment explaining why the phrasing is deliberate.
4. **`check-suppressions.ts` flagged its own docstring and rule-definition regex as violations.** Running the real checker against the repository before adding a self-exclusion produced 9 false violations, all from this module's own explanatory prose (`"eslint-disable* comment"`, `` `@ts-expect-error`/`@ts-ignore` ``) and its `SUPPRESSION_MARKERS`/pattern-definition source lines. Fixed with an explicit, documented `SELF_REFERENTIAL_EXCLUSIONS` set (this file + its test file only) rather than a vague broad exclusion — confirmed via a direct before/after run showing 9 violations without the exclusion, 0 with it.

Four real, independently-verified bugs in one checkpoint — each found by actually running the tool against real input, not by inspection, consistent with this build's evidence discipline throughout Phase 0.
