## 4. Design decisions

1. **GitHub Actions ŌĆö `ADR-0004`.** Resolves the remaining CI/CD-platform-product half of `ADR-CAND-ARCH-024` (the package-manager half was already `ACCEPTED` at `ADR-0002`/`PH0-085`). Evidence: this repository's Git host is GitHub, confirmed by this exact session's own repeated, successful GitHub API use (┬¦2 above); `ADR-CAND-ARCH-024`'s own criterion is "integrated with this repository's Git host," which GitHub Actions satisfies by construction, not by a separate evaluation.
2. **One job, sequential steps ŌĆö not a matrix.** The current codebase is a handful of files; splitting lint/typecheck/test into separate parallel jobs would add runner-startup and redundant-install overhead without a real parallelism benefit at this scale (Prompt 88 ┬¦17: "keep bounded/cache-aware"). Noted as a natural future optimization (┬¦22's "scoped job matrix ŌĆ” once affected coverage" applies) once the test suite grows materially ŌĆö not implemented now to avoid premature complexity.
3. **`pull_request`, not `pull_request_target`.** A fork's PR must never run with write access to repository secrets ŌĆö `pull_request_target` grants the base repository's token/secrets to code from an untrusted fork, exactly the class of risk ┬¦16/┬¦26 name ("untrusted-fork controls," "forked code cannot exfiltrate secrets"). `pull_request` runs fork PRs with a read-only, secret-free token by GitHub's own default behavior. Since no step in this workflow needs a secret at all yet, this choice costs nothing and closes an entire vulnerability class before it could ever matter.
4. **No `preflight` (env-schema validation) step wired into CI.** `scripts/env/validate.ts` (`PH0-086`) requires real `NEXT_PUBLIC_SUPABASE_URL`/keys to validate against ŌĆö no real per-environment Supabase project is provisioned yet (Phase 0 environment topology is a plan, `11_*.md` ┬¦2, not a provisioned fact). Wiring `preflight` today would require either skipping it silently or feeding it fabricated placeholder secrets just to produce a green checkmark ŌĆö both are exactly the kind of dishonest evidence this build's governance forbids. Deferred to whichever Phase 0/1 task first provisions a real CI-accessible environment.
5. **Full-history checkout (`fetch-depth: 0`), not the GitHub Actions default shallow clone.** `check-protected-paths.ts`'s base-ref diff (`origin/<base>...<head-sha>`) requires real commit history to compute a merge-base; a shallow clone would make this silently degrade or fail. Explicit trade-off: slightly slower checkout on a currently-tiny repository, in exchange for a diff check that actually works, disclosed rather than silently accepting a degraded check.
6. **`node-version-file: package.json` (Corepack + `actions/setup-node`'s pnpm cache).** Reads the pinned Node/pnpm versions directly from `package.json` (`engines.node`, `packageManager`) rather than restating them a second time in the workflow file ŌĆö one source of truth, avoiding the exact "two independently-authored copies that could drift" pattern this repository's own incident history (`ERR-2026-001..003`) warns against.
