## 3. Repair 1 — `ISS-2026-008` resolved (root-cause decision, not a symptom patch)

**Root cause identified:** `check-secrets.ts`, `logger.ts`, and `analytics.ts` were each authored at a different prior checkpoint (`PH0-94`, `PH0-93`, `PH0-97` respectively) with no cross-module design review of their key-name pattern lists. Prompt 99's integration test correctly found the three lists disagree on PII-shaped keys (`npwp`/`bankAccountNumber`/`salary`) — but the underlying question ("should a source-code secret scanner also catch hardcoded PII values by key name?") had never actually been decided, only newly noticed.

**Decision (this checkpoint): keep `check-secrets.ts` scoped to credential shapes; do not widen it to general PII detection.** Full rationale recorded in `docs/runtime/KNOWN_ISSUES.md` `ISS-2026-008` (now `RESOLVED`) and `docs/standards/SECURITY_STANDARDS.md` §3 (new paragraph): industry-standard secret scanners target credential shapes, not arbitrary PII; this repository already has three real, tested, purpose-built PII mechanisms for their own data-flow surfaces (`data-classification/registry.ts`+`check-registry.ts` for schema fields, `analytics.ts` for event properties, `logger.ts` for log fields); widening `check-secrets.ts` into a second overlapping PII scanner would be a new capability, which Prompt 100 §24 forbids ("no new capability or cosmetic cleanup; fix root cause, not test symptoms").

This is a **root-cause fix**, not a test-symptom patch: the actual defect was an undocumented, unproven scope boundary presented (at Prompt 99) as an "inconsistency." The fix corrects the documentation and proves the boundary is intentional and stable, rather than either (a) leaving it ambiguous or (b) mechanically forcing three independently-scoped modules into artificial agreement.

**Files changed:**
- `docs/standards/SECURITY_STANDARDS.md` — new paragraph in §3 stating the scope boundary and citing this task.
- `scripts/security/check-secrets.ts` — one-line scope comment above `GENERIC_HARDCODED_SECRET_ASSIGNMENT`; **zero behavior change** (the pattern itself is untouched).
- `scripts/security/check-secrets.test.ts` — new regression test proving `npwp`/`bankAccountNumber`/`salary` remain unmatched (module-level proof, not only integration-level).
- `scripts/verification/phase0-integration.test.ts` — the ISS-2026-008 test rewritten from "documents a disclosed gap" to "proves the boundary holds in both directions" (asserts `check-secrets.ts` still ignores the PII keys **and** `logger.ts` still redacts them — the responsibility is proven to land somewhere).
- `docs/runtime/KNOWN_ISSUES.md` — `ISS-2026-008` marked `RESOLVED` with full before/after evidence (index row + narrative section).

**Root cause:** `PH0-99.md` §5's failure-matrix table quotes, as evidence, the exact backtick-formatted citation that `FEATURE_FLAG_STANDARDS.md` used to contain (fixed during Prompt 99 itself) — an accurate historical quote, not a live existence claim by `PH0-99.md`. **Fix:** `scripts/docs/check-doc-links.ts` gains a second, narrower exemption set (`KNOWN_HISTORICAL_QUOTED_CITATIONS`), scoped to the exact `(file, path)` pair rather than excusing the path globally the way `KNOWN_HISTORICAL_BROKEN_LINKS` does — so a genuinely new citation of `docs/runbooks/deployment-rollback.md` from any other document is still caught. `PH0-99.md` itself is **not edited** (append-only evidence discipline — the same reasoning `ISS-2026-006` already established: a committed, already-pushed historical build log's own prose is not rewritten to look retroactively clean). New regression test added in `scripts/docs/check-doc-links.test.ts`.

This finding and fix are disclosed here rather than folded silently into §3, since it was discovered by this checkpoint's own gate execution, not inherited from Prompt 99's failure matrix.
