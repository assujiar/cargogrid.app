## 3. Design notes

- **Prohibited-field rule is a rejection, not a redaction** — deliberately different from `scripts/observability/logger.ts`'s `redact()`, which masks a sensitive field and still logs the line (an operational-debugging need). An analytics payload has no equivalent need, so `findProhibitedProperties()` causes the *entire event* to be rejected (Prompt 97 §16 says "prohibit," not "mask").
- **Pseudonymization is a real `HMAC-SHA256`** (`node:crypto`, zero dependency), not a placeholder — deterministic per salt, non-reversible without it. The salt itself is disclosed as needing to be a `secret`-classified environment variable once Phase 1 wires real delivery, tying forward to `scripts/data-classification/check-registry.ts`'s existing adoption gate (`PH0-95`) rather than inventing a separate secret-handling rule.
- **Safe-disablement and safe-degrade reuse the exact design `scripts/observability/logger.ts` already proved** (`PH0-93`): `track()`'s `enabled=false` path does zero validation/sink work; a throwing sink never propagates to the caller. Applying the same proven pattern to a second signal type is intentional consistency, not incidental similarity.
