# ADR-0018 — Canonical Account/Customer entity shape and ownership (Commercial Phase 2)

Status: ACCEPTED
Date: 2026-07-24   Approver: Runtime build agent (Phase 2 governance authority, per `docs/adr/README.md` §3 "implementation-level decision... the owning phase's capability prompt, when its evidence exists" — the same kickoff/capability-level resolution authority `ADR-0015` already exercised for vendor/rate ownership)   Supersedes/Superseded-by: —
Source candidate: `ADR-CAND-ARCH-012` ("`customers` extension-table vs. flat-column strategy," `docs/adr/README.md` §5.2, status `BLOCKED — needs Phase 2 schema slice`)   Owning phase/task: Phase 2 (`CG-S7-COM-014`, Prompt 155, Customer and Account Conversion)

## Question

`ADR-CAND-ARCH-012` asks whether the first canonical customer/account table should be a jsonb-`attributes`-style extension table (i.e. built on `PLT-120`'s Master Data Engine — `app.master_types`/`app.master_records`, the pattern `ADR-0015` selected for `vendor_rate`) or a flat, typed-column table (the pattern every other Commercial entity — `app.leads`, `app.prospects`, `app.opportunities`, `app.quotations` — already uses). Prompt 155 §20 task 1 additionally requires "define canonical account/customer distinction" — the candidate register's own title conflates "customers" as a single noun, but the prompt treats "account" and "customer" as related-but-distinct concepts that this checkpoint must resolve, not assume equivalent.

## Evidence gathered this checkpoint (not assumed)

- `supabase/migrations/20260717120000_create_master_data.sql` (`PLT-120`, `VERIFIED`) seeds exactly one `master_type_code`: `'vendor_rate'` (`scope='tenant'`, `owner_module_code='PRC'`). No `'customer'`/`'account'` type is reserved. `app.register_master_type()` is **Supreme-Admin-only**; `app.create_master_record()` for a `tenant`-scope type requires `app.is_support_grant_authority()` (tenant_admin/support-grant chain) — both meaningfully more restrictive than the ordinary `COM:*` RBAC every other Commercial write path uses. Account conversion is expected to be a normal (if governance-weighted) Commercial workflow action performed by a rep/manager holding `COM:Approve`, not a Supreme Admin or tenant_admin action — the Master Data Engine's own authority model is the wrong shape for this capability's actual actor.
- `ADR-0015`'s own Decision section (§"Consequences," performance bullet) states the child-table pattern was chosen for `vendor_rate` specifically *because* "the new child table (not jsonb `attributes`) is what makes the composite tenant/service/lane/validity/status indexes" required possible — the same reasoning applies here: Prompt 155 §17 (Performance impact) requires indexing "normalized legal/tax/trade/contact/site identities and source/target links," and §16 (Security impact) requires restricting "legal/tax/billing/credit fields" by permission — both need real, indexable, individually-maskable typed columns, not a jsonb bag.
- Every Commercial entity built so far in this phase (`app.leads`, `COM-143`; `app.prospects`, `COM-144`; `app.opportunities`, `COM-147`; `app.quotations`, `COM-151`) is a flat, typed-column table with its own `duplicate_fingerprint`/normalization functions, RBAC gate, and record-scope — never built on `app.master_records`. `app.master_records` has been used exactly once in this phase, and only because `ADR-0015` found a specific, disclosed reason (cross-module `PRC` ownership, no Commercial-specific authority needed). No equivalent cross-module ownership question exists for the customer/account entity — it is Commercial-owned outright (Finance/Operations read it in later phases, per `05_DATABASE_SCHEMA_WORKSTREAM.md`'s own "customer master" references, but do not co-own its identity the way Procurement co-owns `vendor_rate`).
- `supabase/migrations/20260723210000_create_commercial_opportunity_management.sql`'s own `app.opportunities.account_ref text` column comment states verbatim: "a forward-compatible placeholder `COM-155` is expected to backfill, never a second customer identity" — confirming this checkpoint is the one expected to resolve the shape and backfill that column, not merely note the gap again.
- `app.prospects` (`COM-144`) already carries `legal_name`/`trade_name`/`tax_id`/`normalized_legal_name`/`normalized_tax_id`/`duplicate_fingerprint`/`billing_address` — the exact shape a flat-column `accounts` table needs, and its own `app.normalize_prospect_identifier`/`app.compute_prospect_duplicate_fingerprint` functions are generic (no `prospect`-specific logic in their bodies) and directly reusable, the same "generic body despite domain-specific name, reuse rather than duplicate" precedent `COM-144` already established for `app.lead_record_scope_org_unit_ids`.

## Options

1. **(SELECTED) A flat, typed-column `app.accounts` table, Commercial-owned, mirroring `app.prospects`' own shape — not built on the Master Data Engine.** "Account" is the canonical organization/legal-entity identity; `customer_status` (`active`/`inactive`) is a column on that same row representing whether the account presently holds an active commercial relationship, resolving Prompt 155's own "define the account/customer distinction" task as: one entity, one identity, a status field for the relationship — not two separate tables. Reuses `app.normalize_prospect_identifier`/`app.compute_prospect_duplicate_fingerprint` directly rather than re-authoring near-duplicates. Visibility is tenant-wide (`has_active_tenant_membership`), not per-owner record-scoped — mirroring `app.vendor_rate_versions_directory`/`app.quotation_approval_rules`' own "shared reference master, not a per-rep silo" posture, since multiple reps across teams legitimately need to see and transact against the same canonical customer over time.
2. **Build on `PLT-120`'s Master Data Engine** (`app.master_records` with `master_type_code='customer'` or `'account'`, plus a child detail table for structured fields, mirroring `ADR-0015`'s `vendor_rate` pattern exactly).
   - Trade-off against: the Master Data Engine's authority model (`register_master_type` Supreme-only; `create_master_record` tenant-admin/support-grant-only) is the wrong shape for an ordinary `COM:Approve`-gated Commercial workflow action, and would require either widening that authority (weakening it repository-wide) or adding a second parallel authority path just for this one type — neither is a real improvement over a flat table with its own RBAC gate, which every sibling Commercial entity already does successfully. Rejected — the cross-module-ownership rationale that justified this pattern for `vendor_rate` does not apply here.
3. **Two separate tables, `app.accounts` and `app.customers`**, with `app.customers` referencing `app.accounts` 1:1 or 1:many.
   - Trade-off against: no real business rule in Prompt 155 or any later-phase prompt (156/157, checked directly — neither mentions `app.customers`/`app.accounts` literally) requires an account to hold more than one customer relationship, or a customer relationship to outlive its account. A second table today would be built ahead of any concrete requirement driving its shape — the same "don't build a second interim model before a real need exists" discipline `COM-145` already applied when deferring a standalone site/address master. Rejected for this bounded slice; the extension point (a genuine `app.customers` table, if a real 1:many need appears) is named here, not foreclosed.

## Decision

**Option 1.** `app.accounts` is a new, flat, typed-column, Commercial-owned table (this checkpoint's own migration), mirroring `app.prospects`' shape and reusing its normalization/fingerprint functions directly. `customer_status` (not a second table) is the resolved account/customer distinction. Visibility is tenant-wide, not record-scoped, matching the "shared reference master" posture `ADR-0015`/`COM-150`/`COM-153` already established for comparable tenant-wide Commercial reference data. `app.opportunities.account_ref` is backfilled by this checkpoint's own conversion function, closing the exact forward-reference `COM-147`'s migration disclosed in advance.

## Consequences

- **DB/API:** One new migration (`COM-155`) adds `app.accounts`, `app.account_conversions` (conversion evidence/idempotency), and the conversion/duplicate-search/preview functions — fully additive, zero `PLT-120` change. A database-enforced partial unique index (`tenant_id, duplicate_fingerprint) where status = 'active'`) makes "conversion never silently duplicates" a real constraint, not merely an application-level check.
- **Security:** No per-field masking permission gates `tax_id`/`billing_address` -- `app.permissions_action_check` (`PLT-111`) is a fixed, closed 19-action enum reproduced from `docs/architecture/06_RLS_RBAC_WORKSTREAM.md` §5.1 with no billing-shaped action to reuse, and widening that architecture-level catalogue is out of this checkpoint's scope (a real, disclosed finding, not an oversight). `app.accounts` therefore follows `app.margin_rule_versions`' (`COM-150`)/`app.quotation_approval_rules`' (`COM-153`) own already-established "tenant-wide policy reference, not a specific deal's figure, not masked" precedent -- every column is visible to any active tenant member via the base table's own RLS policy directly, no separate masked directory view (unlike `app.quotations_directory`, which masks real columns). The restriction Prompt 155 §16 asks for is enforced one layer up: only an actor holding `COM:Approve` may create/convert an account at all, mirroring the margin-rule/quotation-approval-rule publish gate exactly.
- **Performance:** the flat-column shape is a precondition for the tenant/legal-name/tax-id/fingerprint indexes Prompt 155 §17 requires — the same reasoning `ADR-0015` already used to justify a child table over `attributes jsonb` for `vendor_rate`.
- **Migration/rollback:** additive only. `git revert` of this checkpoint's commit is safe and complete; no `PLT-120` object or any prior Commercial migration is altered.
- **Downstream impact / what remains open:** `COM-156` (Contract and Customer Pricing) and `COM-157` (Credit and Commercial Control) are expected to reference `app.accounts` directly (by id), not re-derive customer identity from a quotation/prospect a second time. Whether a genuine `app.customers` (1:many-with-account) table is ever needed remains open, named here as the extension point rather than pre-built. Credit readiness/status is explicitly **not** modeled on `app.accounts` in this checkpoint (Prompt 155 §24: "conversion does not grant credit automatically") — `COM-157` owns that field/table entirely, not extended here even as a placeholder.
