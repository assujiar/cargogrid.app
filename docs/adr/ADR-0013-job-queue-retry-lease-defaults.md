# ADR-0013 — Job queue backoff formula, worker lease duration, and DLQ numeric defaults

Status: ACCEPTED
Date: 2026-07-19   Approver: Runtime build agent (Phase 1 governance authority, per `docs/adr/README.md` §3)   Supersedes/Superseded-by: —
Source candidate: `ADR-CAND-ARCH-028` (newly minted this checkpoint — this task's own research confirmed no existing `ADR-CAND-ARCH-0NN` candidate covers this dimension; see Evidence)   Owning phase/task: Phase 1 (`CG-S6-PLT-029`, Prompt 132, Background Job Framework)

## Question

`02_CANONICAL_DATA_FLOW_MAP.md` §5.4 (Tech Arch §32.17, verbatim) names the qualitative background-job retry pattern — "immediate retry for transient error → exponential backoff → max attempts → DLQ after failure → manual replay by authorized admin → idempotency to avoid duplicate effects" — but gives no numeric formula. Neither the architecture corpus nor this capability's own reusable prompt template (`04-reusable-prompts/60_BACKGROUND_JOB_TEMPLATE.md`) specifies a backoff base/multiplier/cap, a jitter algorithm, or a worker-lease/heartbeat-timeout duration anywhere. What concrete numeric defaults should `app.compute_job_backoff_seconds()` and `app.claim_next_job()`/`app.heartbeat_job()`'s lease-duration parameter use?

## Options

1. **Equal-jitter exponential backoff (base 30s, 2x multiplier, capped at 3600s) with a 300s (5-minute) default worker lease duration (SELECTED).**
   - Trade-off: "equal jitter" (half the computed exponential delay is fixed, half is randomized) is a well-known, named strategy from the widely-cited AWS Architecture Blog post "Exponential Backoff and Jitter" — it avoids both the "thundering herd" problem of zero-jitter retries (every failed job retries at the exact same instant) and the unbounded randomness of "full jitter" (which can retry near-immediately even after many failures, defeating the point of backoff). A 30-second base means the first retry is not wastefully immediate for a job type expected to take real processing time, yet not punitively slow for a genuinely transient error; a 1-hour cap bounds the worst case for a job stuck failing repeatedly without waiting indefinitely between attempts. A 300-second (5-minute) lease matches this repository's own already-established "5 minutes" magnitude for a bounded window (`ADR-0011`'s webhook timestamp tolerance uses the same figure) — long enough that a normal heartbeat cadence (a live worker would heartbeat well before the lease expires) never falsely triggers a stale-lease reclaim, short enough that a genuinely crashed worker's job becomes reclaimable within a reasonable few minutes, not hours.
   - Trade-off against: these are still judgment calls, not values derived from CargoGrid's own measured traffic (none exists yet, this repository is still Phase 1 Platform Core with no live tenant traffic) — explicitly disclosed as the same class of reasoned default `ADR-0002`/`ADR-0004`/`ADR-0011`/`ADR-0012` already used absent blueprint-mandated numbers.
2. **No jitter (pure exponential backoff) or a fixed, non-exponential retry delay.**
   - Trade-off: no jitter reproduces the exact "thundering herd" failure mode the qualitative architecture text's own "exponential backoff" language is meant to prevent (many jobs that failed at the same moment, e.g. a shared downstream outage, would all retry at the exact same moment again); a fixed delay ignores the qualitative "exponential" requirement entirely. Rejected — both weaker than the architecture's own stated qualitative shape.
3. **"Full jitter" (uniformly random between 0 and the computed delay) instead of "equal jitter."**
   - Trade-off: full jitter has a real advantage (lower average retry latency under contention, per the same AWS blog post's own benchmark) but its lower bound is 0 — a job could retry near-instantly even on its 5th consecutive failure, which reads as "backoff isn't really happening" to an operator watching `next_attempt_at` values and complicates this checkpoint's own db-test (a bound of `[0, computed]` is much harder to assert against than `[computed/2, computed]` without a very large sample size). Equal jitter's narrower, still-randomized band is easier to reason about and test while keeping the anti-thundering-herd property. Not rejected outright as inferior in general, but a reasonable, disclosed choice between two legitimate named strategies.

## Decision

**Equal-jitter exponential backoff**: base 30 seconds, 2x multiplier per attempt, capped at 3600 seconds (1 hour) — `app.compute_job_backoff_seconds(p_attempts)` returns a value in `[capped/2, capped]` where `capped = least(3600, 30 * 2^(attempts-1))`. **Worker lease duration**: 300 seconds (5 minutes) default for both `app.claim_next_job()` and `app.heartbeat_job()`'s `p_lease_duration_seconds` parameter (caller-overridable, not hardcoded). **DLQ trigger**: unchanged from `PLT-131`'s own structural rule (`attempts >= max_attempts`, `max_attempts` defaulting to 3, itself a `PLT-131` column default this ADR does not revisit). Implemented in `supabase/migrations/20260719180000_create_background_job_framework.sql`.

## Consequences

- **DB/API:** `app.compute_job_backoff_seconds()` (pure, `volatile` since it uses `random()`, never `immutable`) is the physical backoff mechanism, called from `app.record_job_failure()`'s own retry branch; `app.claim_next_job()`/`app.heartbeat_job()` both accept `p_lease_duration_seconds` as a caller-supplied parameter defaulting to 300, never a hardcoded value a future capability could not override for a job type with genuinely different processing-time characteristics. Proven directly in `scripts/db-tests/background-job.sql` (bounds-checked across multiple attempts values, the stale-lease-reclaim mechanic proven via a manually-expired lease, the not-yet-claimable-before-`next_attempt_at` mechanic proven via a manually-future-dated row).
- **Security:** none directly — this ADR fixes timing/scheduling numbers, not an access-control boundary.
- **Performance:** the equal-jitter formula bounds retry storms without needing a live traffic measurement to validate ahead of time (the qualitative goal Tech Arch §32.17 states); the 5-minute lease is short enough that a crashed worker's job does not sit unreclaimed for an operationally significant duration.
- **Migration/rollback:** additive — `app.jobs.next_attempt_at` is a new nullable column, `app.compute_job_backoff_seconds()` is a new function, and the three `PLT-131` functions this migration extends via `CREATE OR REPLACE FUNCTION` keep identical signatures/exception-message prefixes, so no caller (including `PLT-131`'s own already-passing db-test) is broken.
- **Downstream impact / what remains open:** the adjacent "worker separation" threshold (`04_REPOSITORY_TARGET_STRUCTURE.md:116`, `08_API_INTEGRATION_WORKSTREAM.md:170` — when to split a dedicated `workers/` process boundary out of the in-process Next.js server) is **deliberately not resolved by this ADR** — no live traffic exists yet to measure against, and `08_*.md:170` itself says its own document "does not lower that threshold decision." This remains genuinely open, due whenever real production load first makes it a live question, not a Phase 1 Platform Core concern.
