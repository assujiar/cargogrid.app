# ADR-0014 — PostGIS extension version, spatial column type, and bounded-radius query cap

Status: ACCEPTED
Date: 2026-07-22   Approver: Runtime build agent (Phase 1 governance authority, per `docs/adr/README.md` §3)   Supersedes/Superseded-by: —
Source candidate: `ADR-CAND-ARCH-029` (newly minted this checkpoint — this task's own research confirmed no existing `ADR-CAND-ARCH-0NN` candidate covers this dimension; see Evidence)   Owning phase/task: Phase 1 (`CG-S6-PLT-031`, Prompt 134, PostGIS and Spatial Foundation)

## Question

RPD-015 (`01_MODULE_DEPENDENCY_MAP.md:60/119/237`) already ratifies *that* PostGIS is enabled from Platform Core, and `05_DATABASE_SCHEMA_WORKSTREAM.md` line 108 already ratifies *that* future spatial columns use `geography(Point)`/`geography(LineString)` added directly to owning-domain tables, not a standalone geo table. Neither document states: which PostGIS major version to pin (locally and in CI), what SRID a `geography` column carries (implicit in the type, but never stated as a number anywhere), or what maximum radius a bounded-proximity query (`ST_DWithin`) may request before it counts as the "unbounded global spatial scan" Prompt 134 §17/§23 forbid. What concrete numeric/version defaults should this checkpoint's `app.postgis_max_query_radius_meters()`, the extension version pin, and the CI database service image use?

## Options

1. **PostGIS 3.4.x, SRID 4326 (WGS84) via the `geography` type exclusively, 500,000-meter (500km) bounded-radius query cap (SELECTED).**
   - Trade-off: PostGIS 3.4 is the version this checkpoint's own research found packaged for both this repository's local sandbox (`postgresql-16-postgis-3` via `apt`, resolves to `3.4.2`) and the most recent stable [`postgis/postgis`](https://hub.docker.com/r/postgis/postgis) Docker Hub tag pairing with Postgres 17 (this repository's own already-pinned CI Postgres major version, `ADR-0004`) — no version skew between local dev and CI. SRID 4326 is not actually a free choice: this checkpoint's own direct testing proved casting any non-4326-SRID geometry to `geography` raises `Only lon/lat coordinate systems are supported in geography` — the `geography` type *is* SRID 4326, structurally, so this ADR's real job is naming that fact once, not inventing a policy. 500km is a per-query search-radius cap (not a coverage-area limit) comfortably above any realistic intra-country logistics search — Indonesia's own greatest linear extent is roughly 5,100km end-to-end, so 500km already covers a multi-province search without ever approving a genuinely global unbounded scan.
   - Trade-off against: like every other numeric default in this class, not derived from CargoGrid's own measured traffic (none exists yet — Phase 1 Platform Core has no live tenant data) — explicitly disclosed as the same reasoned-default class `ADR-0002`/`ADR-0004`/`ADR-0011`/`ADR-0012`/`ADR-0013` already used.
2. **`geometry` type instead of `geography`, with an explicit SRID CHECK constraint per column.**
   - Trade-off: `geometry` supports arbitrary SRIDs and planar (Cartesian) distance math, which is faster for small-extent calculations but requires every future domain table to carry its own SRID CHECK constraint and a `ST_Transform` step before any real-world distance query (`geometry` distances are in the SRID's own native units — degrees for 4326, not meters). `05_DATABASE_SCHEMA_WORKSTREAM.md` line 108 already resolved this exact question in favor of `geography` — this option is rejected as re-litigating an already-ratified decision, not a live alternative.
3. **No radius cap at all (unbounded `ST_DWithin`), deferring the limit to a future rate-limiting/API-gateway layer.**
   - Trade-off: matches `ADR-CAND-ARCH-018`'s own still-open "rate-limit numeric thresholds... due at the future live API-gateway/rate-enforcement capability" reasoning for a *different* dimension (webhook rate limits) — but Prompt 134 §17/§23 name the unbounded-scan prohibition as *this* capability's own explicit requirement, not a downstream one, and a GiST index's own selectivity advantage degrades sharply as the search radius approaches planetary scale regardless of any later gateway-level throttle. Rejected — the cap belongs in the one shared primitive (`app.bounded_st_dwithin()`) every future query is expected to call, the same "one governed choke point" reasoning `ADR-0012`'s GraphQL depth/complexity limits already used.

## Decision

**PostGIS `3.4.x`**, installed via `create extension if not exists postgis;` (`supabase/migrations/20260722090000_enable_postgis_spatial_foundation.sql`) and, for CI, the `postgis/postgis:17-3.4` service image (`.github/workflows/ci.yml`, replacing the plain `postgres:17` image the `db` job previously used — no other CI job needs PostGIS). **SRID `4326` (WGS84)** via the `geography` type exclusively — never `geometry` for a location column, per `05_DATABASE_SCHEMA_WORKSTREAM.md` line 108, restated here as the number that decision implies. **Bounded-radius query cap: 500,000 meters (500km)**, enforced by `app.postgis_max_query_radius_meters()`/`app.bounded_st_dwithin()`, the one governed choke point for any future proximity query.

## Evidence

- `docs/architecture/01_MODULE_DEPENDENCY_MAP.md:60/119/237` (RPD-015) — PostGIS enabled from Platform Core, not deferred to Advanced TMS.
- `docs/architecture/05_DATABASE_SCHEMA_WORKSTREAM.md` §6/line 108 — `geography(Point)`/`geography(LineString)` on owning-domain tables, no standalone geo table; already resolves the type-choice half of this ADR.
- This checkpoint's own direct testing against PostGIS 3.4.2 (not assumed from documentation): `SELECT ST_SetSRID(ST_MakePoint(1,1), 3857)::geography` raises `Only lon/lat coordinate systems are supported in geography`, confirming SRID 4326 is structural to the `geography` type, not a convention that could silently drift; `SELECT ST_SetSRID(ST_MakePoint(0, 95), 4326)::geography` **succeeds** with a `NOTICE: Coordinate values were coerced into range` — the concrete silent-clamping defect `app.geojson_point_to_geography()` (this checkpoint's own migration) exists to prevent by explicit range-rejection.
- `docs/adr/README.md` §5.2 — this checkpoint's own research confirmed all 28 existing `ADR-CAND-ARCH-0NN` candidates (including `ADR-CAND-ARCH-028`, newly minted at `PLT-132`) were checked; none covers PostGIS version/SRID/query-radius numeric thresholds.
- `docs/adr/ADR-0004-cicd-platform.md` — this repository's own already-ratified Postgres major-version pin (`17`) for CI, reused here (`postgis/postgis:17-3.4` keeps that same major version, adds PostGIS rather than introducing a second, unpinned version axis).

## Consequences

- **DB/API:** `app.postgis_max_query_radius_meters()`/`app.validate_geography_point()`/`app.geojson_point_to_geography()`/`app.geography_to_geojson_point()`/`app.bounded_st_dwithin()` (all `supabase/migrations/20260722090000_enable_postgis_spatial_foundation.sql`) are the physical mechanism — proven directly in `scripts/db-tests/postgis.sql` (round-trip tolerance, explicit rejection of out-of-range/malformed input, radius-cap enforcement, and a real `EXPLAIN`-plan proof that a GiST index on a `geography` column is actually used for an `ST_DWithin` predicate, not a sequential scan). No standalone table is created — future Operations-phase migrations (`shipments`/`warehouses`) call these functions when they add their own `geography` column.
- **Security:** a future domain column storing a precise individual/customer location is classified `pii`/`confidential` under the already-ratified `docs/standards/DATA_CLASSIFICATION_STANDARDS.md` `pii` category (reused, not a new category) — disclosed as guidance for that future migration, since no concrete column exists yet to classify. Tenant isolation is inherited automatically once a future table adds a `geography` column (a new column on an already-RLS-scoped table, not a new table needing its own policy).
- **Performance:** the 500km bounded-radius cap plus the GiST-index convention this ADR names (and `scripts/db-tests/postgis.sql` proves) together are the concrete "no unbounded global spatial scan" guarantee Prompt 134 §17/§23 require, without needing live traffic to validate ahead of time.
- **Migration/rollback:** additive — a new extension and five new pure/`stable` functions, zero alteration to any existing table/column/function. `git revert` of this checkpoint's commit is safe and complete; `DROP EXTENSION postgis` (were it ever needed) has no dependent object anywhere in this still-Platform-Core-only repository, since no table column uses `geography` yet.
- **Downstream impact / what remains open:** the actual `geography` column(s) on `shipments`/`shipment_milestones`/`warehouses` (Operations, Phase 3) are **deliberately not built here** (§12's own "TMS routing/geofence features... broad coordinate backfill" prohibition) — that future migration inherits this ADR's SRID/type/radius-cap conventions and calls the functions this checkpoint ships, rather than re-deciding any of them.
