# ADR-0015 — Canonical vendor/service/rate lookup ownership (Commercial Phase 2 vs. Procurement Phase 6)

## Numbering note (disclosed, not hidden)

`docs/architecture/01_MODULE_DEPENDENCY_MAP.md` §9 and `docs/architecture/05_DATABASE_SCHEMA_WORKSTREAM.md` §4/§11 both cite this exact vendor/rate-ownership question as **their own local "`ADR-CAND-ARCH-001`"** — a different candidate from `docs/adr/README.md` §5.1's master-register `ADR-CAND-ARCH-001` ("Single `app` schema ownership," already resolved and unrelated). This is the same class of pre-existing local-vs-master numbering divergence already found and disclosed once before, for `ADR-CAND-ARCH-010` (`00_PLATFORM_CORE_WBS.md` §3's "`server/contracts/` folder-existence timing" vs. the master register's differently-topiced "Configuration-engine sub-engine decomposition"). Per that precedent, this ADR does not renumber the two architecture documents' own internal citations (out of scope — they are closed, `VERIFIED` Step 3 artifacts) and instead mints the real master-register entry as a fresh number, `ADR-CAND-ARCH-030` (next after `029`, PostGIS, `ADR-0014`), rather than silently overloading the already-used `001`.

## Question

`docs/architecture/05_DATABASE_SCHEMA_WORKSTREAM.md` line 93 already resolves the *qualitative* ownership question ("Option (a)": Procurement owns the canonical vendor-rate data from Phase 1 onward; Commercial reads via a view) but that resolution has never been promoted to a binding `docs/adr/ADR-NNNN` record, and it leaves one concrete sub-question genuinely open: **who is authorized to write `vendor_rate`-typed master-data records during Phase 2 and Phase 3–5, before Procurement (Phase 6) ships any dedicated UI or write path?** Prompt 142 §"Required work" item 4 requires this resolved by a runtime ADR before Prompt 149 (Rate and Cost Lookup) may execute.

## Evidence gathered this checkpoint (not assumed)

- `docs/architecture/05_DATABASE_SCHEMA_WORKSTREAM.md` line 93 (already-`VERIFIED` Step 3 architecture): "`vendor_rates` is owned by `PRC` (Procurement) from Phase 1's master-data foundation onward — the table is created in Phase 1's `MDM` slice (empty, no Procurement UI yet) with `PRC` as the documented sole writer once Phase 6 ships; Commercial's Phase 2 costing reads it via a view (`app.v_active_vendor_rates`...) rather than copying rows."
- Direct inspection of `supabase/migrations/20260717120000_create_master_data.sql` (Platform Core, `PLT-120`, already `VERIFIED`) confirms this was **already built exactly as architecture required**, not merely planned: `app.master_types` carries a seeded row `('vendor_rate', 'Vendor Rate', 'tenant', 'PRC', 'platform-core-foundation')` (line 126), with the table's own comment (line 124) citing `05_DATABASE_SCHEMA_WORKSTREAM.md §4/§11's ADR-CAND-ARCH-001 resolution` directly. No literal `vendor_rates` table exists — the generic `app.master_records` mechanism holds one row per canonical rate record under `master_type_code='vendor_rate'`, the same "generic engine, not a bespoke table per instance" pattern every other Platform Core capability (Status, Numbering, Forms, Notifications, Documents, Feature Flags) already established.
- `app.create_master_record`/`app.update_master_record` (same migration, lines 272/327) gate tenant-scoped writes on `app.is_support_grant_authority(actor, tenant_id)` — the **already-existing, already-tested** tenant_admin/support-grant authority chain from `PLT-115`, not a per-`owner_module_code` restriction. `owner_module_code` is documentation-only metadata about the type's *eventual* full-lifecycle owner (Prompt 120 §20's own comment: "ties a type to the business-domain module that will eventually own its real business semantics") — it is not, and today cannot be, an enforced write-block, since no `PRC`-specific authority/role exists anywhere in the repository (Procurement is `NOT_STARTED`, Phase 6). **Consequence: a Tenant Admin can already create/update a `vendor_rate` master record for their own tenant today, with zero new write path required.**
- `149_RATE_COST_LOOKUP_PROMPT.md` §13 (Database impact) requires Commercial's own migration to "add/extend canonical tenant-aware vendor/service/rate references, rate versions, lane/zone/mode/equipment/weight/volume breaks, components, currency, minimum/surcharge, validity, approval status and selected-rate snapshot indexes" — structured, indexed fields `app.master_records`' generic `attributes jsonb` column cannot efficiently support (no composite btree/partial index over jsonb keys at the volume/query-budget the prompt requires).
- `docs/architecture/13_FULL_WORK_BREAKDOWN_STRUCTURE.md` §12 "Duplicates" check already treats this as resolved: "the one flagged near-duplicate risk (Commercial's Phase-2 vendor-rate lookup vs. Procurement's Phase-6 full lifecycle) is `ADR-CAND-ARCH-001` [local], already resolved as a single-owner table with a read-only view, not two masters."

## Options

1. **(SELECTED) Confirm Option (a) as already-built; Commercial adds a child detail table + read view on top of the existing `master_records`/`vendor_rate` registration; write authority stays the existing generic tenant-admin/support-grant chain until Phase 6.**
   - Commercial (`CG-S7-COM-008`, Prompt 149) creates one new migration adding `app.vendor_rate_versions` (or equivalently named), a child table keyed by `master_record_id references app.master_records(id)` (`master_type_code` constrained to `'vendor_rate'` via a check/trigger), carrying the structured rate-version fields the generic `attributes` jsonb cannot index efficiently (lane/zone/mode/equipment/weight/volume breaks, components, currency, minimum/surcharge, validity window, approval status), plus a read-only view `app.v_active_vendor_rates` (approved + within validity window + tenant-scoped) that Commercial's lookup/comparison/snapshot service queries — never a second, independently-writable rate master.
   - Trade-off: no new authority model is invented — the same `app.is_support_grant_authority()` gate `create_master_record`/`update_master_record` already use extends naturally to the new child table's own create/update functions, keeping one write-authority story across the whole Master Data Foundation rather than a Commercial-specific one that Phase 6 would have to unwind.
   - Trade-off against: until Phase 6 ships a dedicated Procurement authority/role, *any* tenant_admin/support-grant-authorized actor can write rate data — acceptable and explicitly disclosed (not a security gap: rate data is tenant-scoped, RLS-isolated, and audited via `capture_audit_event` on every write, identical to every other master-data type today) but must be re-examined at Phase 6 kickoff, when a narrower `PRC`-specific authority may be introduced without changing this table's shape.
2. **Commercial owns and creates its own independent `vendor_rate`/`vendor_service` schema, migrated to Procurement at Phase 6.**
   - Trade-off: matches the module map's originally-recommended-but-now-moot "Option (b)" framing, but directly contradicts `05_DATABASE_SCHEMA_WORKSTREAM.md` line 93's already-`VERIFIED` "Option (a)" resolution and would require a genuine data migration/re-parenting at Phase 6 that Option 1 above avoids entirely (same tables, only the authority/UI layer changes at Phase 6). Rejected — re-litigates an already-ratified Step 3 decision without new evidence to justify it, and `docs/architecture/**` documents are not re-derived without a real conflict (none exists here — Option 1 fully satisfies the already-ratified intent).
3. **Defer any lookup capability to Phase 6; Phase 2 costing uses only ad-hoc manual entry, no canonical source.**
   - Trade-off: violates the Phase 2 README's binding rule ("Phase 2 builds the canonical basic vendor/service/rate lookup foundation once") and Prompt 149's own explicit mandate to implement a real canonical lookup this phase. Rejected.

## Decision

**Option 1.** `app.master_types`/`app.master_records` (already `VERIFIED`, `PLT-120`) remain the single source of truth for `vendor_rate` identity; Commercial's Phase 2 Rate and Cost Lookup capability (`CG-S7-COM-008`, Prompt 149) adds exactly one child table for structured, indexable rate-version detail plus one read view (`app.v_active_vendor_rates`) — never a second independently-writable vendor-rate master. Write authority for the interim period (Phase 2 through Phase 5, before Procurement ships) is the already-existing tenant-admin/support-grant chain (`app.is_support_grant_authority`), reused unchanged — Prompt 149 must not invent a parallel authority model. At Phase 6, Procurement extends this same table set with full vendor lifecycle (vendors, contacts, assessments, RFQs, POs, performance scores per `05_DATABASE_SCHEMA_WORKSTREAM.md` line 75/190) and may introduce a narrower `PRC`-specific write authority as its own kickoff decision — an additive, expand-and-contract change to the authority layer only, never a data migration of the rate records themselves.

## Consequences

- **DB/API:** Prompt 149's own migration is scoped to: the child rate-version-detail table (FK to `app.master_records`, `master_type_code='vendor_rate'` enforced), its create/update functions (reusing `app.is_support_grant_authority`), and `app.v_active_vendor_rates` (or equivalently named read view). No change to `app.master_types`/`app.master_records`/`PLT-120`'s existing functions — fully additive.
- **Security:** Tenant isolation and RLS are inherited unchanged from `app.master_records`' own already-`VERIFIED` policy; the new child table needs its own RLS policy scoped identically (tenant-aware, mirroring the parent). Every write is audited via `app.capture_audit_event` exactly as `create_master_record`/`update_master_record` already do — Prompt 149 must call the same audit convention, not a new one.
- **Performance:** the new child table (not jsonb `attributes`) is what makes the composite tenant/service/lane/validity/status indexes Prompt 149 §17 requires possible — this ADR's decision is a precondition for that capability's own performance requirement, not an optional refinement.
- **Migration/rollback:** additive only — a new child table plus a new view, zero alteration to any Platform Core migration. `git revert` of Prompt 149's own commit is safe and complete.
- **Downstream impact / what remains open:** Phase 6 Procurement's own kickoff (`PRC-` range) must decide (a) whether to introduce a `PRC`-specific write authority narrower than today's tenant-admin/support-grant gate, and (b) the full vendor lifecycle tables (`vendors`, `vendor_contacts`, `vendor_assessments`, `purchase_orders`, `vendor_performance_scores`) that extend, but do not replace, the `vendor_rate` master-data registration and its Phase 2 child table this ADR establishes. Neither question blocks Phase 2 and neither is decided here.
